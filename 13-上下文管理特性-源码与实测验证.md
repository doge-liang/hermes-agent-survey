# Hermes Agent 上下文管理特性 — 源码研究 + 真实请求验证

> 测试环境: Hermes Agent **v0.15.1**, OpenAI SDK 2.24.0, Anthropic SDK 0.87.0
> 验证方式: 用真实 `AIAgent.run_conversation()` 驱动 + mock 后端捕获真实 HTTP 请求
> 验证平台: `validation_platform/` (mock_backend.py + driver.py + requests.jsonl)

## TL;DR(先读这一段)

本报告回答一个具体问题: **Hermes 这个 agent 框架,在真正发请求给推理后端时,会往请求里塞哪些"上下文管理"相关的字段,这些字段在什么条件下出现,推理框架(KV cache / 调度层)能拿它们做什么?**

"上下文管理"在这里指 agent 为了让多轮对话既省钱又不超窗口、还能命中上游缓存,而对请求做的一系列处理: 主动标注哪些前缀可以缓存(Anthropic 的 `cache_control`)、给请求带上会话亲和标识(`session_id` / `prompt_cache_key` / `x-grok-conv-id`)、让 system prompt 逐字节稳定以便命中 prefix cache、在历史太长时把旧对话压缩成摘要,等等。这些动作对推理框架是"免费的优化信号"——只要你读懂它们,就能更聪明地复用 KV cache、把同一会话路由到同一节点。

本报告与第 12 号报告(Agent-Hints 总结)的根本区别在驱动方式: **这一次用 Hermes 自己的真实对话循环 `AIAgent.run_conversation()` 来跑**,而不是在测试代码里手工复刻一遍 client 的构建逻辑。手工复刻容易和真实代码走偏; 用真实对话循环驱动,捕获到的就是"在真实触发条件下,Hermes 实际发出去的请求长什么样"。

> **术语对照(贯穿全仓一致)**
> - **import-driver**: 在 Python 里直接 `import` Hermes 并调用 `AIAgent.run_conversation()` 来驱动它发请求。本报告用的就是这种方式。它从对话循环"半路"切入,绕过了 CLI 入口的 config 解析与工具组装,因此看到的工具数 / system 长度会偏小(详见报告 15 §1)。
> - **真实 CLI**: 像真实用户那样在命令行跑 `hermes chat`,走完整入口。报告 15 用这种方式补齐了 import-driver 的盲区。
> - **agent hint**: agent 随推理请求一起下发给后端、用于辅助后端做 KV/调度决策的元数据(如会话键、缓存断点)。
> - **host 门控**: Hermes 按"这次请求要发往哪个域名"(请求 base_url 的主机名)来决定某些 hint 发不发。localhost 通常命中不了真实域名,所以本地 mock 测不到这类 hint(详见报告 15 §4.3)。
> - **api_mode**: Hermes 把出站协议分成四类——`chat_completions`(OpenAI 标准聊天接口)、`anthropic_messages`(Anthropic Messages 接口)、`codex_responses`(OpenAI Responses API,Codex/xAI 走这条)、`bedrock_converse`(AWS Bedrock)。本报告主要涉及前三类。

## 0. 实测概览

> 真实安装路径 `~/.hermes/hermes-agent/` 对应 v0.15.1。本报告全部结论都用这套真实安装的 venv 做端到端验证: 一边直接调用 Hermes 的内部函数看返回值,一边用 mock 后端捕获 Hermes 真实发出的 HTTP 请求,两路互相印证。

### 0.1 OpenRouter 会话标识实测

**为什么要单独验 OpenRouter**: OpenRouter 是一个"模型聚合中转",同一个 OpenRouter 账号背后可能转发到 OpenAI、xAI、Anthropic 等多家。Hermes 给 OpenRouter 流量注入会话标识的逻辑,既不像原生 OpenAI(默认什么都不带),也不像原生 xAI(只带 grok 专属头),需要单独看清楚。

先用最直接的方式验证——绕过对话循环,**直接调用** `OpenRouterProfile`(OpenRouter 这个 provider 的 profile 对象)上的两个构建函数,看它们对不同 model / session_id 组合分别返回什么:

```
build_extra_body(session_id="SID-123", model="openai/gpt-5")        → {'session_id': 'SID-123'}
build_extra_body(session_id=None,      model="openai/gpt-5")        → {}
build_api_kwargs_extras(session_id="SID-123", model="x-ai/grok-4")  → ({}, {'extra_headers': {'x-grok-conv-id': 'SID-123'}})
build_api_kwargs_extras(session_id="SID-123", model="openai/gpt-5") → ({}, {})   ← 非 grok 无 header, 仅 body.session_id
```

读法: `build_extra_body` 负责往请求 **body** 里加字段——可见只要给了 session_id,它就在 body 顶层放一个 `session_id`; session_id 为 `None` 时则什么都不加。`build_api_kwargs_extras` 负责往请求 **header** 里加字段——只有当 model 是 grok 系列时,它才额外塞一个 `x-grok-conv-id` 头(值同样是 session_id); 非 grok 模型(如 `openai/gpt-5`)不加任何 header。

再用端到端方式印证一遍(provider=openrouter, base_url 指向 mock, 跑真实 `AIAgent.run_conversation`)。下表是 mock 捕获到的 `POST /v1/chat/completions` 请求里的实际字段:

| 模型 | api_mode | `body.session_id` | header |
|------|----------|-------------------|--------|
| `qwen/qwen-2.5-72b-instruct` | chat_completions | ✅ `20260604_114054_43c77b` | (无) |
| `x-ai/grok-4` | chat_completions | ✅ `20260604_114055_39b4d5` | `x-grok-conv-id`(同值) |
| `openai/gpt-5` | **codex_responses** | (改用 `prompt_cache_key`) | (无 session_id header) |

表格逐行解释:

- **qwen 模型**走标准 `chat_completions` 协议,body 顶层带 `session_id`,没有任何专属 header。
- **grok 模型**同样走 `chat_completions`,body 带 `session_id`,**额外**带一个 `x-grok-conv-id` 头——而且这个头的值和 `body.session_id` 完全相同(同一个 session_id 写了两处)。
- **`openai/gpt-5`** 比较特殊: 它经 OpenRouter 时被 Hermes 路由到了 **`codex_responses`** 模式(即走 OpenAI Responses API,而非标准聊天接口)。这条路径不放 `body.session_id`,而是改用 `prompt_cache_key`(值仍是 session_id),也没有 session_id 相关的 header。

实测 chat_completions 请求的 body keys 是: `['messages', 'model', 'stream', 'stream_options', 'session_id']`。结论很清楚: **OpenRouter 对所有(走 chat_completions 的)模型都在 body 顶层注入 `session_id`**——它落在 chat_completions 标准协议内,不需要任何自定义 header 就能携带。对推理框架来说,这是做"会话亲和"(把同一会话的请求粘到同一节点)最直接、最现成可用的标识。

> ⚠️ **一个容易踩的细节**: 上面 `openai/gpt-5` 这一行说明了一件事——同样是"OpenAI 的模型",经过不同 provider 时行为完全不同。经 **provider=openrouter** 时,`openai/gpt-5` 被路由到 `codex_responses` 模式,此时 body 用 `prompt_cache_key`(=session_id)而非 `body.session_id`,而且**没有** `session_id` / `x-client-request-id` 这两个 header——那两个 header 只在请求真正发往 **OpenAI Codex 官方后端**(provider=openai-codex)时才注入(详见 §三点五)。换句话说,"用什么字段携带会话标识"取决于 provider profile 和 api_mode,不取决于模型名本身。

## 一、Hermes 支持的上下文管理特性总览

下表是本报告涉及的全部上下文管理特性的"地图": 每个特性在哪一层被触发、把东西注入到请求的哪个位置、什么条件下才触发(并标了源码位置作锚点)。后文逐个展开。

| # | 特性 | 触发层 | 注入位置 | 触发条件 (源码) |
|---|------|--------|---------|----------------|
| 1 | **Anthropic Prompt Caching** (cache_control) | 每次 API 调用前 | 消息 body | provider/model/base_url 组合判定 |
| 2 | **prompt_cache_key / session_id** | transport build_kwargs | header + body | Codex/xAI backend |
| 3 | **Prefix Cache 稳定性** (系统提示词字节稳定) | 系统提示词构建 | system 内容 | 始终 (3层结构 + 日期精度时间戳) |
| 4 | **消息规范化** (KV cache 一致性) | 每次 API 调用前 | 消息 body | 始终 (strip + JSON sort_keys + surrogate 清理) |
| 5 | **上下文压缩** (context compression) | 对话循环 (preflight + real-time) | 消息列表重构 + 摘要请求 | token 超阈值 |
| 6 | **reasoning / thinking config** | transport build_kwargs | header/body | provider 支持 + reasoning_config |
| 7 | **Context Length 探测** | agent 初始化 | `/api/show` / `/models` | 本地端点 (localhost/RFC-1918) |
| 8 | **会话轮换** (session rotation) | 压缩完成后 | 新 session_id + parent_session_id | 压缩触发 |

把这 8 个特性按"目的"归一下类,便于理解:

- **缓存类(让上游少做重复计算)**: 特性 1(主动标缓存断点)、特性 3(让前缀字节稳定以命中 prefix cache)、特性 4(把消息序列化做成 bit-perfect 以便前缀逐字节匹配)。
- **会话亲和类(让请求路由到持有 KV 的节点)**: 特性 2(会话/缓存键)、特性 8(压缩后换 session_id 但用 parent 链接)。
- **窗口管理类(别超上下文窗口)**: 特性 5(压缩)、特性 7(先探测窗口大小才能算压缩阈值)。
- **推理控制类**: 特性 6(深度思考预算)。

## 二、逐特性: 触发条件 + 实测请求

### 特性 1: Anthropic Prompt Caching (cache_control)

**这是什么**: Anthropic 的 prompt caching 允许调用方在消息里打 `cache_control: {type: ephemeral}` 标记,告诉 Anthropic "这个标记之前的内容请缓存起来,下次同样的前缀直接复用,省掉重复 prefill"。Hermes 需要决定两件事: (1) 这次请求该不该打这种标记; (2) 标记该用 **native** 形态(打在 content block 内部,Anthropic 原生协议要求的写法)还是 **envelope** 形态(打在消息顶层,某些中转协议要求的写法)。

**触发判定**(`agent/agent_runtime_helpers.py:1139` 的 `anthropic_prompt_cache_policy()` 函数,这个函数就是上面那个"该不该缓存、用哪种布局"的决策中心):

| provider / base_url / model | should_cache | native_layout | 布局 |
|---|---|---|---|
| `provider=anthropic` (api.anthropic.com) | ✅ | ✅ native | content block 内 |
| base_url 以 `/anthropic` 结尾 + Claude | ✅ | ✅ native | content block 内 |
| OpenRouter (base_url 含 openrouter.ai) + Claude | ✅ | ❌ envelope | 消息顶层 |
| Nous Portal + Claude / Qwen | ✅ | ❌ envelope | 消息顶层 |
| MiniMax (api.minimax.io) | ✅ | ✅ native | content block 内 |
| **其他 (含 localhost custom)** | ❌ | — | 不注入 |

表格读法: `should_cache` 列是"要不要打缓存标记",`native_layout` 列是"打的话用 native 还是 envelope"。可以看到原生 Anthropic 端点、以 `/anthropic` 结尾的第三方端点、MiniMax 都走 native(标在 content block 内部); OpenRouter 和 Nous Portal 转发 Claude 时走 envelope(标在消息顶层); 而最朴素的 localhost custom 端点**根本不打缓存标记**。

**实测验证**(driver 场景 A/B/E,看 Hermes 内部计算出的策略标志):

```
场景 A: provider=custom, base_url=localhost     → _use_prompt_caching=False  ✓ (基线不注入)
场景 B: provider=anthropic, api_mode=anthropic_messages → _use_prompt_caching=True native_layout=True  ✓
场景 E: base_url 以 /anthropic 结尾            → _use_prompt_caching=True native_layout=True  ✓
```

**关键发现**: cache_control 触发**不看 model 名,而看 base_url / provider**。这点很反直觉——场景里 model 明明是 `anthropic/claude-sonnet-4`(一个 Claude 模型),但只要 base_url 是 localhost,缓存策略仍然算成 `False`、不打标记。原因是: 缓存协议是后端能力,不是模型属性; 一个本地自建端点即便跑着 Claude 权重,也未必实现了 Anthropic 的 prompt caching 协议,所以 Hermes 保守地按"端点是谁"来判定。

**实测 cache_control 断点位置**(场景 C,采用 `system_and_3` 策略,真实捕获)。`system_and_3` 是 Hermes 给这种布局起的名字,意思是"缓存断点 = system 字段 + 最后 3 条非 system 消息":

```
turn 1 (1条消息):  cache_control 标记在 [system, msg[0]:user]
turn 2 (3条消息):  cache_control 标记在 [system, msg[0]:user, msg[1]:assistant, msg[2]:user]
turn 3 (5条消息):  cache_control 标记在 [system, msg[2]:user, msg[3]:assistant, msg[4]:user]
                                              ↑ msg[0,1] 不再标记, 断点向后滑动
turn 4 (7条消息):  cache_control 标记在 [system, msg[4]:user, msg[5]:assistant, msg[6]:user]
```

为什么这样设计: Anthropic 限制一个请求最多打 4 个缓存断点。Hermes 把 1 个固定留给 system(system 是整段对话里最长、最稳定、最值得缓存的前缀),剩下 3 个永远跟着对话末尾走。随着轮次增加,早期消息(如 turn 3 里的 `msg[0]`、`msg[1]`)就不再单独标断点了——因为它们已经被前一个断点覆盖在缓存前缀里了,再单独标也没意义。于是断点像一个长度固定的窗口,在对话上**向后滑动**。

证实: **始终是 system + 最后 3 条非系统消息(最多 4 个断点)**,断点随对话增长向后滑动。

> 注: 这里的"system"在 anthropic_messages 协议下是请求里一个**独立的 `system` 字段**(不在 `messages` 数组里),所以它和"最后 3 条非 system 消息"加起来恰好是 4 个断点。

实测请求 body(场景 B, `POST /v1/messages`),可以直接看到 `cache_control: {type: ephemeral}` 分别打在 system 字段和 messages 里:

```json
{
  "max_tokens": 64000,
  "model": "claude-sonnet-4-20250514",
  "system": [{"type": "text", "text": "You are Hermes Agent...",
              "cache_control": {"type": "ephemeral"}}],
  "messages": [{"role": "user", "content": [{"type": "text", "text": "...",
                "cache_control": {"type": "ephemeral"}}]}],
  "stream": true
}
```

请求 header 里还带 `anthropic-beta: interleaved-thinking-2025-05-14,fine-grained-tool-streaming-2025-05-14`(开启 Anthropic 的两个 beta 特性: 交错思考、细粒度工具流式)。

> 注: 这里 body 里的 `max_tokens: 64000` 是该场景该模型解析出的具体值,**不是 Hermes 的全局默认**——`max_tokens` 没有统一默认值,而是按 profile / 模型逐个解析(很多情况下整键缺省)。容易和它混淆的另一个 64000 是**上下文压缩的阈值地板** `MINIMUM_CONTEXT_LENGTH=64000`(见特性 5),两者数值相同但含义无关。

### 特性 3: Prefix Cache 稳定性 (系统提示词字节稳定)

**这是什么**: prefix cache(前缀缓存)是推理框架的通用优化——如果新请求的开头若干 token 和上一个请求逐字节相同,后端就能直接复用那段已经算好的 KV,跳过重复 prefill。system prompt 是每个请求最长、最靠前的公共前缀,只要它**逐字节稳定**,前缀缓存命中率就高。Hermes 为此特意把 system prompt 设计成"整天字节稳定"。

**机制**(`agent/system_prompt.py`):

- **三层结构**: system prompt 拆成 `stable`(身份 / 工具 / 技能,几乎从不变)+ `context`(AGENTS.md 等项目上下文文件)+ `volatile`(记忆 / 时间戳,最易变)。三层从稳定到易变排列,把易变的内容往后放,以最大化前面那段稳定前缀的长度。
- **时间戳只用日期精度**: 时间戳格式是 `%A, %B %d, %Y`(如 "Wednesday, June 04, 2026"),**只到"天",不含小时分钟**。这样一来,只要还在同一天,system prompt 里的时间戳就一个字节都不会变——整天字节稳定。如果带了分钟,那每分钟 system prompt 都会变一次,前缀缓存几乎永远命中不了。
- **缓存与失效**: 拼好的 system prompt 缓存在 `_cached_system_prompt`,**只有在发生压缩时才主动失效**(`invalidate_system_prompt`)——因为压缩会重排消息结构,system 需要重新拼。

**实测验证**: 场景 C 跑了 4 轮对话,全程 session_id 保持不变(`20260604_xxx_651ba4`),且每一轮的 system 消息**字节完全一致**(len=1717 恒定不变)。这意味着这 4 轮请求顶着同一段一字不差的前缀,上游 prefix cache 可以稳定命中。

### 特性 4: 消息规范化 (KV cache 一致性)

**这是什么**: 上面特性 3 让 system prompt 稳定,这里特性 4 进一步让**整个消息序列**在序列化层面稳定。哪怕语义相同,如果 JSON 序列化的字节不同(比如 key 顺序不一样、多了个空格、有个孤立的代理字符),前缀逐字节匹配就会断,prefix cache 就命中不了。Hermes 在每次 API 调用前做 5 步规范化,把这些"看不见的字节抖动"消除掉。

**机制**(`agent/conversation_loop.py:1030-1079`,每次 API 调用前依次执行这 5 步):

1. `_sanitize_api_messages` —— 删除孤立的 tool_call / tool_result(只有调用没结果、或只有结果没调用的残缺项),必要时注入 stub 占位,保证消息结构合法。
2. `_drop_thinking_only_and_merge_users` —— 删掉"纯思考"轮(只有 thinking、没有实际输出的轮次),并把相邻的两条 user 消息合并成一条(避免连续同 role)。
3. `content.strip()` —— 去掉每条消息内容首尾的空白字符。
4. **tool_calls.arguments 的 JSON 规范化** —— 对工具调用参数用 `json.dumps(sort_keys=True, separators=(",",":"))` 重新序列化: 排序所有 key、去掉分隔符多余空格。这一步是 bit-perfect 前缀的关键——同样的参数无论模型按什么顺序生成,序列化出来的字节都一样。
5. `_sanitize_messages_surrogates` —— 替换孤立的代理字符(Unicode U+D800–DFFF 范围),这些字符在 JSON 编码时会出问题,清掉以保证序列化稳定。

**实测验证**: 所有捕获到的请求,其消息 content 都已 strip(无首尾空白); 走 chat_completions 的请求都带 `stream_options: {include_usage: true}`(真实对话循环默认是流式的,并要求在流尾返回 usage,这点对压缩判定很关键,见特性 5 与 §三 优化建议 2)。

### 特性 5: 上下文压缩 (context compression) ★ 核心

**这是什么**: 多轮对话会越攒越长,迟早超过模型的上下文窗口。Hermes 的应对是**压缩(compaction)**: 当历史 token 数超过阈值时,把较早的对话喂给模型生成一段结构化摘要,用这段摘要替换掉原始的长历史,从而把上下文长度压回去。这是上下文管理里最复杂、对推理框架影响最大的特性。

**触发条件**(`agent/context_compressor.py` + `conversation_loop.py`):

- **阈值**: `threshold_percent` 默认 **0.50**(即上下文窗口的 50%),并且有个最小地板 **64000 tokens**。完整公式是 `max(context_length × 0.5, 64000)`——窗口的一半,但不低于 64000。
- **两阶段触发**:
  - **Preflight(API 调用前的预检)**: 在发请求前用粗略估算的 token 数判断会不会超阈值,代码在 `conversation_loop.py:639`。
  - **Real-time(API 返回后的实时判定)**: 等 provider 在响应 usage 里返回了真实 token 数,再判定一次,代码在 `conversation_loop.py:3965`。
- **`awaiting_real_usage` 机制**: 这是个防误压的设计。preflight 只是粗估,可能高估; 如果粗估一超阈值就立刻压缩,可能压了个寂寞(其实没真超)。所以当 preflight 粗估超阈值时**不立即压缩**,而是把这次判定**延迟到下一轮拿到真实 provider usage 再确认**。这就要求 provider 必须准确回报 usage,否则 Hermes 会一直延迟、压不下去(详见 §三 优化建议 2)。

**实测触发**(场景 F,人为把 context_length 设成 8000、threshold 设成 4000,并喂大块内容 big_content):

```
turn 1: prompt_tokens=2843
turn 2: prompt_tokens=8088
📦 Preflight compression: ~7,639 tokens >= 4,000 threshold
🗜️ Compacting context — summarizing earlier conversation...
日志: context compression done: messages=7->5   ← 真正压缩 (7条→5条)
```

读法: turn 1 时 prompt 才 2843 token,没超 4000; turn 2 涨到 8088,预检估出 ~7639 token 已超 4000 阈值,于是触发压缩,把消息从 **7 条压成 5 条**(把中间一段历史换成了一条摘要)。

**实测: 压缩产生的摘要请求**(真实捕获, `POST /v1/chat/completions`)。注意这是一个独立的"辅助请求"——Hermes 专门发一条请求,让模型把旧历史总结成摘要:

```
seq=12  model=mock-model  messages=1 (单条 user!)  无 system prompt
messages[0].role = user
messages[0].content (9998 字符) =
  "You are a summarization agent creating a context checkpoint.
   Treat the conversation turns below as source material for a compact
   record of prior work. Produce only the structured summary...
   NEVER include API keys, tokens, passwords... replace with [REDACTED]...
   Create a structured checkpoint... [后接全部待压缩历史的序列化]"
```

这个摘要请求有三个值得注意的特征:

- **用主 provider**(日志 `Auxiliary auto-detect: using main provider custom`): 摘要不是另开一个模型,而是复用主对话的同一个 provider。
- **单条 user 消息、无 system prompt**: 它和主对话请求结构完全不同——主对话有 system + 多轮 messages,而摘要请求把"你是摘要 agent"这套指令直接塞进**唯一一条 user 消息**里,没有独立 system。
- **含防泄密指令**: 提示词里明确要求遇到 API key / token / 密码等凭据时替换成 `[REDACTED]`,避免把敏感信息写进长期保存的摘要。

> 注: 该摘要请求的 `max_tokens` 不是固定值,而是 `int(summary_budget × 1.3)`,其中 `summary_budget` 有个下限常量 `_MIN_SUMMARY_TOKENS=2000`(即 budget 不会低于 2000,但会按内容量往上浮动)。

**实测: 压缩后的主请求**(真实捕获)。摘要生成完后,Hermes 用它重建消息列表再发主对话请求:

```
seq=13  messages=6:
  msg[0] role=system    (len=1717)  保护的系统提示词
  msg[1] role=user      (turn 1)    保护的头部 (protect_first_n)
  msg[2] role=assistant [CONTEXT COMPACTION — REFERENCE ONLY] Earlier turns...  ← 摘要消息
  msg[3] role=user      (turn 3)    保护的尾部
  msg[4] role=assistant            保护的尾部
  msg[5] role=user      (turn 4)    保护的尾部
```

读法与设计要点:

- **摘要消息以 assistant role 插入**: 因为它前面那条(`msg[1]`)是 user,如果摘要也用 user role 就会出现连续两条 user,破坏 user/assistant 交替结构; 用 assistant 正好衔接。
- **内容前缀 `[CONTEXT COMPACTION — REFERENCE ONLY]`**(即 SUMMARY_PREFIX,摘要前缀): 这串标记是为了防止模型把摘要内容当成"要执行的指令"——它明确告诉模型"下面这段只是历史参考,别照着做"。这是防"摘要逃逸"(摘要里的旧指令被当成新指令重新执行)的关键。
- **结构 = `[系统] + [头部保护] + [摘要] + [尾部保护]`**: 压缩不是把所有历史都揉成摘要,而是**保护两头、只摘中间**。`protect_first_n` 保住开头几轮(任务最初的目标往往在这里),`protect_last_n` 保住最近几轮(当前正在做的事),中间那段才被摘要替换。

**会话轮换**(特性 8): 压缩完成后,Hermes 会生成一个**新的 session_id**,并通过 `parent_session_id` 字段把新旧会话链接起来(这样既换了缓存键、又保留了血缘关系)。本测试因为 driver 是手动把上一轮的 history 传回下一轮(没走 Hermes 自己的 session 持久化),所以没观测到 session_id 变化,但源码 `conversation_compression.py:509` 确认了这个轮换行为。

### 特性 6: reasoning / thinking config

**这是什么**: "reasoning"(推理 / 深度思考)指让模型在给出最终答案前先做一段内部思考。不同 provider 表达这个意图的字段不同——Anthropic 用 `thinking`,OpenAI chat 路径用 `extra_body.reasoning`,Codex 用 `body.reasoning`。Hermes 根据 provider 和用户配置的 reasoning 强度(effort),注入对应字段。

**实测触发**(场景 D, provider=anthropic, reasoning_config={effort: high}):

```json
// POST /v1/messages body:
{
  "thinking": {"type": "enabled", "budget_tokens": 16000},
  "max_tokens": 64000,
  "temperature": 1
}
```

读法: 在 Anthropic 路径下,`effort=high` 被翻译成 `thinking.budget_tokens=16000`(给思考分配 16000 token 预算),同时把 `temperature` 设成 1(Anthropic 开 thinking 时通常要求 temperature=1)。

**chat_completions 路径**(由 `_supports_reasoning_extra_body` 这个门控函数决定): 在标准聊天接口下,Hermes 只在 base_url 含 `openrouter` / `nous` / `github` / `lmstudio` 等子串时才注入 `extra_body.reasoning`。localhost custom 端点**不注入**(本测试场景 A 已验证: 基线请求里没有任何 reasoning 字段)。这是一个 host 门控的例子——发不发 reasoning 取决于请求发往哪个域名。

### 特性 7: Context Length 探测 (本地端点)

**这是什么**: 要算压缩阈值(特性 5 的 `max(ctx × 0.5, 64000)`),Hermes 得先知道 `ctx`——也就是这个端点背后模型的上下文窗口有多大。对于云端 provider,这个值能从模型元数据查到; 但对于本地自建端点,Hermes 需要主动去**探测**。

**实测**: 每个连接到 localhost mock 的 agent 在初始化时都会发一个 `POST /api/show`(这是 Ollama 的接口,用于查询模型的 context length):

```
[1] POST /api/show    (场景 A 初始化)
[3] POST /api/show    (场景 B 初始化)
...
```

原因: base_url 是 `127.0.0.1`,Hermes 判定它是本地端点(localhost / RFC-1918 私有网段),于是主动探测真实 context length,用于后续计算压缩阈值。这是上下文管理的**前置行为**——Hermes 必须先知道上下文窗口大小,才能算出"什么时候该压缩"。对应地,如果是 OpenAI 兼容端点,Hermes 会改用 `/v1/models` 来探测。

## 三、对推理框架亲和性优化的关键结论

### 3.1 真实请求中可观测的上下文管理信号

把前面所有特性"落到 wire 上能看到什么"汇总成一张信号表。左列是请求里实际可观测的字段,中列是它出现的条件,右列是推理框架能拿它做什么:

| 信号 | 出现条件 | 推理框架可用于 |
|------|---------|---------------|
| `cache_control: {type:ephemeral}` (content/envelope) | Anthropic-wire / OpenRouter Claude | **KV cache 断点对齐** |
| `prompt_cache_key` (body) | Codex/xAI | cache 桶路由 |
| `session_id` / `x-client-request-id` (header) | Codex backend | 会话亲和路由 |
| `x-grok-conv-id` (header) | xAI | 会话亲和路由 |
| system 消息字节稳定 (跨轮恒定) | 始终 | **prefix cache 复用** |
| tool_calls.arguments 已 sort_keys | 始终 | **bit-perfect prefix 匹配** |
| `stream_options.include_usage` | chat_completions 流式 | 需在流尾返回准确 usage |

这张表是给推理框架开发者看的"采集清单": 凡是右列标了用途的信号,都值得在你的网关 / 调度层里读出来加以利用。

### 3.2 优化建议

1. **prefix cache 对齐 cache_control 断点**: Hermes 用 `system_and_3` 策略,断点固定在 system + 最后 3 条消息,且随对话后移。推理框架应当在这些断点处保留 KV cache——断点的位置就是 Hermes 替你算好的"最值得缓存的边界",顺着它做缓存最省力。

2. **usage 反馈的准确性至关重要**: Hermes 的压缩依赖 `response.usage.prompt_tokens` 做 real-time 判定(就是上面的 `awaiting_real_usage` 机制)。**流式响应必须在流尾准确返回 prompt_tokens**,否则 Hermes 会一直延迟压缩。本测试早期就踩过这个坑: mock 一开始固定回报 usage=100,结果 Hermes 永远觉得"没超阈值",压缩空转(日志 messages=5->5,压了个寂寞)。所以你的后端如果想让 Hermes 的压缩正常工作,usage 必须是真实的。

3. **本地端点探测**: Hermes 对 localhost / RFC-1918 端点会发 `POST /api/show`(Ollama)或 `/v1/models` 探测 context length。自建推理框架应当实现这些端点并返回**真实的上下文窗口大小**,否则 Hermes 会退回兜底值,导致压缩阈值算不准(要么过早压、要么撑爆窗口)。

4. **system prompt 字节稳定**: Hermes 刻意让 system prompt 整天字节稳定(时间戳只到日期精度)。这等于给了推理框架一个承诺——可以安全地把 system prompt 前缀的 KV **长期缓存**,跨多轮甚至跨会话复用,而不用担心它每分钟变一次。

5. **会话亲和路由**: 对 Codex / xAI 流量,可以按 `session_id` / `x-grok-conv-id` / `prompt_cache_key` 做会话粘性,把同一会话的请求路由到**持有该会话 KV cache 的同一节点**,最大化 KV 复用。

## 三点五、特性 2 实测: Codex/xAI 会话标识与 prompt_cache_key

> 本节用真实 `run_conversation()` 驱动,provider=openai-codex / xai,base_url 指向 mock,请求走 `codex_responses` api_mode(即 OpenAI Responses API),样本见 `codex_xai_samples.json`。
>
> **背景**: Responses API 是 OpenAI 较新的接口,和传统 Chat Completions 结构差别很大(用 `input` 不用 `messages`、system 走独立 `instructions` 字段)。Codex 和 xAI 的请求都走这条路径,所以放在一起看。

### 触发条件(`agent/agent_init.py:291-322` + `transports/codex.py`)

Hermes 判定一个请求是不是 Codex / xAI 路径,看两个布尔标志: `is_codex_backend`(是不是 Codex 后端)和 `is_xai_responses`(是不是 xAI 的 Responses 路径)。判定依据是 provider 名和 base_url:

| provider | api_mode | is_codex_backend | is_xai_responses |
|----------|----------|------------------|------------------|
| `openai-codex` | codex_responses | ✅ | ❌ |
| `xai` / `xai-oauth` | codex_responses | ❌ | ✅ |
| base_url=chatgpt.com/backend-api/codex | codex_responses | ✅ | ❌ |
| base_url=api.x.ai | codex_responses | ❌ | ✅ |

可以看到: 既可以靠 provider 名命中(`openai-codex` / `xai`),也可以靠 base_url 命中(官方 Codex / xAI 域名)。两个标志互斥——一个请求要么是 Codex、要么是 xAI,不会同时是。

### 实测字段注入对比(真实捕获, `POST /v1/responses`)

下表把 Codex(gpt-5.1)和 xAI(grok-4)两条路径的请求逐字段对比,标出每个字段在 header 还是 body、两边各有没有:

| 字段 | 位置 | Codex (gpt-5.1) | xAI (grok-4) |
|------|------|-----------------|--------------|
| `session_id` | **header** | ✅ | ❌ |
| `x-client-request-id` | **header** | ✅ | ❌ |
| `x-grok-conv-id` | **header** | ❌ | ✅ |
| `prompt_cache_key` | **body** | ✅ | ✅ |
| `reasoning: {effort, summary}` | body | ✅ `{effort:medium, summary:auto}` | ❌ (grok-4 不支持) |
| `include: [reasoning.encrypted_content]` | body | ✅ | ✅ |
| `store` | body | `False` | `False` |
| `instructions` (= system prompt) | body | ✅ | ✅ |
| `input` (= 输入项, 非 messages) | body | ✅ | ✅ |

逐行的要点:

- **会话标识的差异**: Codex 用两个 header(`session_id`、`x-client-request-id`); xAI 用一个 header(`x-grok-conv-id`)。但**两边的 body 都有 `prompt_cache_key`**——这是共同点。
- **reasoning**: Codex 带 `reasoning: {effort: medium, summary: auto}`; grok-4 不支持 reasoning,所以这个字段不出现。
- **`include: [reasoning.encrypted_content]`**: 两边都有。这是 Responses API 特有的——把上一轮的加密推理内容带回来重放,保证跨轮的思考连续性。
- **`store: false`**: 两边都关。意思是"不要让服务端存储这次 response"——因为 Hermes 自己管理多轮状态,不依赖 OpenAI 服务端的 response 存储。
- **`instructions` / `input`**: 两边都用 Responses API 的结构(system 走 `instructions`,对话走 `input`),而不是 chat 的 `messages`。

### 关键发现: 所有会话标识 = agent.session_id (单一来源)

这是本节最重要的结论。把 6 个请求的会话标识字段全部取出来对比,发现它们**全部相等**(`全等=True`):

```
Codex: session_id == x-client-request-id == prompt_cache_key == "20260604_110454_eba5e9"
xAI:   x-grok-conv-id == prompt_cache_key == "20260604_110502_dac825"
```

也就是说,无论是 header 里的会话路由键还是 body 里的缓存键,Hermes 写进去的**都是同一个 `agent.session_id`**。这对推理框架是个好消息: **你不用区分这么多字段,统一用这一个值,就能同时做会话亲和路由(按它把请求粘到同一节点)和 KV cache 路由(按它找缓存桶)。**

### Codex Responses API 请求完整结构(真实捕获)

```json
{
  "model": "gpt-5.1",
  "instructions": "You are Hermes Agent, an intelligent AI assistant...",  // system prompt 独立字段
  "input": [{"role": "user", "content": "codex test"}],                    // 非 messages!
  "reasoning": {"effort": "medium", "summary": "auto"},
  "include": ["reasoning.encrypted_content"],                              // 重放加密推理 (跨轮连续性)
  "store": false,                                                          // Hermes 自管状态, 禁用服务端存储
  "prompt_cache_key": "20260604_110454_eba5e9",
  "stream": true
}
```

header: `session_id: 20260604_110454_eba5e9` + `x-client-request-id: 20260604_110454_eba5e9`

**与 chat_completions 的根本差异**(推理框架如果同时要支持这两种协议,必须注意):

- 用 `input` 数组而非 `messages` 数组承载对话。
- system prompt 走 `instructions` 这个独立字段,不混在输入项里。
- `store: false` —— Hermes 不依赖 OpenAI 服务端的 response 存储,自己管理多轮状态。所以你的后端不需要(也不该)替它存。
- `include: [reasoning.encrypted_content]` —— 跨轮重放加密的推理项,这是 Codex 维持上下文连续性的关键: 上一轮的思考被加密带回,模型才能接着想。

## 三点六、chat_completions 路径下的会话标识 (回答: OpenAI SDK 有 session_id 吗?)

> **核心结论**: session_id 注入与否,取决于 **ProviderProfile**(各 provider 自己的注入逻辑),**不是 OpenAI SDK 本身的能力**。OpenAI SDK 只是个传输管道,具体往请求里塞什么,由 Hermes 的 provider profile 决定。
> 源码: `agent/transports/chat_completions.py:502,512` 把 session_id 传给 `profile.build_extra_body()` / `build_api_kwargs_extras()`,再由各 provider profile 自行决定加不加、加在哪。

### 实测对比(v0.15.1 真实安装, 端到端 + 直接调用双重验证)

| provider (profile) | model | api_mode | body.session_id | header.x-grok-conv-id |
|--------------------|-------|----------|-----------------|----------------------|
| `custom` (默认 profile) | mock-model | chat_completions | ❌ 无 | ❌ |
| `openrouter` | qwen/qwen-2.5-72b-instruct | chat_completions | ✅ 注入 | ❌ |
| `openrouter` | x-ai/grok-4 | chat_completions | ✅ 注入 | ✅ 注入 (同值) |
| `openrouter` | openai/gpt-5 | **codex_responses** | (改用 `prompt_cache_key`) | ❌ |
| `qwen` | qwen-max | chat_completions | ❌ (需 portal.qwen.ai) | ❌ |

逐行读法: 默认的 `custom` profile 什么都不带(基线); `openrouter` 给 qwen 带 `body.session_id`、给 grok 再额外带 `x-grok-conv-id` 头(同值); 经 OpenRouter 的 `openai/gpt-5` 因被路由到 codex_responses,改用 `prompt_cache_key`; `qwen` profile 在 localhost 下不带 session 标识(它的 `metadata.sessionId` 需要 base_url 真的是 portal.qwen.ai 才注入)。

**直接调用验证**(直接调真实安装 `~/.hermes/.../venv` 里的代码, v0.15.1),和上表互为印证:

```
OpenRouterProfile.build_extra_body(session_id="SID-123", model="openai/gpt-5")  → {'session_id': 'SID-123'}   ← 注入
OpenRouterProfile.build_extra_body(session_id=None,      model="openai/gpt-5")  → {}
OpenRouterProfile.build_api_kwargs_extras(..., model="x-ai/grok-4")             → extra_headers={'x-grok-conv-id': 'SID-123'}
OpenRouterProfile.build_api_kwargs_extras(..., model="openai/gpt-5")            → {}   (非 grok 无 header)
```

实测各 provider 的 body keys 对比(直接看请求 body 顶层有哪些字段):

```
custom:           ['messages', 'model', 'stream', 'stream_options']                       ← 无会话标识
openrouter:       ['messages', 'model', 'stream', 'stream_options', 'session_id']          ← body.session_id ✅
openrouter+grok:  body.session_id + header.x-grok-conv-id (两者同值)
qwen:             ['messages', 'model', 'max_tokens', 'stream', 'stream_options', 'vl_high_resolution_images']
```

注意 `custom` 和 `openrouter` 的差别仅在多了一个 `session_id`; 而 `qwen` 带的是 `vl_high_resolution_images`(视觉相关)而非会话标识——印证了它在 localhost 下没注入 sessionId。

### 关键结论(v0.15.1)

1. **OpenAI 官方 Chat Completions API 本身没有标准 `session_id` 字段。**
   Hermes 走标准 OpenAI 端点(默认 profile / provider=custom/openai)时,请求里**不携带任何会话亲和标识**——只有 `messages` / `model` / `stream` / `stream_options`(以及可能的 `tools`)。这点用真实 `hermes` CLI 做了端到端验证。

2. **OpenRouter: body 顶层注入 `session_id`(全模型)。** grok 模型额外带 `x-grok-conv-id` header。
   - `openrouter/__init__.py` 的 `build_extra_body` 对**所有**(走 chat_completions 的)OpenRouter 模型都注入 `body.session_id`。
   - ⚠️ 例外: `openai/gpt-5` 等模型经 OpenRouter 被路由到 `codex_responses` 模式,此时用 `body.prompt_cache_key`(=session_id)代替 `body.session_id`。

3. **OpenRouter 上的 Grok 模型注入 `x-grok-conv-id` header**(只对 grok 系列,靠模型名前缀判断):
   ```python
   if session_id and model.startswith(("x-ai/grok-", "xai/grok-")):
       extra_headers["x-grok-conv-id"] = session_id
   ```

4. **Qwen Portal 的 `metadata.sessionId`** 需要 base_url 真的是 `portal.qwen.ai`(由 `_is_qwen_portal()` 判定),localhost mock 命中不了——这是一个 host 门控。实测 qwen 在 localhost 下只注入了 `vl_high_resolution_images`,没有 metadata。

5. **Hermes 默认 chat_completions 路径不使用 OpenAI 官方的 `prompt_cache_key` / `user` 字段。**
   OpenAI 官方 Chat Completions 其实是支持这两个字段做缓存路由的,但 Hermes 没用它们——它只在 Responses API(Codex/xAI)路径才注入 `prompt_cache_key`。所以如果你的后端是标准 OpenAI 兼容接口,默认从 Hermes 这儿拿不到 `prompt_cache_key`。

### 对推理框架的意义

把"不同接入方式各能拿到什么会话亲和信号"汇总成一张决策表:

| 接入方式 | 会话亲和信号 | 推理框架可用 |
|----------|-------------|-------------|
| 标准 OpenAI Chat Completions (custom/openai) | **无** | 只能靠 prefix 内容 hash 做被动 cache |
| OpenRouter (chat_completions) | `body.session_id` (+grok: header) | 按 session_id 做会话粘性 |
| Codex/xAI (Responses API) | header `session_id`/`x-grok-conv-id` + body `prompt_cache_key` | 按这一个值做粘性+缓存键 |
| Qwen Portal | `body.metadata.sessionId` | 按 sessionId 做粘性 |

**如果你自建推理框架、想从标准 OpenAI Chat Completions 流量里拿到会话亲和**: 默认是拿不到 session_id 的——需要三选一: (a) 让用户改走 OpenRouter profile(它会注入 `body.session_id`); (b) 给 Hermes 加一个自定义 provider profile,在其 `build_extra_body` 里自己注入 `prompt_cache_key` / `session_id`; (c) 推理框架自己按 "system prompt + 首条消息" 的 prefix hash 做**被动会话识别**——因为 Hermes 的 system prompt 是字节稳定的(特性 3),这个 prefix hash 可以当作隐式会话键用。

**关键技巧**(让真实 run_conversation 跑通而不超时 / 不逃逸)。下面每一条都是实测撞出来的踩坑经验:

- `enabled_toolsets=[]` —— 给 `run_conversation` 这个**内部参数**传空列表,禁用所有工具,避免触发可选工具的 lazy-install(如 edge-tts 语音依赖)把进程卡住,同时让模型单轮就结束(没工具可调)。
  > 注意: 这里的 `enabled_toolsets=[]` 是 **import-driver 调内部函数时**传的参数,它和"config 配置文件里的 `toolsets:[]`"是**两个不同的层**,效果也不同——前者真的得到 0 个工具,后者经 CLI 转换后仍会带上一批核心工具。本测试是 import-driver,走的是前者(详见报告 15 §1 与报告 17 §2.1 的勘误)。
- `skip_memory=True` + `skip_context_files=True` —— 跳过记忆和上下文文件(如 AGENTS.md)的扫描,加快启动、保持 system prompt 简洁可控。
- **多轮**: 把上一轮 `run_conversation` 返回的 `result['messages']` 作为下一轮的 `conversation_history` 传回去——手动串起多轮(本测试没走 Hermes 自己的 session 持久化,这也是没观测到 session 轮换的原因,见特性 8)。
- **触发压缩**: override `context_compressor.context_length` + `protect_last_n` / `protect_first_n`,再喂大 content,把 token 撑过阈值。
- **触发 cache_control**: `provider=anthropic`,或 base_url 以 `/anthropic` 结尾(不能只靠 model 名——见特性 1 的关键发现)。
- **触发 codex_responses**: 用 `api_mode="codex_responses"` override(只验 body 特征),或 `provider="openai-codex"`(额外触发 session header)。注意 mock 的 `/responses` 端点**必须发 Responses API 的 SSE 事件序列**,否则 transport 报 "stream did not emit a terminal response"。
- **触发 chat reasoning**: mock 的模型名不在 Hermes 的内置 metadata 里,所以需要 monkeypatch `agent._supports_reasoning_extra_body = lambda: True` 强行打开门控(真实 CLI 不需要这个 hack,见报告 15 §3 RC-04)。

## 五、OpenAI SDK 路径测试增强 (2026-06-04 补充)

**动机**: 初版 driver 对 Anthropic 路径覆盖很充分(B/C/D/E 共 4 个场景),但 **OpenAI SDK 路径**(chat_completions + codex_responses)只有 3 个场景(A 基线 / F 压缩 / G OpenRouter),而且其中 **codex_responses 实际没跑通**——因为初版 mock 的 `/responses` 端点返回的是非流式 JSON,而 Hermes 的 codex transport 是用 `responses.create(stream=True)` 调的,于是报 "stream did not emit a terminal response"(流里没收到终止事件)。

**增强内容**:

1. **mock_backend**: 给 `_responses` 端点补上完整的 Responses API SSE 事件序列(`response.created` → `output_item.added` → `output_text.delta` → `response.completed`),让 codex 路径能正常跑通流式。
2. **driver 新增 3 个场景**:
   - **H** `api_mode="codex_responses"`(custom)—— 验 codex 的 body 特征。
   - **I** `provider="openai-codex"` —— 让 `is_codex_backend=True`,触发 session header。
   - **J** `provider=openrouter` + reasoning —— 验 `extra_body.reasoning`。

**端到端实测结果**(真实安装 v0.15.1, 与 §三点五互相印证):

| 场景 | api_mode | 会话/缓存标识 | 推理配置 |
|------|----------|---------------|----------|
| H. codex (custom) | codex_responses | `body.prompt_cache_key`(=sid), `store=false`, **无 header** | `reasoning={effort:medium, summary:auto}` |
| I. codex (openai-codex) | codex_responses | `header.session_id` + `header.x-client-request-id` + `body.prompt_cache_key` **三者同值** | 同上 |
| J. chat + reasoning | chat_completions | `body.session_id` (openrouter) | `extra_body.reasoning={effort:high, enabled:true}` |

**关键结论**: 是否注入 `session_id` / `x-client-request-id` **这两个 header**,取决于 `is_codex_backend`(即 `provider=="openai-codex"`,或 base_url 命中 chatgpt.com/backend-api/codex,见 `chat_completion_helpers.py:574`); 而 `body.prompt_cache_key` 只要有 session_id 且 provider 非 xai / github 就注入。

对比场景 H 和 I 就能看清这个分界: 两者**同样走 codex_responses**,但 H 是第三方中转(custom),`is_codex_backend=False`,所以**只拿得到 body 里的缓存键 `prompt_cache_key`,拿不到 header 级的会话标识**; I 是官方 codex 端点(`is_codex_backend=True`),才同时有 header 和 body。

**对推理框架的含义**: 如果你想要 **header 级**的会话路由信号(`session_id` / `x-client-request-id`),上游必须是**官方 codex 端点或 provider=openai-codex**——第三方中转那条路只给 body 缓存键。换句话说,header 会话标识是 Codex 官方路径的"专属待遇",中转路径享受不到。
