| provider / base_url / model | should_cache | native_layout | 布局 |
|---|---|---|---|
| `provider=anthropic` (api.anthropic.com) | ✅ | ✅ native | content block 内 |

| MiniMax (api.minimax.io) | ✅ | ✅ native | content block 内 |

**实测验证** (driver 场景 A/B/C/E):

```

场景 E: base_url 以 /anthropic 结尾            → _use_prompt_caching=True native_layout=True  ✓
```

**关键发现**: cache_control 触发**不看 model 名，而看 base_url/provider**。
场景中 model 是 `anthropic/claude-sonnet-4` 但 base_url 是 localhost 时，缓存策略仍为 False。

**实测 cache_control 断点位置** (场景 C, `system_and_3` 策略, 真实捕获):

### 特性 3: Prefix Cache 稳定性 (系统提示词字节稳定)

**机制** (`agent/system_prompt.py`):
- 三层结构: `stable`(身份/工具/技能) + `context`(AGENTS.md) + `volatile`(记忆/时间戳)
- **时间戳只用日期精度** (`%A, %B %d, %Y` 如 "Wednesday, June 04, 2026"), 不含分钟 → 整天字节稳定
- 系统提示词缓存在 `_cached_system_prompt`, **仅压缩时失效** (`invalidate_system_prompt`)

**实测验证**: 场景 C 4 轮对话, session_id 保持不变 (`20260604_xxx_651ba4`),
system 消息字节完全一致 (len=1717 恒定) → 多轮命中上游 prefix cache。

---

### 特性 4: 消息规范化 (KV cache 一致性)

**机制** (`agent/conversation_loop.py:1030-1079`, 每次 API 调用前 5 步):
1. `_sanitize_api_messages` — 删除孤立 tool_call/result, 注入 stub
2. `_drop_thinking_only_and_merge_users` — 删除纯 thinking 轮, 合并相邻 user
3. `content.strip()` — 去除首尾空白
4. **tool_calls.arguments JSON 规范化** — `json.dumps(sort_keys=True, separators=(",",":"))` → bit-perfect prefix
5. `_sanitize_messages_surrogates` — 替换孤立代理字符 (U+D800-DFFF)

**实测验证**: 所有捕获请求的 content 均已 strip; chat_completions 请求带
`stream_options: {include_usage: true}` (真实 loop 默认流式)。

### 特性 5: 上下文压缩 (context compression) ★ 核心

**触发条件** (`agent/context_compressor.py` + `conversation_loop.py`):
- 阈值: `threshold_percent` 默认 **0.50** (50% 上下文窗口), 最小 64000 tokens
- 两阶段:
  - **Preflight** (API 调用前, 粗估 token): `conversation_loop.py:639`
  - **Real-time** (API 返回后, 真实 usage): `conversation_loop.py:3965`
- **`awaiting_real_usage` 机制**: preflight 粗估超阈值时**不立即压缩**, 而是延迟到下一轮真实
  provider usage 确认 (避免粗估高估导致的无效压缩)

**实测触发** (场景 F, context_length=8000, threshold=4000, big_content):
```
turn 1: prompt_tokens=2843
turn 2: prompt_tokens=8088
📦 Preflight compression: ~7,639 tokens >= 4,000 threshold
🗜️ Compacting context — summarizing earlier conversation...
日志: context compression done: messages=7->5   ← 真正压缩 (7条→5条)
```

**实测: 压缩产生的摘要请求** (真实捕获, `POST /v1/chat/completions`):
```
seq=12  model=mock-model  messages=1 (单条 user!)  无 system prompt
messages[0].role = user
messages[0].content (9998 字符) =
  "You are a summarization agent creating a context checkpoint.
   Treat the conversation turns below as source material for a compact
   record of prior work. Produce only the structured summary...
   NEVER include API keys, tokens, passwords... replace with [REDACTED]...
   Create a structured checkpoint... [后接全部待压缩历史的序列化]"
```
要点:
- 摘要请求**用主 provider** (`Auxiliary auto-detect: using main provider custom`)
- **单条 user 消息, 无 system prompt** (与主对话不同)
- 含防泄密指令 ([REDACTED] 凭据)

**实测: 压缩后的主请求** (真实捕获):
```
seq=13  messages=6:
  msg[0] role=system    (len=1717)  保护的系统提示词
  msg[1] role=user      (turn 1)    保护的头部 (protect_first_n)
  msg[2] role=assistant [CONTEXT COMPACTION — REFERENCE ONLY] Earlier turns...  ← 摘要消息
  msg[3] role=user      (turn 3)    保护的尾部
  msg[4] role=assistant            保护的尾部
  msg[5] role=user      (turn 4)    保护的尾部
```
要点:
- 摘要消息以 **assistant role** 插入 (因前面是 user, 避免连续同 role)
- 内容前缀 `[CONTEXT COMPACTION — REFERENCE ONLY]` (SUMMARY_PREFIX, 防模型把摘要当指令执行)
- 结构: `[系统] + [头部保护] + [摘要] + [尾部保护]`

**会话轮换** (特性 8): 压缩后生成新 session_id, 通过 parent_session_id 链接 (本测试因
driver 手动传 history 未观测到 session_id 变化, 但源码 `conversation_compression.py:509` 确认)。

## 三、对推理框架亲和性优化的关键结论

### 3.1 真实请求中可观测的上下文管理信号

| 信号 | 出现条件 | 推理框架可用于 |
|------|---------|---------------|
| `cache_control: {type:ephemeral}` (content/envelope) | Anthropic-wire / OpenRouter Claude | **KV cache 断点对齐** |
| `prompt_cache_key` (body) | Codex/xAI | cache 桶路由 |
| `session_id` / `x-client-request-id` (header) | Codex backend | 会话亲和路由 |
| `x-grok-conv-id` (header) | xAI | 会话亲和路由 |
| system 消息字节稳定 (跨轮恒定) | 始终 | **prefix cache 复用** |
| tool_calls.arguments 已 sort_keys | 始终 | **bit-perfect prefix 匹配** |
| `stream_options.include_usage` | chat_completions 流式 | 需在流尾返回准确 usage |

### 3.2 优化建议

1. **prefix cache 对齐 cache_control 断点**: Hermes 用 `system_and_3` 策略, 断点在
   system + 最后 3 条消息且随对话后移。推理框架应在这些断点处保留 KV cache。

2. **usage 反馈的准确性至关重要**: Hermes 的压缩依赖 `response.usage.prompt_tokens` 做
   real-time 判定 (`awaiting_real_usage` 机制)。**流式响应必须在流尾准确返回 prompt_tokens**,
   否则 Hermes 会持续延迟压缩 (本测试早期 mock 返回固定 100 导致压缩 messages=5->5 空转)。

3. **本地端点探测**: Hermes 对 localhost/RFC-1918 端点会 `POST /api/show` (Ollama) 或
   `/v1/models` 探测 context length。自建推理框架应实现这些端点返回真实上下文窗口,
   否则 Hermes 用兜底值 (可能导致压缩阈值不准)。

4. **system prompt 字节稳定**: Hermes 刻意让系统提示词整天字节稳定 (日期精度时间戳),
   推理框架可安全地长期缓存系统提示词 prefix 的 KV。

5. **会话亲和路由**: 对 Codex/xAI 流量, 按 `session_id`/`x-grok-conv-id`/`prompt_cache_key`
   做会话粘性, 把同会话请求路由到持有 KV cache 的同一节点。

### 关键发现: 所有会话标识 = agent.session_id (单一来源)

实测核对 (6 个请求全部 `全等=True`):
```
Codex: session_id == x-client-request-id == prompt_cache_key == "20260604_110454_eba5e9"
xAI:   x-grok-conv-id == prompt_cache_key == "20260604_110502_dac825"
```
即: Hermes 把同一个 `agent.session_id` 同时写入 header (会话路由) 和 body (缓存键)。
**推理框架可统一用这一个值做会话亲和 + KV cache 路由。**

### Codex Responses API 请求完整结构 (真实捕获)

```json
{
  "model": "gpt-5.1",
  "instructions": "You are Hermes Agent, an intelligent AI assistant...",  // system prompt 独立字段
  "input": [{"role": "user", "content": "codex test"}],                    // 非 messages!
  "reasoning": {"effort": "medium", "summary": "auto"},
  "include": ["reasoning.encrypted_content"],                              // 重放加密推理 (跨轮连续性)
  "store": false,                                                          // Hermes 自管状态, 禁用服务端存储
  "prompt_cache_key": "20260604_110454_eba5e9",
  "stream": true
}
```
header: `session_id: 20260604_110454_eba5e9` + `x-client-request-id: 20260604_110454_eba5e9`

**与 chat_completions 的根本差异** (推理框架需注意):
- 用 `input` 而非 `messages`
- system prompt 走 `instructions` 独立字段, 不在输入项里
- `store: false` — Hermes 不依赖 OpenAI 服务端的 response 存储, 自己管理多轮状态
- `include: [reasoning.encrypted_content]` — 跨轮重放加密推理项, 这是 Codex 上下文连续性的关键

## 三点六、chat_completions 路径下的会话标识 (回答: OpenAI SDK 有 session_id 吗?)

> 核心结论: **session_id 注入与否取决于 ProviderProfile, 不是 OpenAI SDK 本身的能力。**
> 源码: `agent/transports/chat_completions.py:502,512` 把 session_id 传给
> `profile.build_extra_body()` / `build_api_kwargs_extras()`, 由各 provider profile 决定。

### 实测对比 (v0.15.1 真实安装, 端到端 + 直接调用双重验证)

| provider (profile) | model | api_mode | body.session_id | header.x-grok-conv-id |
|--------------------|-------|----------|-----------------|----------------------|
| `custom` (默认 profile) | mock-model | chat_completions | ❌ 无 | ❌ |
| `openrouter` | qwen/qwen-2.5-72b-instruct | chat_completions | ✅ 注入 | ❌ |
| `openrouter` | x-ai/grok-4 | chat_completions | ✅ 注入 | ✅ 注入 (同值) |
| `openrouter` | openai/gpt-5 | **codex_responses** | (改用 `prompt_cache_key`) | ❌ |
| `qwen` | qwen-max | chat_completions | ❌ (需 portal.qwen.ai) | ❌ |

**直接调用验证** (真实安装 `~/.hermes/.../venv` 代码, v0.15.1):
```
OpenRouterProfile.build_extra_body(session_id="SID-123", model="openai/gpt-5")  → {'session_id': 'SID-123'}   ← 注入
OpenRouterProfile.build_extra_body(session_id=None,      model="openai/gpt-5")  → {}
OpenRouterProfile.build_api_kwargs_extras(..., model="x-ai/grok-4")             → extra_headers={'x-grok-conv-id': 'SID-123'}
OpenRouterProfile.build_api_kwargs_extras(..., model="openai/gpt-5")            → {}   (非 grok 无 header)
```

实测 body keys 对比:
```
custom:           ['messages', 'model', 'stream', 'stream_options']                       ← 无会话标识
openrouter:       ['messages', 'model', 'stream', 'stream_options', 'session_id']          ← body.session_id ✅
openrouter+grok:  body.session_id + header.x-grok-conv-id (两者同值)
qwen:             ['messages', 'model', 'max_tokens', 'stream', 'stream_options', 'vl_high_resolution_images']
```

### 关键结论 (v0.15.1, 用户已升级到此版本)

1. **OpenAI 官方 Chat Completions API 本身没有标准 `session_id` 字段。**
   Hermes 走标准 OpenAI 端点 (默认 profile / provider=custom/openai) 时, 请求里
   **不携带任何会话亲和标识** — 只有 `messages/model/stream/stream_options(/tools)`。
   (真实 `hermes` CLI 端到端验证)

2. **OpenRouter (v0.15.1): body 顶层注入 `session_id` (全模型)。** grok 模型额外带 `x-grok-conv-id` header。
   - 这是 v0.13.0→v0.15.1 升级带来的关键变化: v0.13.0 时 OpenRouter **不注入** `body.session_id`,
     升级后 `openrouter/__init__.py` 的 `build_extra_body` 对**所有** OpenRouter 模型都注入。
   - ⚠️ 例外: `openai/gpt-5` 等模型经 OpenRouter 被路由到 `codex_responses` 模式, 此时用
     `body.prompt_cache_key`(=session_id) 代替 `body.session_id`。

3. **OpenRouter 上的 Grok 模型注入 `x-grok-conv-id` header** (v0.13.0 和 v0.15.1 都有):
   ```python
   if session_id and model.startswith(("x-ai/grok-", "xai/grok-")):
       extra_headers["x-grok-conv-id"] = session_id
   ```

4. **Qwen Portal 的 `metadata.sessionId`** 需要 base_url 真为 `portal.qwen.ai`
   (`_is_qwen_portal()` 判定), localhost mock 未触发 — 实测 qwen 只注入了
   `vl_high_resolution_images`, 无 metadata。

5. **Hermes 默认 chat_completions 路径不使用 OpenAI 官方的 `prompt_cache_key` / `user` 字段。**
   OpenAI 官方 Chat Completions 其实支持这两个字段做缓存路由, 但 Hermes 仅在
   Responses API (Codex/xAI) 路径注入 `prompt_cache_key`。

### 对推理框架的意义

| 接入方式 | 会话亲和信号 | 推理框架可用 |
|----------|-------------|-------------|
| 标准 OpenAI Chat Completions (custom/openai) | **无** | 只能靠 prefix 内容 hash 做被动 cache |
| OpenRouter (chat_completions) | `body.session_id` (+grok: header) | 按 session_id 做会话粘性 |
| Codex/xAI (Responses API) | header `session_id`/`x-grok-conv-id` + body `prompt_cache_key` | 按这一个值做粘性+缓存键 |
| Qwen Portal | `body.metadata.sessionId` | 按 sessionId 做粘性 |

**若自建推理框架想从标准 OpenAI Chat Completions 流量获得会话亲和**: 默认拿不到 session_id —
需要 (a) 让用户走 OpenRouter profile, 或 (b) 给 Hermes 加一个自定义 provider profile
在 `build_extra_body` 里注入 `prompt_cache_key`/`session_id`, 或 (c) 推理框架自己按
system prompt + 首条消息的 prefix hash 做被动会话识别 (Hermes 的 system prompt 字节稳定,
prefix hash 可作为隐式会话键)。
