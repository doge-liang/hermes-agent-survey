# Hermes Agent 工具系统与注册表架构深度分析

> **TL;DR(一句话总览)**:Hermes 的"工具系统"是整个 agent 能"动手做事"(读文件、跑命令、调外部服务)的那一层。它用一套**自注册 + 中央注册表 + 薄编排层**的三层结构,把 80 多个零散的工具模块、外部 MCP 服务器、插件工具,统一收拢到一个进程内的全局表里,再由一个很薄的分发器在每次模型要"调工具"时把请求路由过去。围绕这个核心,还挂了一圈配套设施:schema 清理(迁就不同后端的格式洁癖)、渐进式工具披露(工具太多时只暴露三个"搜索/查看/调用"的桥接工具)、懒加载依赖(用到某功能才去装它的 Python 包)、四层审批防御(挡住 `rm -rf /` 这类危险命令)、以及工具结果持久化(防止超长输出撑爆上下文窗口)。
>
> **读者画像**:本报告假设你懂推理服务 / KV cache / LLM 工程,但**完全不了解 Hermes 内部**。所有 Hermes 专有概念都会先用一句话解释再使用。
>
> **关键名词速记**:本报告反复出现的"agent hint"指的是 agent 随推理请求一起下发给后端、用于辅助 KV/调度决策的元数据(详见报告 12);"toolset"(工具集)是 Hermes 对工具的分组单位;"MCP"(Model Context Protocol)是一套让 agent 连接外部工具服务器的标准协议。本报告侧重工具的**注册、发现、分发与治理**机制本身,与 KV/调度信号的提取规格请参阅报告 17。
>
> **源码坐标说明**:本报告所有 `文件名:行号` 坐标均对照用户实际安装的 Hermes 源码树(`~/.hermes/hermes-agent`,版本 v0.15.1)核验过,文中标注的常量、行为与计数均为该树的实测值。

## 一、整体架构:为什么要分成三层

### 1.1 三层各自解决什么问题

工具系统采用**自注册 + 中央注册表 + 薄编排层**三层架构。先用一段话说清楚这三层各自要解决什么问题,后面再逐层展开。

- **最底层:80 多个工具模块(`tools/*.py`)**。每个文件是一个或一组工具的实现(比如读文件、跑 shell、调搜索 API)。它们的特点是"自给自足":每个模块在被 import 的那一刻,就主动把自己**登记**到中央注册表里。这种"模块自己负责把自己注册进系统"的模式,叫**自注册(self-registration)**。好处是加一个新工具不需要去改任何中心化的清单文件——写好模块、确保它被 import 到,就自动生效了。

- **中间层:中央注册表(`tools/registry.py`)**。它是一张进程内的全局表,记录"系统里现在到底有哪些工具、每个工具长什么样、由谁来执行"。所有工具——无论是内置的、来自 MCP 服务器的、还是插件提供的——最终都汇集到这里。它对外提供两个核心能力:`get_definitions`(把所有可用工具的 schema 吐给模型,让模型知道自己能调什么)和 `dispatch`(按名字把一次工具调用路由到对应的执行函数)。

- **最上层:薄编排层(`model_tools.py`)**。"薄"是关键词:它本身不实现任何具体工具,只负责在"模型说要调某工具"和"注册表真正执行"之间,插入一串必要的横切关注点——参数类型纠正、权限审批、插件钩子等。把这些通用逻辑集中在编排层,而不是散落进每个工具,是为了让单个工具实现保持简单。

最上面再由实际的入口程序(仓库根目录的 `run_agent.py`、CLI 入口 `cli.py`、批处理入口 `batch_runner.py`)来驱动整个对话循环,在需要时调用编排层。

> ⚠️ **坐标勘误(报告 17 已修正)**:`run_agent.py` 位于**仓库根目录**,不是 `agent/run_agent.py`。本报告引用的若干 `run_agent.py:行号` 均以仓库根的该文件为准。

整体数据流可以画成下面这张图:

```
tools/*.py (80+ 工具模块)
    ↓ import 时调用 registry.register()        ← 自注册
tools/registry.py (中央注册表)
    ↓ get_definitions / dispatch                ← 统一收口
model_tools.py (薄编排层)
    ↓ 被调用                                     ← 横切关注点
run_agent.py / cli.py / batch_runner.py        ← 入口与对话循环
```

### 1.2 这套设计对"接推理后端 / KV 调度"意味着什么

之所以在一份偏 KV 调度的调研里先讲清工具架构,是因为**工具的数量和 schema 直接决定了每个推理请求前缀里"tools 段"有多大**。模型每一轮都要带上所有可用工具的 schema,这部分会成为请求前缀里一段稳定、可复用的内容(正是 KV cache 最该命中的部分)。理解了"工具是怎么被组装进请求的",才能正确估算前缀长度、判断哪些字段稳定可缓存。这一点在报告 15 §4.1 里有实测佐证:在 `toolsets:[]` 这个配置下,真实 CLI 出站请求里有 **29 个工具**(注意:这是环境/配置相关的捕获值,**不是常量**,换配置即变)。

## 二、注册表核心(registry.py)

注册表是整个系统的心脏。它要回答两个问题:"一个工具被登记进来时,需要记住关于它的哪些信息?"(数据结构),以及"怎么保证多个来源同时注册时不出乱子?"(并发与冲突治理)。

### 2.1 ToolEntry:每个工具的"档案"

每登记一个工具,注册表就为它建一条 `ToolEntry` 记录(定义在 `registry.py:77` 的 `class ToolEntry`)。可以把它理解成这个工具的完整档案,字段含义如下:

| 字段 | 说明 |
|------|------|
| `name` | 工具的唯一名称,模型就是用这个名字来点名调用它的 |
| `toolset` | 工具所属的工具集(分组),决定它在哪个开关下被启用 |
| `schema` | 工具的参数定义,采用 JSON Schema(OpenAI function calling 格式),模型据此知道该传什么参数 |
| `handler` | 真正干活的执行函数 |
| `check_fn` | **可用性检查函数**——返回这个工具当前是否可用(例如检测本机是否装了 Docker)。结果带 30 秒 TTL 缓存,避免每次都重复探测 |
| `is_async` | 标记 handler 是不是 `async` 协程,分发时据此决定走同步还是异步桥接 |
| `dynamic_schema_overrides` | **运行时 schema 覆写**——一个零参回调,在每次需要时返回一份 schema 增量。用于 `delegate_task` 这类参数随上下文变化的工具 |

这里有两个字段值得展开:

**`check_fn` 的 30 秒 TTL 缓存**(`registry.py:121` 处 `_CHECK_FN_TTL_SECONDS = 30.0`)。有些工具是否可用,取决于昂贵的外部探测(比如 `tools/terminal_tool.check_terminal_requirements` 要去看终端环境)。如果每次组装工具列表都重新探测一遍,开销很大。于是注册表把 `check_fn()` 的返回值缓存 30 秒(`_check_fn_cached`,`registry.py:126`),30 秒内重复问同一个 `check_fn` 直接返回缓存结果;并且把探测中抛出的异常一律当作"不可用(False)"吞掉,避免一个工具的探测失败拖垮整个列表组装。配置变更后可以调 `invalidate_check_fn_cache()`(`registry.py:144`)主动清空这个缓存。

**`dynamic_schema_overrides` 的用途**。绝大多数工具的 schema 是固定的,登记一次即可。但少数工具(如把任务委派给子 agent 的 `delegate_task`)的合法参数会随当前可用的能力而变。对这类工具,注册表存的不是死 schema,而是一个回调,在真正要把工具暴露给模型时才执行它、拿到最新的 schema 增量。

### 2.2 注册流程与冲突治理

注册一个工具的标准写法很直白——模块在自己内部调一次 `registry.register()`:

```python
registry.register(
    name="read_file",
    toolset="file_tools",
    schema={...},
    handler=read_file_handler,
    check_fn=lambda: True,
    ...
)
```

真正有讲究的是注册表的**冲突治理**。多个来源(内置工具、插件、MCP 服务器)都往同一张表里塞工具,难免重名。如果放任后注册的覆盖先注册的,一个恶意或写错的插件就能悄悄"偷换"掉一个内置工具(比如把 `write_file` 换成一个会偷偷上传文件的实现)。Hermes 用下面几条规则来防这件事(对应 `registry.register` 体内 `registry.py:240` 起的逻辑):

- **影子工具检测(shadow detection)**:如果新注册的工具与已有工具**同名但 toolset 不同**,默认**拒绝**注册(`registry.py:280` 起的 reject 分支)。"影子工具"指的就是这种"用同名顶替掉另一个工具集里已有工具"的情况——默认禁止,正是为了挡住上面说的偷换攻击。

- **MCP 工具之间的例外**:当冲突双方都是 MCP 工具时,允许覆盖(MCP 服务器重连、工具列表刷新时,同名工具被新版本替换是正常现象)。

- **显式 `override=True`**:插件如果**确实**想替换某个内置工具,必须在 `register()` 时显式传 `override=True`(`registry.py:271`)。这是一个需要主动声明的"我知道我在覆盖"的开关,并且会在 `agent.log` 里以 INFO 级别记一条日志,让这次覆盖**可审计**。

- **线程安全**:注册表用一把可重入锁 `threading.RLock`(`registry.py:161`)保护所有读写,并维护一个 `_generation` 版本号(`registry.py:167`),每次表内容变更就自增(`registry.py:218`)。这个版本号让上层能廉价地判断"工具表自上次以来变没变过"——不必逐项 diff,比一下版本号即可。

### 2.3 工具发现(startup):用 AST 解析做快速预筛

启动时,系统需要把 `tools/` 目录下所有"会注册工具"的模块 import 一遍,以触发它们的自注册。问题是:import 一个 Python 模块是有副作用、有开销的,不能盲目把目录下所有 `.py` 都 import。于是 `discover_builtin_tools()`(`registry.py:57`)先用 **AST 解析**做一道快速预筛。

**AST**(Abstract Syntax Tree,抽象语法树)是把源码解析成的结构化语法树。Hermes 用 `ast.parse(source, ...)`(`registry.py:50`)把每个候选模块的源码解析成 AST,**不执行**它,只静态扫一眼里面有没有出现 `registry.register()` 调用。只有确认包含注册调用的模块,才用 `importlib.import_module()`(`registry.py:70`)真正 import 进来触发注册。这样既保证了"漏不掉真正的工具模块",又避免了"为了发现而 import 一堆无关模块"。

发现过程会排除三个文件:`__init__.py`、`registry.py`、`mcp_tool.py`(见 `registry.py:63` 的排除集合)。前两个是包/注册表自身,后者(MCP 集成)不走这条静态发现路径(它在运行时按服务器配置动态注册,见第四节)。

## 三、工具分发:从"模型说要调"到"真的调了"

### 3.1 同步 / 异步统一:让调用方不必关心 handler 是不是协程

工具的 handler 有的是普通同步函数,有的是 `async` 协程。但分发器希望对外提供一个**统一**的调用入口,让调用方不用关心这个区别。这就是 `registry.dispatch(name, args)`(`registry.py:390`)做的事:

- handler 是同步的:直接调用。
- handler 是异步的:通过 `_run_async()` 桥接执行(`registry.py:402` 处 `from model_tools import _run_async`)。`_run_async` 是异步桥接的**唯一真源**(single source of truth,定义在 `model_tools.py:39` 一带),`dispatch` 与上层编排都复用它,保证全系统只有一处处理"在同步代码里跑协程"这件麻烦事。

之所以麻烦,是因为"在哪个事件循环里跑这个协程"取决于当前所处的执行环境。`_run_async()` 区分三种运行模式:

1. **CLI 主线程**:用一个持久化的事件循环来跑。
2. **Worker 线程(并行执行场景)**:用线程局部(thread-local)的持久化循环——每个 worker 线程有自己的循环,互不干扰。
3. **已经身处一个运行中的事件循环里**:不能在已运行的循环上再 `run_until_complete`(会报错),于是开一个新线程、在新线程里跑这个协程。

把这三种情况都封在 `_run_async` 一处,工具作者写 `async def handler` 时就完全不用操心这些,直接交给系统桥接即可。

### 3.2 handle_function_call():分发主流程的八道关卡

`registry.dispatch` 只是"最后一脚把请求踢给 handler"。真正面向模型的入口是薄编排层里的 `handle_function_call()`(主分发器,`model_tools.py:861`)。它在 dispatch 前后串起一整条横切处理链。按顺序是八步:

```
1. 参数类型强制转换 (coerce_tool_args)
2. Tool Search 桥接工具拦截
3. Agent 循环工具检查 (todo/memory 等拒绝在分发层执行)
4. Pre-tool-call 插件钩子 (可阻止执行)
5. ACP 编辑审批 (write_file/patch 额外检查)
6. 注册表分发 (registry.dispatch)
7. Post-tool-call 插件钩子
8. Transform 钩子 (插件可修改结果)
```

逐道关卡解释如下:

1. **参数类型强制转换(`coerce_tool_args`,`model_tools.py:606`)**。模型生成的参数经常类型不严谨——该传数组却给了个 JSON 字符串、该传列表却给了个裸字符串。这一步按工具 schema 期望的类型对参数做纠正(例如把看起来是 JSON 数组的字符串解析成真数组、把裸字符串包成单元素列表),容错地让后续执行不至于因小格式问题就崩。

2. **Tool Search 桥接工具拦截**(`model_tools.py:904` 起)。如果调用的是渐进式工具披露引入的三个桥接工具(`tool_search` / `tool_describe` / `tool_call`,见第六节),在这里就被拦下来交给 `tools/tool_search.py` 的专用 dispatch 处理(`dispatch_tool_search` 等),不走常规注册表路径。

3. **Agent 循环工具检查**。某些工具属于"agent 主循环"专有,不能在通用分发层直接执行——它们需要主循环的上下文。这组工具记在 `_AGENT_LOOP_TOOLS = {"todo", "memory", "session_search", "delegate_task"}`(`model_tools.py:556`),分发层遇到它们会拒绝执行(原文所说的"todo/memory 等",完整集合即此四个)。

4. **Pre-tool-call 插件钩子**。在真正执行前,给插件一个观察/否决的机会——插件可以在这里**阻止**某次工具执行。

5. **ACP 编辑审批**。ACP(Agent Client Protocol)是 Hermes 与 Zed 等编辑器集成时用的协议。当工具会改动文件(`write_file` / `patch`)时,这里要额外走一道编辑审批(`acp_adapter/edit_approval.py` 的 `maybe_require_edit_approval`,经 `model_tools.py:1028` 调用)。只有 ACP 会话才通过 ContextVar 绑定了"审批请求方";CLI/gateway 路径没有这个绑定,因此这道关卡只对 ACP 会话生效(`model_tools.py:1024` 注释说明了这一点)。

6. **注册表分发(`registry.dispatch`)**。前面都放行后,这一步才把调用真正交给 handler 执行,并测量 dispatch 时延。

7. **Post-tool-call 插件钩子**。执行完后通知插件(观察用)。这条路径做了快路径优化:没有任何插件监听时,代价只是一次 dict 查找(`model_tools.py:831` 一带的 `has_hook("post_tool_call")` 判断),不监听就几乎零开销。

8. **Transform 钩子**。最后,插件还能在这里**修改**工具返回的结果再交回给模型。

### 3.3 顺序 vs 并发执行:单次 dispatch 是顺序的,并发由上层管

要分清两个层面:

- **单个工具的 dispatch 本身是顺序的**——`registry.dispatch` 一次只处理一个调用,从头跑到尾。
- **多个工具调用的并发**由上层 `run_agent.py` 负责。当一轮模型输出里包含多个工具调用、且允许并行时,`run_agent.py` 的 `_execute_tool_calls_concurrent()`(`run_agent.py:4844`)接管。它实际上是个转发器,真正的并发执行逻辑在 `agent/tool_executor.py` 的 `execute_tool_calls_concurrent`。

并发用 `ThreadPoolExecutor` 实现,worker 数上限是 **8**(`agent/tool_executor.py:52` 处 `_MAX_TOOL_WORKERS = 8`;实际 `max_workers = min(可运行的调用数, _MAX_TOOL_WORKERS)`,见 `tool_executor.py:483`)——也就是说最多同时跑 8 个工具,但不会超过本轮实际待跑的工具数。

并发不是无条件开的:对于 MCP 工具,要看它所属服务器有没有声明 `supports_parallel_tool_calls: true`(`tools/mcp_tool.py:3519`/`3640`)。这是一个**按服务器**的显式 opt-in——只有服务器明确声明"我的工具可以并发跑"时,该服务器的工具才会被并行调度;否则保守地串行执行,避免对不支持并发的外部服务造成竞争问题。

## 四、MCP 工具集成(mcp_tool.py,~3900 行)

`tools/mcp_tool.py` 是工具系统里最庞大的单个模块(**3915 行**),专门负责把外部 MCP 服务器接入进来。**MCP(Model Context Protocol)**是一套标准协议,让 agent 能连接到独立运行的"工具服务器",把这些服务器提供的工具当成自己的工具来用。

### 4.1 架构:专用后台事件循环 + 每服务器一个长存活 Task

MCP 通信全是异步、长连接的,不能阻塞主线程。于是 Hermes 起一个**专用的后台事件循环**,每个 MCP 服务器在其中作为一个**长期存活的 `asyncio.Task`** 跑着,负责维持连接、监听通知、转发调用。

支持三种传输方式(`mcp_tool.py:5` 一带的文档与 `mcp_tool.py:51` 起的能力说明):

- **Stdio**:把 MCP 服务器作为**子进程**拉起,通过标准输入输出通信(适合本地命令行式的服务器)。
- **HTTP / StreamableHTTP**:走 HTTP 的流式协议(`LATEST_PROTOCOL_VERSION = "2025-03-26"`,`mcp_tool.py:182`)。
- **SSE**(Server-Sent Events):面向使用 SSE 协议的 MCP 服务器(`mcp_tool.py:52`),需在配置里写 `transport: sse`。

### 4.2 MCPServerTask 的生命周期

每个服务器 Task 从连接到清理,走一条固定的生命周期:

```
连接 → 初始化(握手) → 工具发现(list_tools) → 注册到全局 registry
  → 等待 lifecycle 事件 → 断开 → 清理(注销工具)
```

含义:先建立连接并完成协议握手,然后向服务器请求它提供的工具清单(`list_tools`),把发现的每个工具**注册进全局注册表**(于是它们和内置工具一样可被模型调用);之后这个 Task 进入"等事件"状态,直到连接断开,最后清理时**注销**之前注册的这些工具(避免一个已下线服务器的工具还残留在表里)。

### 4.3 动态工具发现与断线重连:为不稳定的外部依赖做韧性设计

外部 MCP 服务器是不可控的——可能临时改了工具列表,也可能网络抖动断线。Hermes 为此做了一整套韧性机制:

- **动态工具发现**:服务器可以主动发一个 `notifications/tools/list_changed` 通知(`mcp_tool.py:223` 一带支持该通知类型),告诉 Hermes "我的工具列表变了"。Hermes 收到后做**差异更新**——只增删变化的部分,而不是粗暴地整表重建。

- **指数退避重连**:连接断了会自动重连,最多重试 **5 次**(`_MAX_RECONNECT_RETRIES = 5`,`mcp_tool.py:262`),每次失败后等待时间指数增长,但**退避上限封在 60 秒**(`_MAX_BACKOFF_SECONDS = 60`,`mcp_tool.py:264`)——避免无限制地越等越久。

- **断路器(circuit breaker)**:这是从分布式系统借来的模式。当对某服务器**连续失败 3 次**(`_CIRCUIT_BREAKER_THRESHOLD = 3`,`mcp_tool.py:2004`)后,断路器"打开"(open):此后对它的调用直接短路返回失败,不再实际尝试(免得一个已挂的服务器拖慢每次调用)。冷却 **60 秒**(`_CIRCUIT_BREAKER_COOLDOWN_SEC = 60.0`,`mcp_tool.py:2005`)后转为"半开"(half-open):放一个探测请求过去试水,成功就恢复正常,失败就重新打开、再冷却(`mcp_tool.py:1992` 起的状态机注释)。

- **Keepalive(保活)**:每 **3 分钟**(`_KEEPALIVE_INTERVAL = 180`,即 180 秒,`mcp_tool.py:1311`)主动发一次轻量的 `list_tools()` 当心跳探活。这个间隔特意设得比常见的负载均衡 / NAT 空闲超时(通常 300–600 秒)短(`mcp_tool.py:1309` 注释解释了原因),好在中间设备掐断空闲连接之前先把它"盘活"。保活失败则触发重连。

### 4.4 安全:外部工具是不可信输入,处处设防

把外部服务器接进来,等于把不可信代码和不可信文本引入了 agent。Hermes 在多处设防:

- **OAuth 2.1 PKCE 认证 + 会话过期自动恢复**。对需要授权的 MCP 服务器,走 OAuth 2.1 的 PKCE 流程(`mcp_tool.py:1576` 起,经 `tools/mcp_oauth_manager.py` 的中央 `MCPOAuthManager` 统一管理);会话过期时能自动恢复授权。

- **凭证从错误消息里剥离(`_sanitize_error`)**。错误消息常常会把 URL 里的 token、key 等敏感串带出来打进日志。`_sanitize_error`(配套正则在 `mcp_tool.py:274` 起)用一组模式把 GitHub PAT(`ghp_...`)、OpenAI 式 key(`sk-...`)、以及 `token=`/`key=`/`API_KEY=`/`password=`/`secret=` 等查询参数统统替换掉,防止凭证泄进日志。

- **MCP 工具描述的提示注入检查**。外部服务器返回的工具**描述文本**本身可能是攻击载荷(在描述里偷偷写"忽略你之前的指令…")。Hermes 会扫描工具描述里的**提示注入(prompt injection)模式**(`mcp_tool.py:340` 起定义模式、`mcp_tool.py:368` 的扫描函数),识别如"system prompt 注入""role 标签注入"等可疑迹象。

- **Sampling 支持的双重治理**。MCP 协议允许服务器反过来请求 agent 帮它做 LLM 补全(`sampling/createMessage`)。这是把控制权交给外部服务器,风险大,因此 Hermes 给它套了两道治理:**滑动窗口速率限制**(`_check_rate_limit`,滑动窗口实现,`mcp_tool.py:778`)防止服务器疯狂索取补全;以及**工具循环治理**——每次 sampling 引发的工具调用轮数上限由 `max_tool_rounds`(默认 5,`mcp_tool.py:760`)控制,设为 0 即彻底禁用 sampling 里的工具循环。

## 五、Schema 清理器(schema_sanitizer.py)

不同推理后端对工具 schema 的格式"洁癖"程度不一样:有的宽松、什么都收;有的(尤其本地后端)严格,遇到不认识的关键字直接拒整个请求。`tools/schema_sanitizer.py` 的职责就是在把工具 schema 发出去之前,**预先把它"洗"成各家都能接受的样子**。它分主动清理和被动清理两类。

### 5.1 主动清理(无条件做)

这几项是不管发往哪个后端都先做掉的:

- **裸字符串 schema → 规范对象**:把不合规范的裸字符串形态的 schema,补成 `{"type": "object", "properties": {}}` 这样的合法对象形态(保证顶层一定是带 `properties` 的 object,见 `schema_sanitizer.py:72` 起)。
- **可空类型规范化**:把 `type: ["string", "null"]` 这种"用类型数组表示可空"的写法,改成单类型 + `nullable: true` 的写法。
- **`strip_nullable_unions`**:折叠 `anyOf` / `oneOf` 里那些**只是为了允许 null** 的分支(`schema_sanitizer.py:86`,并保留 `nullable: true` 提示)。换句话说,如果一个 union 存在的唯一目的就是"这个字段可以是 null",就把这个啰嗦的 union 压平成"单类型 + 可空"。
- **顶层禁止组合器**:把顶层的 `allOf`、`anyOf`、`oneOf`、`enum`、`not` 这五个关键字剥掉(`_TOP_LEVEL_FORBIDDEN_KEYS = ("allOf", "anyOf", "oneOf", "enum", "not")`,`schema_sanitizer.py:96`)。原因是 **Codex 后端**(`chatgpt.com/backend-api/codex`)更严格,要求函数参数 schema 顶层不得出现这些组合器(`schema_sanitizer.py:102` 注释)。**关键限定**:只剥**顶层**;嵌套在某个属性内部的组合器**保留不动**(`schema_sanitizer.py:113`)。剥掉顶层组合器只是丢掉一个提示,**不改变哪些参数值是合法的**。

### 5.2 被动清理(仅当 llama.cpp 报错时才触发)

下面两项不是无条件做,而是**遇到 llama.cpp 这类本地后端因 schema 报错时**才补做(`schema_sanitizer.py:3` 文档说明 llama.cpp 的 `json-schema-to-grammar` 对某些关键字很敏感,别家默默接受的东西能让它整个请求失败):

- **`strip_pattern_and_format()`**:移除它不支持的正则 `pattern` 与 `format` 关键字。
- **`strip_slash_enum()`**:移除 enum 取值里含 `/` 的项(这类值会让它的语法生成出问题)。

之所以做成"被动、报错才触发",是因为这些关键字对支持它们的后端是有用的(能约束模型输出),没必要为了迁就一个后端就对所有请求一律剥掉——只在确实撞上 llama.cpp 报错时,才退而求其次地清掉它们再重试。

## 六、渐进式工具披露(tool_search.py):工具太多时只暴露三把"钥匙"

当 MCP 工具和插件工具堆得很多时,把**全部**工具 schema 都塞进每个请求会有两个坏处:挤占上下文窗口、且让模型在一长串工具里挑花眼。Hermes 的对策叫**渐进式工具披露(progressive tool disclosure)**:把非核心工具**藏起来**,只在请求里放三个"桥接工具",让模型先用它们去**搜索、查看、调用**那些被延迟(deferred)的工具。

| 工具 | 功能 |
|------|------|
| `tool_search` | 按关键词搜索可用工具,用 **BM25** 排名返回最相关的几个(`TOOL_SEARCH_NAME`,`tool_search.py:43`;BM25 打分见 `tool_search.py:347`) |
| `tool_describe` | 加载指定工具的完整参数 schema(只有真要用某工具时才把它的 schema 拉进上下文) |
| `tool_call` | 调用某个被延迟的工具 |

**BM25** 是信息检索里的经典文本相关性打分算法。这里把每个工具的名字(并把 snake_case / 点分名拆成词,见 `tool_search.py:302`)和描述当作"文档"建索引,模型给出关键词时按 BM25 算相关性、返回 top 几个候选——相当于给工具集做了个搜索引擎。

**激活门控**有三档(配置项 `enabled`,`tool_search.py:67`):

- `off`:无条件**不**启用渐进式披露(所有工具照常全量暴露)。
- `on`:无条件启用。
- `auto`(默认):仅当**可延迟工具的 schema token 量超过模型上下文窗口的一定比例**才启用。这个比例由 `threshold_pct` 控制,默认 **10%**(`tool_search.py:68`/`:83`)。文档给了个直观例子(`tool_search.py:221`):一个 200K 上下文、10% 阈值的模型,大约在工具 schema 累计到 **20K token** 时翻转启用。

这样设计的好处:工具不多时(没越过阈值)完全无感,照常全量给;只有真到了"工具 schema 多到开始挤占上下文"时,才自动切换成"搜索-查看-调用"的按需披露模式。

## 七、懒加载依赖(lazy_deps.py):用到才装,且只在安全边界内装

很多可选功能(各种 TTS/STT 引擎、特定 provider SDK、平台连接器)依赖额外的 Python 包。全部预装会让安装臃肿;于是 Hermes 采用**懒加载(lazy install)**:用到某功能时才去装它的依赖。这件事由 `tools/lazy_deps.py` 管,核心是一张**白名单** `LAZY_DEPS`(`lazy_deps.py:77`),按功能命名空间组织:

- `provider.*`(如 `provider.anthropic`、`provider.bedrock`、`provider.azure_identity`)——各推理 provider 的 SDK;
- `search.*`、`tts.*`、`stt.*`、`image.*`、`memory.*`——搜索、语音合成、语音识别、图像、记忆等功能;
- `platform.*`(telegram、discord、slack、matrix…)——各 IM 平台连接器;
- `terminal.*`、`skill.*`、`tool.*`——终端、技能、工具相关依赖。

**`ensure()` 安装流程**(`lazy_deps.py:418` 的 `ensure(feature, *, prompt=True)`):spec 验证 → 安全检查 → 用户确认(TTY)→ **uv → pip → ensurepip** 梯次安装 → 验证。逐步解释:

1. **spec 验证**:用保守的正则(`lazy_deps.py:181`)和 `packaging.specifiers` 校验包名/版本约束的格式合法。
2. **安全检查**:确认这个 spec 在 `LAZY_DEPS` 白名单内。
3. **用户确认(TTY)**:当 `prompt=True` 且 stdin 是 TTY(交互终端)时,先问用户确认再装(`lazy_deps.py:425`)。非交互调用方(gateway、cron、batch)拿到 `prompt=False`,跳过确认,改由配置开关决定(见下)。
4. **梯次安装**:优先用 `uv`(快、且不需要 venv 里有 pip,`lazy_deps.py:356`);`uv` 不可用则退回 `python -m pip`,必要时先用 `ensurepip` 引导出 pip(`lazy_deps.py:370` 起)。这就是注释里说的 "**uv → pip → ensurepip ladder**"(`lazy_deps.py:345`)。
5. **验证**:装完确认能正常 import。

**安全模型**(`lazy_deps.py:20` 起的文档):

- **仅 venv 内安装**——绝不碰系统 Python(`lazy_deps.py:28`)。
- **仅按 PyPI 包名安装**——spec 只能是 `"package>=1.0,<2"` 这类形式(`lazy_deps.py:29`)。
- **白名单约束**——只有出现在 `LAZY_DEPS` 里的 spec 才允许装(`lazy_deps.py:32`),杜绝任意包注入。
- **离线检测**——安装失败(离线、镜像挂了、PyPI 404 / 包被隔离)时,以 `FeatureUnavailable` 异常如实抛出真实的 pip stderr,不静默吞错(`lazy_deps.py:39`)。
- **可整体关闭**——配置 `security.allow_lazy_installs: false`(默认 true)即可禁用全部懒安装(`lazy_deps.py:35`),适合离线或强安全环境把自己钉死在"装好的就这些"。

> 关联实测:报告 15 的契约 4 正是利用 `security.allow_lazy_installs: false` 来关掉懒安装,避免测试子进程在首次用到 edge-tts 等可选工具时触发下载而卡死。

## 八、审批机制(approval.py):四层防御挡住危险命令

agent 能跑 shell 命令,就意味着它能造成真实破坏(删库、格盘、把机器搞死机)。`tools/approval.py` 用**四层递进**的防御来把关,从"无条件硬挡"到"让 LLM 帮忙判断 + 人确认"逐级放宽:

1. **Hardline(硬线):无条件阻止,不可绕过**。最危险的那批操作直接写死禁止——`rm -rf /`、`mkfs`(格式化文件系统,`approval.py:255`)、fork bomb(`:()  {  :  |  :  &  }  ;  :` 这种把进程表撑爆的炸弹,`approval.py:260`)等。这些模式记在 `HARDLINE_PATTERNS`(`approval.py:249`),**任何模式(包括 yolo / 自动批准)都绕不过**。

2. **Sudo stdin 守卫**:专门阻止用 `sudo -S`(从标准输入喂密码)来做密码猜测攻击(`approval.py:287` 起;当未配置 `SUDO_PASSWORD` 时,任何显式 `sudo -S` 都被拦)。

3. **危险模式:47 个正则**。一批"危险但有时合法"的命令(`chmod 777`、SQL `DROP`、`curl | sh`、`git reset --hard` 等)由 `DANGEROUS_PATTERNS` 的 **47 个正则**匹配(`approval.py:275` 注释明确写了 "12 HARDLINE + 47 DANGEROUS patterns")。它们与 Hardline 的区别是:**在 yolo 模式下可以放行**(`approval.py:228` 注释),即它们不是无条件禁止,而是触发审批。

4. **智能审批 / 交互确认**:命中危险模式后,走一个更聪明的判断——由辅助 LLM 评估这次命令的风险,给出 **APPROVE / DENY / ESCALATE** 三种裁决,再叠加用户确认。`ESCALATE`(升级)表示"模型也拿不准,交给人定夺"。

这四层的设计哲学是**分层、就地拦截**:最危险的根本不给任何人开口子(Hardline);中等危险的允许在明确授权下放行,但默认要过审批;其余靠 LLM+人协同判断。把审批逻辑集中在 `approval.py` 而非散进各工具,也保证了"一处加固、全局生效"。

## 九、工具结果持久化(tool_result_storage.py):防止超长输出撑爆上下文

工具的输出可能极长(一次 `grep` 出几万行、一个 API 返回几 MB JSON)。如果原样塞回对话历史,几次就把上下文窗口撑爆了。`tools/tool_result_storage.py` 用**三层防御**应对(`tool_result_storage.py:3` 文档明确"operates at three levels"):

1. **工具内部预截断**(first line of defense,`tool_result_storage.py:6`):很多工具在**返回之前**就自行把输出截断到合理长度。这是第一道关,从源头控制单个结果的体量。

2. **单结果持久化**(`maybe_persist_tool_result`,`tool_result_storage.py:9`):一个工具返回后,如果它的输出**超过该工具注册的阈值**,就把完整内容**通过 stdin 写进沙箱里的临时文件**(`_write_to_sandbox`,`tool_result_storage.py:78`——之所以走 stdin 而不是拼进命令行,是因为内容常常超过命令行约 128 KB 的长度上限,走 stdin 才没有这个天花板,`tool_result_storage.py:85`),然后在上下文里把这段输出**替换成一段摘要 + 持久化标记**(`<persisted-output>` 标签,`tool_result_storage.py:37`)。模型看到的是"内容已存盘"的指针,需要时可以再去取,而不是把全文顶在上下文里。

3. **回合总预算**(`tool_result_storage.py:20`):前两层管的是"单个结果太大"。但还有一种情况——单看每个结果都没超阈值,可一回合里**很多中等大小的结果加起来**也能撑爆上下文。于是设一个**回合总预算** `MAX_TURN_BUDGET_CHARS`(**200K** 字符,`tool_result_storage.py:20`):当本回合所有工具结果累计超过它时,把**最大的那些尚未持久化的结果**溢出(overflow)到磁盘,直到总量回落到预算之内。

三层各管一个失效模式——单工具自律、单结果兜底、整回合总量控制——叠起来才能稳妥地把上下文占用压在安全线内。
