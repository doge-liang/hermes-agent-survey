# Hermes Agent 工具系统与注册表架构深度分析

## 一、整体架构

工具系统采用**自注册 + 中央注册表 + 薄编排层**三层架构：

```
tools/*.py (80+ 工具模块)
    ↓ import时调用 registry.register()
tools/registry.py (中央注册表)
    ↓ get_definitions / dispatch
model_tools.py (薄编排层)
    ↓ 被调用
run_agent.py / cli.py / batch_runner.py
```

## 二、注册表核心 (registry.py)

### 2.1 ToolEntry 数据结构

| 字段 | 说明 |
|------|------|
| `name` | 工具唯一名称 |
| `toolset` | 所属工具集 |
| `schema` | JSON Schema (OpenAI function calling格式) |
| `handler` | 执行函数 |
| `check_fn` | 可用性检查（如检测Docker是否安装），TTL缓存30秒 |
| `is_async` | handler是否为async |
| `dynamic_schema_overrides` | 运行时schema覆写（用于delegate_task等） |

### 2.2 注册流程

```python
registry.register(
    name="read_file",
    toolset="file_tools",
    schema={...},
    handler=read_file_handler,
    check_fn=lambda: True,
    ...
)
```

保护逻辑：
- **影子工具检测**：同名工具若toolset不同，默认拒绝注册
- MCP工具例外：两者都是MCP工具时可覆盖
- `override=True`：插件可替换内置工具
- 线程安全：`threading.RLock` 保护 + `_generation` 版本号

### 2.3 工具发现 (startup)

`discover_builtin_tools()` 使用**AST解析**快速判断模块是否包含 `registry.register()`：
- 扫描 `tools/` 目录所有 `.py` 文件
- 用AST解析确认后通过 `importlib.import_module()` 导入触发注册
- 排除 `__init__.py`、`registry.py`、`mcp_tool.py`

## 三、工具分发

### 3.1 同步/异步统一

`registry.dispatch(name, args)` 统一处理：
- 同步handler：直接调用
- 异步handler：通过 `_run_async()` 桥接

`_run_async()` 三种运行模式：
1. CLI主线程：持久化事件循环
2. Worker线程（并行执行）：线程局部持久化循环
3. 已有运行中循环的上下文：新线程中运行

### 3.2 handle_function_call() 分发流程

```
1. 参数类型强制转换 (coerce_tool_args)
2. Tool Search桥接工具拦截
3. Agent循环工具检查 (todo/memory等拒绝在分发层执行)
4. Pre-tool-call 插件钩子 (可阻止执行)
5. ACP编辑审批 (write_file/patch额外检查)
6. 注册表分发 (registry.dispatch)
7. Post-tool-call 插件钩子
8. Transform 钩子 (插件可修改结果)
```

### 3.3 顺序 vs 并发执行

- 单工具dispatch是顺序的
- 并发执行由 `run_agent.py` 的 `_execute_tool_calls_concurrent()` 管理
- `ThreadPoolExecutor`（最多8 worker）
- MCP工具可声明 `supports_parallel_tool_calls: true`

## 四、MCP工具集成 (mcp_tool.py, ~3900行)

### 4.1 架构

专用后台事件循环，每个MCP服务器作为长期存活的 `asyncio.Task`。

三种传输：**Stdio** (子进程) / **HTTP/StreamableHTTP** / **SSE**

### 4.2 MCPServerTask生命周期

```
连接 → 初始化(握手) → 工具发现(list_tools) → 注册到全局registry
  → 等待lifecycle事件 → 断开 → 清理(注销工具)
```

### 4.3 动态工具发现与断线重连

- 服务器可发送 `notifications/tools/list_changed` 触发差异更新
- 指数退避重连：最多5次重试，退避上限60秒
- 断路器：连续3次失败后open，60秒冷却后半开探测
- Keepalive：每3分钟发送 `list_tools()` 探测健康

### 4.4 安全

- OAuth 2.1 PKCE认证 + 会话过期自动恢复
- 凭证模式从错误消息剥离（`_sanitize_error`）
- MCP工具描述检查提示注入模式
- Sampling支持：滑动窗口速率限制 + 工具循环治理

## 五、Schema清理器 (schema_sanitizer.py)

**主动清理：**
- 裸字符串schema → `{"type": "object", "properties": {}}`
- `type: ["string", "null"]` → 单类型 + `nullable: true`
- `strip_nullable_unions`：折叠 `anyOf`/`oneOf` 中的null分支
- 顶层禁止组合器：`allOf`、`anyOf`、`oneOf`、`enum`、`not`（Codex后端拒绝）

**被动清理（仅llama.cpp报错时触发）：**
- `strip_pattern_and_format()`：移除不支持的正则/格式关键字
- `strip_slash_enum()`：移除含 `/` 的enum值

## 六、渐进式工具披露 (tool_search.py)

当MCP和插件工具过多时，将非核心工具替换为三个桥接工具：

| 工具 | 功能 |
|------|------|
| `tool_search` | 按关键词搜索（BM25排名） |
| `tool_describe` | 加载指定工具的参数schema |
| `tool_call` | 调用指定延迟工具 |

激活门控：`off`/`on`/`auto`（可延迟工具token超过上下文窗口10%时激活）

## 七、懒加载依赖 (lazy_deps.py)

`LAZY_DEPS` 白名单按功能命名空间组织：
- `provider.*` (anthropic, bedrock, azure_identity)
- `search.*`, `tts.*`, `stt.*`, `image.*`, `memory.*`
- `platform.*` (telegram, discord, slack, matrix...)
- `terminal.*`, `skill.*`, `tool.*`

**ensure()流程：** spec验证 → 安全检查 → 用户确认(TTY) → uv→pip→ensurepip梯次安装 → 验证

安全模型：仅venv内安装、仅PyPI包名、白名单约束、离线检测

## 八、审批机制 (approval.py)

四层防御：
1. **Hardline**：无条件阻止（rm -rf /、mkfs、fork bomb）— 不可绕过
2. **Sudo stdin守卫**：阻止sudo -S密码猜测
3. **危险模式**：47个正则（chmod 777、SQL DROP、curl|sh、git reset --hard等）
4. **智能审批/交互确认**：辅助LLM评估 → APPROVE/DENY/ESCALATE → 用户确认

## 九、工具结果持久化 (tool_result_storage.py)

三层防御防止上下文溢出：
1. **工具内部预截断**：各工具自行截断输出
2. **单结果持久化**：超过阈值 → stdin传内容写入沙箱临时文件 → 上下文替换为摘要
3. **回合总预算**：超过 `MAX_TURN_BUDGET_CHARS`(200K) → 溢出最大结果到磁盘
