## 一、核心设计模式

本章讲的是"在代码库里反复出现、跨多个文件复用的套路"。理解了这七个模式,读 Hermes 其余源码会顺畅很多 —— 因为它们是作者解决"既要重构解耦、又要不破坏既有契约"这类工程张力的标准答案。

### 1.1 `_ra()` 模式 —— 重构时保持测试 Patch 兼容性

**背景痛点**:Hermes 早期把几乎所有 agent 逻辑都堆在仓库根的单文件 `run_agent.py` 里(这个文件如今仍有 4831 行,见 §五)。后来为了可维护性,作者把它拆成 `conversation_loop.py`、`system_prompt.py`、`agent_init.py`、`tool_executor.py` 等独立模块。问题在于:**大量已有测试是通过 `mock.patch("run_agent.OpenAI", ...)` 这种字符串路径来打桩(mock)的** —— 它们假设被替换的符号住在 `run_agent` 这个模块命名空间里。一旦代码搬到新模块、改成 `from run_agent import OpenAI` 直接导入,patch 就打不中了(因为新模块持有的是导入那一刻的旧引用,测试替换 `run_agent.OpenAI` 影响不到它)。这是 Python monkeypatch 的经典陷阱。

**解法**:不直接导入,而是每次用到时通过一个惰性函数 `_ra()` 去**实时取** `run_agent` 模块对象,再从它身上取属性。这样测试对 `run_agent.X` 的替换,总能被新模块"看见"。

```python
# agent/conversation_loop.py
def _ra():
    """Lazy reference to ``run_agent`` so patches like
    ``run_agent.handle_function_call`` / ``run_agent._set_interrupt`` /
    ``run_agent.OpenAI`` and have those patches reach this code path.
    """
    import run_agent
    return run_agent

# Usage: _ra().handle_function_call(...) instead of importing directly
```

关键在最后一行的用法注释:调用处一律写成 `_ra().handle_function_call(...)`,而不是顶部 `import` 后直接 `handle_function_call(...)`。`_ra()` 内部的 `import run_agent` 因为 Python 的模块缓存(`sys.modules`)几乎零成本,但它返回的永远是"当前那个被测试可能 patch 过的模块对象"。

此模式在所有从 `run_agent.py` 提取出来的模块中统一使用:`conversation_loop.py`、`system_prompt.py`、`agent_init.py`、`tool_executor.py` 等。它本质上是"用一层间接换取重构自由度,同时零成本维持向后兼容的 patch 契约"。

> 注:`run_agent.py` 位于**仓库根目录**,不在 `agent/` 子目录下。

### 1.2 `ContextVar` 门控模式

**背景**:Hermes 有多条入口路径 —— 交互式 CLI、网关(gateway,把 Hermes 接到 Telegram/IRC 等平台的服务进程)、以及 ACP(Agent Client Protocol,编辑器/IDE 集成用的协议)。有些行为只在某一条路径下需要,比如"编辑文件前弹出审批对话框"只在 ACP 下有意义,CLI 和网关会话不该被它挡住。

`ContextVar` 是 Python 标准库 `contextvars` 提供的"上下文局部变量",它的值随**当前执行上下文**(包括异步任务、线程)隔离,天然适合"只在某条调用链上打开一个开关"的场景。Hermes 用它做**门控**(gating):某段逻辑是否生效,取决于当前上下文有没有把对应的 ContextVar 设上。

```python
# acp_adapter/edit_approval.py
_EDIT_APPROVAL_REQUESTER: ContextVar = ContextVar("edit_approval_requester", default=None)
# Only ACP codepath sets this; CLI/gateway sessions bypass edit approval

# tools/skill_provenance.py
_skill_write_origin: ContextVar = ContextVar("skill_write_origin", default="foreground")
# "foreground" = user-controlled, "background_review" = agent-created
```

- 第一个 `_EDIT_APPROVAL_REQUESTER` 默认是 `None`:只有 ACP 路径会把它设成一个真正的审批请求器对象;CLI/网关会话里它一直是 `None`,于是"编辑审批"这段逻辑被自然跳过(bypass)。这就是"用默认值实现安全的旁路"。
- 第二个 `_skill_write_origin` 记录"这次技能(skill,Hermes 里 agent 可调用的可复用能力包)写入的来源":`"foreground"` 表示用户主动操作,`"background_review"` 表示是 agent 在后台自我审查时创建的。后续安全策略会据此区别对待(详见 §3.2 的 agent-created 策略行)。

### 1.3 双层缓存模式(技能索引)

**背景**:Hermes 启动时要把所有可用技能扫描一遍,拼成一段"技能索引"塞进系统提示词(让 LLM 知道有哪些技能可用)。技能可能有成百上千个文件,每次启动都全量扫盘会很慢。作者用了**两层缓存**来摊销这个成本。

```python
# agent/prompt_builder.py — build_skills_system_prompt()
# 1. 进程内LRU缓存：键=(skills_dir, external_dirs, available_tools, ...), 容量8
# 2. 磁盘快照：~/.hermes/.skills_prompt_snapshot.json
#    按mtime/size文件清单校验有效性
#    有效 -> 直接读取元数据，跳过所有文件I/O
#    无效 -> 完整文件系统扫描 + 回写快照
```

两层的分工:

- **第一层:进程内 LRU 缓存**(LRU = Least Recently Used,最近最少使用淘汰)。缓存键是一个元组 `(skills_dir, external_dirs, available_tools, ...)` —— 即"哪些技能目录、哪些外部目录、当前开了哪些工具"。容量 8,意味着最多记住 8 种不同配置组合的结果。同一进程内反复用相同配置,直接命中内存,零 I/O。
- **第二层:磁盘快照** `~/.hermes/.skills_prompt_snapshot.json`。跨进程(下次启动)复用。它怎么知道快照还有效?靠一份**文件清单**,记录每个技能文件的 `mtime`(修改时间)和 `size`(大小)。启动时拿当前磁盘状态去比对:若清单一致(没有文件被改动),说明快照仍代表最新状态,**直接读快照里的元数据、完全跳过文件 I/O**;若不一致(有文件增删改),才做一次完整文件系统扫描,并把新结果**回写快照**供下次用。

这套"内存 LRU + 磁盘清单校验快照"的组合,是把"启动慢"这个体验问题压到几乎不可感知的标准做法。

### 1.4 三层系统提示词设计(最大化前缀缓存)

**背景与动机**:这是全报告与推理服务最相关的一个设计,值得细讲。LLM 推理后端普遍支持**前缀缓存**(prefix caching):如果两次请求的开头(前缀)逐字节相同,后端可以复用第一次为该前缀算好的 KV cache,跳过这部分的 prefill 计算,显著降低首 token 时延(TTFT)。Hermes 的系统提示词(system prompt,放在对话最前面、定义 agent 身份与规则的一大段文本)正是每个请求都顶着的最长共享前缀。**要让前缀缓存命中,就必须让系统提示词在会话内尽量逐字节稳定。**

但系统提示词里天然混着"会变的东西"(比如当前时间)。如果整段一起重算,哪怕只有时间戳变了,整个前缀的字节都会变,缓存全部作废。作者的解法是**按"易变程度"把系统提示词切成三层**,让稳定的部分永远在前、易变的部分集中到后面,从而保住尽可能长的稳定前缀。

```python
# agent/system_prompt.py
def build_system_prompt_parts(agent, system_message=None):
    return {
        "stable":    # 身份、工具指导、技能索引、环境提示（会话内不变）
        "context":   # 上下文文件(AGENTS.md/.cursorrules) + caller system_message
        "volatile":  # 记忆快照、USER.md、时间戳行（每轮可变，但仅日期精度）
    }
    # 用 \n\n 连接，缓存在 _cached_system_prompt 中
    # 仅在压缩事件时失效 -> 最大化上游前缀缓存命中率
```

三层的语义:

- **`stable`(稳定层)**:agent 身份、工具使用指导、技能索引、环境提示。这些在一次会话内**完全不变**,是最该被前缀缓存复用的部分,所以放最前面。
- **`context`(上下文层)**:项目上下文文件(如 `AGENTS.md`、`.cursorrules` 这类约定文件)加上调用方传入的 `system_message`。它在会话内一般也稳定,但不如 `stable` 那么"天生不变"。
- **`volatile`(易变层)**:记忆快照、`USER.md`(用户画像)、时间戳行。**每轮都可能变**,所以放最后,把变化局限在前缀的尾部。注释特别点出时间戳"仅日期精度"(only date precision)—— 这是个精细取舍:如果时间戳精确到秒,那它每轮都不同、易变层每轮都变;降到只到日期,同一天内多轮对话的 `volatile` 也保持不变,缓存命中窗口更长。

三层用 `\n\n` 连接成完整提示词,缓存在 `_cached_system_prompt` 里,**只在压缩事件(见 §2.5、报告 14)发生时才失效重建**。这样设计的净效果就是注释最后那句:最大化上游(推理后端)的前缀缓存命中率。

> 关于 cache_control 断点如何与系统提示词配合(断点策略 = system + 最后 3 条非系统消息),详见报告 14 与报告 17 §1.1 的 W7。

### 1.5 `ProviderTransport` 桥接模式

**背景**:Hermes 要对接多种 LLM 后端协议,主要有四类(术语统一为 `api_mode`):`chat_completions`(OpenAI 经典 Chat Completions)、`anthropic_messages`(Anthropic Messages API)、`codex_responses`(OpenAI Responses API,codex 系)、`bedrock_converse`(AWS Bedrock Converse)。这些协议的消息格式、工具格式、请求体字段、响应结构各不相同。如果在主循环里写一堆 `if api_mode == ...` 分支,代码会迅速腐烂。

**解法:桥接(Bridge)模式**。定义一个抽象基类 `ProviderTransport`(ABC = Abstract Base Class,抽象基类,强制子类实现指定方法),把"协议相关的差异"全部收敛到它的几个抽象方法里;主循环只面向这个统一接口编程,不关心具体是哪种协议。

```python
# agent/transports/base.py
class ProviderTransport(ABC):
    @property
    @abstractmethod
    def api_mode(self) -> str: ...
    
    @abstractmethod
    def convert_messages(self, messages, **kwargs) -> Any: ...
    @abstractmethod
    def convert_tools(self, tools) -> Any: ...
    @abstractmethod
    def build_kwargs(self, model, messages, tools, **params) -> Dict: ...
    @abstractmethod
    def normalize_response(self, response, **kwargs) -> NormalizedResponse: ...
```

每个抽象方法对应"协议适配"的一个环节:

- `api_mode`:声明这个传输属于上述四类中的哪一类。
- `convert_messages`:把 Hermes 内部统一的消息列表,转换成该协议要求的形状。
- `convert_tools`:同理,转换工具定义。
- `build_kwargs`:组装最终发给后端 SDK 的请求参数字典(模型名、消息、工具、各种 provider 专属参数)。
- `normalize_response`:把后端返回的、各异的响应,**归一化**成一个统一的 `NormalizedResponse`,让上层不必关心来源差异。

每接入一个新协议,只需新增一个子类实现这五个方法,主循环代码一行不改。这是"用抽象隔离变化"的范本(报告 02 对四类传输有更细的对比)。

### 1.6 `PlatformEntry` + 注册表扩展模式

**背景**:Hermes 网关支持把 agent 接到很多聊天平台(Telegram、IRC……,见 §五的平台适配器规模)。新增一个平台,理想情况下应该"只加新文件、不改老代码"(开闭原则)。Hermes 用**注册表**(registry)实现这一点:每个平台把自己的元信息打包成一个 `PlatformEntry`,注册进全局 `platform_registry`。

```python
# gateway/platform_registry.py
platform_registry.register(PlatformEntry(
    name="irc", label="IRC",
    adapter_factory=lambda cfg: IRCAdapter(cfg),
    check_fn=check_requirements,
    validate_config=lambda cfg: bool(cfg.extra.get("server")),
    required_env=["IRC_SERVER"],
    install_hint="pip install irc",
))
# Gateway优先查询注册表；未命中回退到内置if/elif分支
```

`PlatformEntry` 各字段的含义:

- `name` / `label`:内部标识符 / 给人看的显示名。
- `adapter_factory`:一个工厂函数(这里是 lambda),按配置创建该平台的适配器实例 `IRCAdapter(cfg)`。
- `check_fn`:运行前检查依赖是否就绪。
- `validate_config`:校验配置是否合法(这里要求 `cfg.extra` 里必须有 `server`)。
- `required_env`:必需的环境变量(IRC 需要 `IRC_SERVER`)。
- `install_hint`:缺依赖时给用户的安装提示(`pip install irc`)。

最后一行注释点出回退策略:**网关优先查注册表;若没命中,再回退到内置的 if/elif 分支**。这是"新机制(注册表)与旧代码(硬编码分支)共存、渐进迁移"的务实做法 —— 不强求一次性重写所有平台。

### 1.7 凭证冷却状态机

**背景**:Hermes 可以配多个 API 凭证(credential)轮流用(凭证池,credential pool),以分摊速率限制、提升可用性。但凭证会出各种状况:被限流、额度耗尽、token 失效。需要一个**状态机**来管理每个凭证"现在能不能用、什么时候能再用"。

```python
# agent/credential_pool.py
# STATUS_OK → STATUS_EXHAUSTED(冷却: 401=5min, 429=1h, 或provider指定)
# STATUS_DEAD(永久失败: token_invalidated/token_revoked/invalid_grant)
# → 除非通过显式重新认证清除令牌，否则不重新进入轮转
# 
# 软租约并发控制: acquire_lease() / release_lease()
# → 最小租约凭证优先，并发限制 D = max(1, floor(N/2))
```

三个状态:

- **`STATUS_OK`**:正常,可参与轮转。
- **`STATUS_EXHAUSTED`**:暂时不可用,进入**冷却**(cooldown)。冷却时长按错误类型定:`401`(未授权)冷却 5 分钟,`429`(限流)冷却 1 小时,或采用 provider 指定的时长。冷却结束后自动恢复。
- **`STATUS_DEAD`**:永久失败 —— token 被作废/吊销/授权失效(`token_invalidated` / `token_revoked` / `invalid_grant`)。**这种状态不会自动恢复**,除非用户显式重新认证、清除旧令牌,否则该凭证永远不再进入轮转。这是"区分'临时不可用'与'根本坏了'"的关键。

最后两行是**并发控制**:`acquire_lease()` / `release_lease()` 是一对"软租约"(soft lease)操作 —— 借用/归还一个凭证的使用权。策略是"最小租约凭证优先"(谁手上正在用的并发最少,就优先派给谁,实现负载均衡),并设并发上限 `D = max(1, floor(N/2))`(`N` 是凭证总数)—— 即最多让一半凭证同时在用,留出余量避免把所有凭证一起打到限流。

## 二、关键实现细节

本章是一组"单点上的精巧处理":每个都是作者在某个具体问题上,为了正确性或鲁棒性做的一处不显眼但重要的设计。

### 2.1 错误分类的智能歧义消除

**背景**:agent 收到后端的错误后,要决定"该重试还是该放弃、归为哪类错误"。但 HTTP 状态码和错误文案常常**有歧义**,粗暴分类会导致要么白白重试(浪费配额)、要么过早放弃。`error_classifier.py` 专门做消歧。

```python
# agent/error_classifier.py
# 402状态码歧义消除：
#   包含 "usage limit" + 瞬时信号("try again", "resets at") → rate_limit
#   纯粹的 "insufficient credits" → billing
#
# 5xx + 请求校验信号快速失败：
#   检测 unknown_parameter / unsupported_parameter / invalid_request_error
#   → 直接标记为不可重试的 format_error（避免3+次重试轰炸）
```

两个消歧场景:

- **`402` 状态码消歧**:`402 Payment Required` 既可能是"临时用量超限、过一会儿会恢复",也可能是"账户余额不足、得充钱"。区别对待靠看文案里的**瞬时信号**:若含 `"usage limit"` 且伴随 `"try again"` / `"resets at"` 这类"会恢复"的措辞,判为 `rate_limit`(限流,值得等待重试);若只是干巴巴的 `"insufficient credits"`,判为 `billing`(计费问题,重试无意义)。
- **`5xx` + 请求校验信号快速失败**:`5xx` 通常是服务端临时故障、值得重试。但有时后端把"请求参数非法"也错误地报成 `5xx`。作者检测错误体里有没有 `unknown_parameter` / `unsupported_parameter` / `invalid_request_error` 这类**请求格式问题**的标志,一旦命中就**直接标记为不可重试的 `format_error`** —— 因为这种错误重试多少次都一样,与其连打 3 次以上无谓重试(注释里的 "避免 3+ 次重试轰炸"),不如立刻失败、让上层去修参数。

### 2.2 Token 预算的精确控制

**背景**:做上下文压缩(把过长的历史摘要掉)时,有一段历史**绝不能压**(否则会丢掉正在进行的任务)。这段叫"保护区"(protect tail,保护尾部)。怎么界定保护区的边界,既要够大、又不能无限大,是个预算问题。

```python
# agent/context_compressor.py
# 保护区确定：优先使用token预算(反向累加直到超过protect_tail_tokens)
# → protect_tail_count作为硬底线
# 
# 关键保护：_ensure_last_user_message_in_tail()
# → 确保最后一条用户消息总是在尾部，防止活跃任务丢失
```

- **保护区边界**:优先按 **token 预算**界定 —— 从最新一条消息往回(反向)累加 token 数,直到累计超过 `protect_tail_tokens`(尾部保护的 token 配额)为止,这之内的都保护。同时 `protect_tail_count`(保护的消息条数)作为**硬底线**:即使按 token 算应该保护的条数很少,也至少保住这么多条。两者取"更保守"的一方,确保保护区既不会因消息短而保得太少、也不会无界增长。
- **关键兜底** `_ensure_last_user_message_in_tail()`:无论上面怎么算,这个函数都**强制确保最后一条用户消息一定落在保护尾部里**。原因很直接:最后一条用户消息往往就是"当前正在处理的任务",一旦它被压缩进摘要,agent 就可能"忘了自己在干嘛"。这是防止活跃任务丢失的最后一道保险。

> 压缩阈值与两阶段触发机制详见 §2.5 与报告 14、报告 17 §1.2 的 I7–I11。

### 2.3 Nous 速率守卫的真假 429 区分

**背景**:`429 Too Many Requests` 是限流信号,但它有"真假"之分。真限流是"你确实把这个账号的配额打满了,得长时间等";假限流(更准确说是"上游瞬时容量不足")是"后端这一刻忙不过来,马上重试就好"。如果把假 429 当真 429 处理,会误触发"断路器"(circuit breaker,一种"出错就熔断、暂停一段时间"的保护机制),连累其他本来好好的模型。`nous_rate_guard.py` 专门区分这两者。

```python
# agent/nous_rate_guard.py
# is_genuine_nous_rate_limit():
#   信号1: 当前429响应的x-ratelimit-*头中有 remaining==0 且 reset>=60s 的桶
#   信号2: 上次成功响应捕获的速率限制状态中有接近耗尽的桶
#   两个信号都没有 → 归类为上游供应商瞬时容量不足
#                  → 不触发跨会话断路器，不影响其他模型
```

判定函数 `is_genuine_nous_rate_limit()` 看两个证据:

- **信号 1**:当前这个 429 响应的 `x-ratelimit-*` 响应头里,有没有某个"桶"(bucket,限流配额的计量单元)显示 `remaining==0`(剩余配额为 0)**且** `reset>=60s`(要等 60 秒以上才恢复)。这是"配额真打满了"的硬证据。
- **信号 2**:上一次成功响应时捕获到的限流状态里,有没有某个桶已经"接近耗尽"。这是"趋势上确实快满了"的旁证。

**两个信号都没有**,就判定这不是真限流,而是上游供应商的瞬时容量不足。这种情况下**不触发跨会话断路器、不影响其他模型** —— 只是当前这一发请求重试一下,避免一次偶发的拥塞把整个系统的可用性拖下水。

### 2.4 消息去重的双重策略

**背景**:Hermes 有个"后台审查"(background review)机制 —— agent 在后台 fork 出一个分身,复盘刚才做过的事、提炼经验。复盘完要把"新增的审查动作"汇总成一段摘要并入主历史。但后台分身是从主历史**继承**(inherit)而来的,里面已经有一堆和主历史重复的消息,直接全并入会重复。所以要先**去重**,只留真正新增的。

```python
# agent/background_review.py — summarize_background_review_actions()
# 优先通过 tool_call_id 匹配跳过已存在于继承历史中的消息
# 回退到内容相等性比较
# → 只编译真正新增的审查动作摘要
```

去重用**两级策略**:

- **优先按 `tool_call_id` 匹配**:`tool_call_id` 是每次工具调用的唯一标识。如果某条消息的 `tool_call_id` 已经存在于继承来的历史里,说明它是"老的",跳过。这是最可靠的判重方式(有唯一键)。
- **回退到内容相等比较**:有些消息没有 `tool_call_id`(比如纯文本消息)。对这些,退而求其次,**逐条比内容**是否相等来判重。

两级结合,最终只编译出"真正新增的审查动作"的摘要,避免把继承来的旧消息重复写回主历史。

### 2.5 压缩锁防并发

**背景**:接 §2.4,主 agent 和它 fork 出的后台审查分身,**可能同时想压缩同一个会话**。压缩会重写历史,两个进程同时改同一份历史会冲突、损坏状态。需要一把锁来串行化。

```python
# agent/conversation_compression.py
# 基于session_id的原子锁存储在state.db中
# 格式: compression_locks(session_id PK, holder pid:tid:nonce, expires_at)
# → 防止父agent与后台审查fork并发压缩同一会话
```

这把锁的实现要点:

- **基于 `session_id`、存在 `state.db` 里**:`session_id` 是会话的唯一标识;锁存进 SQLite 状态库 `state.db`,因此**跨进程**有效(光用进程内的内存锁挡不住另一个进程)。
- **表结构** `compression_locks(session_id PK, holder, expires_at)`:`session_id` 作主键(`PK`),保证同一会话同一时刻只有一行锁记录,这就是"原子锁"的来源 —— 靠数据库主键唯一约束做互斥。`holder` 记录持锁者身份,格式 `pid:tid:nonce`(进程 id : 线程 id : 随机数),精确到线程并加随机数避免误判。`expires_at` 是过期时间,防止持锁进程崩溃后锁永远不释放(死锁)。

净效果就是注释那句:防止父 agent 与后台审查 fork 并发压缩同一会话。

> 这把锁对应报告 17 §1.2 I12 的"session 轮换"那条信号链的一部分;压缩在 wire 上的可观测特征详见报告 14、报告 15(RC-08)。

### 2.6 Agent 缓存按需失效

**背景**:网关进程要同时服务很多会话,每个会话对应一个 `AIAgent` 对象(创建一个 agent 有开销)。为了复用,网关把 agent 缓存起来;但缓存不能无限长,否则内存爆炸、僵尸会话堆积。需要"容量上限 + 空闲淘汰"两道闸。

```python
# gateway/run.py
# _AGENT_CACHE_MAX_SIZE = 128, _AGENT_CACHE_IDLE_TTL_SECS = 3600
# _enforce_agent_cache_cap(): LRU驱逐
# _session_expiry_watcher(): 空闲>1h的agent被驱逐
```

- `_AGENT_CACHE_MAX_SIZE = 128`:最多缓存 128 个 agent。`_enforce_agent_cache_cap()` 在超额时按 **LRU**(最近最少使用)驱逐最久没用的那个。
- `_AGENT_CACHE_IDLE_TTL_SECS = 3600`:空闲存活上限 3600 秒(1 小时)。后台守护 `_session_expiry_watcher()` 定期巡检,把**空闲超过 1 小时**的 agent 驱逐掉,回收内存。

两道闸互补:容量闸应对"瞬时会话太多",空闲闸应对"会话用完不回收"。

> 这块缓存的"会话键 + 配置签名"(决定何时复用、何时另起炉灶)是报告 17 §1.2 I17 的内部状态信号,只在经网关入站时存在。

### 2.7 文件写入的原子性保证

**背景**:配置文件、状态文件在写入过程中如果进程崩溃或断电,可能留下半截损坏的文件,下次启动直接读崩。并发写同一文件也会互相覆盖。标准解法是**原子替换**(atomic replace):先写到临时文件,刷盘后再用 `rename` 一步到位替换目标 —— `rename` 在同一文件系统上是原子操作,要么完全成功、要么完全没发生,绝不会留下中间态。

```python
# utils.py + hermes_cli/config.py
# atomic_replace(filepath, content): 写入临时文件 → fsync → os.rename
# atomic_yaml_write(): 同上，专门为YAML配置文件设计
# → 防止并发写入冲突、防止写入中断导致文件损坏
```

- `atomic_replace(filepath, content)`:三步 —— 先把内容写进**临时文件**,`fsync`(强制把数据从操作系统缓冲刷到磁盘,确保真的落盘),最后 `os.rename` 原子替换目标。`fsync` 这一步关键:不刷盘的话,`rename` 完成但数据可能还在缓冲里,断电照样丢。
- `atomic_yaml_write()`:同样的机制,专为 YAML 配置文件定制。

两者共同防住"并发写冲突"和"写到一半中断导致文件损坏"两类问题。

### 2.8 多进程认证 Token 同步

**背景**:OAuth 的 `refresh_token`(刷新令牌,用来换取新的访问令牌)在多进程共享场景下有竞争问题:进程 A 手里缓存的 `refresh_token` 可能已经被进程 B 刷新过、失效了(很多 OAuth 实现刷新后旧 refresh_token 即作废)。A 拿着旧的去刷,会失败。

```python
# agent/credential_pool.py — _refresh_entry()
# 池的refresh_token可能与共享存储不同步(另一进程已刷新)
# → 重试逻辑: 刷新失败 → 从共享存储重新同步 → 重试一次
# → 优雅处理跨进程OAuth刷新竞争
```

`_refresh_entry()` 的处理:刷新一旦失败,不立刻报错,而是**先从共享存储重新同步**最新的 token(也许进程 B 刚刷新过、新 token 已经在共享存储里了),拿到最新值后**再重试一次**。这一来一回,优雅化解了跨进程 OAuth 刷新竞争 —— 失败往往不是真失败,只是本地缓存过时了,重新同步即可。

## 三、安全关键实现

Hermes 是个能执行 shell 命令、写文件、跑代码的 agent,安全防线是生死攸关的。本章四节展示它的"纵深防御"(defense in depth):多层独立的关卡,任一层都可能拦下危险操作。

### 3.1 危险命令不可绕过防线

**背景**:agent 可能(被诱导或出错)生成毁灭性命令,比如 `rm -rf /`。Hermes 把命令风险分两档,最危险的一档**任何模式都拦死**,次危险的一档可经审批放行。

```python
# tools/approval.py
# HARDLINE_PATTERNS (12项): rm -rf /、system dir delete、mkfs、
#   dd of=/dev/sd*、fork bomb、kill -9 -1、shutdown/reboot
# → 即使 --yolo 模式也无法绕过
# 
# DANGEROUS_PATTERNS (47项): chmod 777、SQL DROP/DELETE without WHERE、
#   curl|sh、git reset --hard/push --force、docker compose生命周期操作
# → 可通过审批机制绕过
```

两档防线:

- **`HARDLINE_PATTERNS`(12 项)—— 硬红线**:`rm -rf /`、删系统目录、`mkfs`(格式化磁盘)、`dd of=/dev/sd*`(直接写裸盘覆盖)、fork bomb(进程炸弹)、`kill -9 -1`(杀光所有进程)、`shutdown`/`reboot`。这些**即使开了 `--yolo` 模式(用户主动声明"全自动、别问我")也无法绕过** —— 它们造成的破坏不可逆,作者认为没有任何场景值得为它们开口子。
- **`DANGEROUS_PATTERNS`(47 项)—— 危险模式**:`chmod 777`(权限全开)、不带 `WHERE` 的 SQL `DROP`/`DELETE`(会清空整表)、`curl|sh`(把网上脚本直接灌进 shell 执行)、`git reset --hard`/`push --force`(丢本地改动/强推覆盖远端)、docker compose 生命周期操作。这些危险但有合理用途,所以**可以通过审批机制绕过**(用户确认后放行)。

两档的本质区别:硬红线是"绝对禁止",危险模式是"需要人点头"。

### 3.2 技能安全扫描的三态判定

**背景**:技能(skill)可能来自不同信任级别的来源 —— 内置的、官方信任的、社区的、agent 自己创建的。一个来自陌生社区的技能里如果藏着危险代码,不该和内置技能一样无条件放行。Hermes 用"风险三态 × 来源信任度"的策略矩阵来管。

```python
# tools/skills_guard.py
# 判定: critical=dangerous, high=caution, medium/low=safe
# 策略矩阵:
#   builtin: safe=allow, caution=allow, dangerous=allow
#   trusted: safe=allow, caution=allow, dangerous=block
#   community: safe=allow, caution=block, dangerous=block
#   agent-created: safe=allow, caution=allow, dangerous=ask(返回错误)
# community/trusted的dangerous判定即使--force也不可覆盖
```

先把扫描出的风险等级**三态化**:`critical`(严重)归为 `dangerous`,`high`(高)归为 `caution`(警惕),`medium`/`low`(中/低)归为 `safe`(安全)。

然后按**来源**应用不同策略(`allow`=放行,`block`=拦截,`ask`=询问):

| 来源 | safe | caution | dangerous |
|------|------|---------|-----------|
| `builtin`(内置) | allow | allow | allow |
| `trusted`(信任) | allow | allow | block |
| `community`(社区) | allow | block | block |
| `agent-created`(agent 创建) | allow | allow | ask(返回错误) |

读这张表的逻辑:**来源越不可信,门槛越高**。内置技能全放行(自家代码);信任源放行到 caution、但 dangerous 拦死;社区源最严,caution 就开始拦;agent 自己创建的 dangerous 项走 `ask`(以返回错误的形式要求确认)。最关键的一句在表后:**community/trusted 的 dangerous 判定即使加 `--force` 也不可覆盖** —— 这和 §3.1 的硬红线同理,对不可信来源的危险技能不留后门。

> 这里的"来源"与 §1.2 的 `_skill_write_origin`(foreground vs background_review)相关联:后台 agent 创建的技能走 `agent-created` 这一行的策略。

### 3.3 线程上下文传播(Fail-closed)

**背景**:Hermes 会在子线程里执行工具。但"审批回调"、"sudo 回调"这些安全相关的上下文(决定危险命令要不要弹审批)是绑在父线程的 `contextvars` 上的(见 §1.2)。子线程默认**看不到**父线程的 contextvars,如果不显式传过去,子线程里执行危险命令时审批机制会"失灵"—— 这是个静默的安全漏洞。`thread_context.py` 负责把上下文安全地传到子线程。

```python
# tools/thread_context.py — propagate_context_to_thread()
# 1. 调用时(父线程): 快照contextvars.Context + 审批/sudo回调
# 2. 执行时(子线程): ctx.run(_inner) 运行目标 + 安装/清除回调
# 3. Fail-closed: 回调安装失败 → 危险命令审批拒绝执行
```

三步:

1. **父线程快照**:在发起调用时,把当前的 `contextvars.Context`(整个上下文快照)连同审批/sudo 回调一起打包。
2. **子线程重放**:子线程用 `ctx.run(_inner)` 在那份快照上下文里运行目标逻辑,并在进入时**安装**回调、退出时**清除**回调。这样子线程里也能正确触发审批。
3. **Fail-closed(失败即关闭)**:这是最重要的安全原则。万一回调安装失败(上下文没传成功),**不是默默放行,而是让危险命令的审批直接拒绝执行**。"Fail-closed" 的含义就是"出错时倒向安全的一边(拒绝)",而不是 "fail-open"(出错时放行)。对安全系统,这个默认方向是底线。

### 3.4 子进程密钥清理

**背景**:Hermes 执行代码工具时会 fork 子进程。父进程的环境变量里往往有一堆密钥(API key、token、密码)。如果原样继承给子进程,而子进程跑的是不可信代码,密钥就泄露了 —— 不可信代码能直接读 `os.environ` 把它们偷走或外传。

```python
# tools/code_execution_tool.py — _scrub_child_env()
# 在生成子进程前剥离所有含KEY/TOKEN/SECRET/PASSWORD/CREDENTIAL的环境变量
# 仅透传安全前缀(PATH/HOME/USER/LANG/XDG_等)和操作性HERMES_*变量
```

`_scrub_child_env()`(scrub = 擦洗)在 fork 子进程**之前**清理环境:

- **剥离敏感变量**:凡是变量名里含 `KEY`/`TOKEN`/`SECRET`/`PASSWORD`/`CREDENTIAL` 的,一律不传给子进程。这是按名字的黑名单式过滤。
- **只透传安全变量**:运行必需的安全前缀变量(`PATH`/`HOME`/`USER`/`LANG`/`XDG_` 等)和 Hermes 操作性变量(`HERMES_*`)才放行。这是白名单式放行。

黑白名单结合,确保子进程拿到的是一个"够用但不含密钥"的干净环境。

## 四、性能关键实现

本章讲 Hermes 为了"启动快、运行省"做的几处优化。每处都标了量化收益(节省的毫秒数),能直观看出取舍价值。

### 4.1 懒加载 OpenAI SDK(~240ms 节省)

**背景**:`from openai import OpenAI` 这一句 import 本身就要约 240ms(SDK 庞大、初始化重)。但不是每次运行都真的用到它(比如某些纯本地命令)。把这 240ms 延迟到**首次真正使用**时再付,启动就快了。难点是:很多代码靠 `isinstance(client, OpenAI)` 做类型判断、靠 `patch("run_agent.OpenAI", ...)` 打桩(见 §1.1),延迟导入不能破坏这两个契约。

```python
# agent/process_bootstrap.py
class _OpenAIProxy:
    """将 from openai import OpenAI (~240ms) 延迟到首次使用"""
    # 保持 isinstance(client, OpenAI) 和 patch("run_agent.OpenAI", ...) 兼容
```

解法是一个代理类 `_OpenAIProxy`:平时它占着 `OpenAI` 这个名字的位置,直到第一次有人真用它,才触发那 240ms 的真实导入。注释里两条"保持兼容"是这个代理的设计约束 —— 它必须让 `isinstance` 检查和测试 patch 仍然像直接导入真 `OpenAI` 一样工作,否则优化就破坏了正确性。

### 4.2 延迟可行性检查(~400ms 节省)

**背景**:做上下文压缩需要一个"辅助模型"来生成摘要。要确认这个辅助模型的上下文窗口够不够大,得探测一下 —— 这个探测约 400ms。但**短会话根本不会触发压缩**,提前探测就是白花这 400ms。

```python
# agent/conversation_compression.py
# 首次压缩尝试时才探测辅助模型上下文窗口是否足够大
# → 短会话完全跳过此开销
```

解法很简单但有效:把探测**推迟到第一次真正尝试压缩时**才做。短会话从头到尾不压缩,这 400ms 完全省掉。这是"惰性求值"(lazy evaluation)思想的又一次应用 —— 只为真正发生的事付代价。

### 4.3 持久事件循环

**背景**:工具调用很多是异步的(asyncio)。如果每次工具调用都新建一个 asyncio 事件循环、用完就销毁,既有创建/销毁开销,还会引发一个隐蔽 bug:某些被缓存复用的 client 绑定在旧循环上,旧循环一关,再用它就报 `Event loop is closed`。

```python
# model_tools.py
_tool_loop = None  # CLI主线程持久循环
_worker_thread_local = threading.local()  # 每worker线程局部持久循环
# → 避免每个工具调用都创建/销毁asyncio循环
# → 防止cached client的"Event loop is closed"错误
```

解法是**让事件循环持久化、复用**而非每次新建:

- `_tool_loop`:CLI 主线程上的一个长期存活的事件循环。
- `_worker_thread_local`:用 `threading.local()` 给**每个 worker 线程**各自维护一个线程局部的持久循环(线程间不共享,各用各的)。

这样既避免了反复创建/销毁的开销,也杜绝了"cached client 撞上已关闭循环"的 `Event loop is closed` 错误。

### 4.4 配置读取的双路径

**背景**:配置在很多地方被频繁读取,其中一些是**每次 API 轮次都要走的热路径**(hot path)。读配置时如果每次都返回一份深拷贝(deepcopy,完整复制一份,改它不影响原件),安全但慢(约 135us 每次);热路径上累积起来不可忽视。但如果直接返回缓存引用又有风险:调用方一旦改了它,就污染了全局配置。作者提供**两条路径**让调用方按需选择。

```python
# hermes_cli/config.py
# load_config(): 返回deepcopy（安全但每次~135us）
# load_config_readonly(): 返回缓存引用（热路径，调用者绝不能修改）
# → 每次API轮次调用的get_provider_request_timeout()使用readonly路径
```

- `load_config()`:返回 **deepcopy**,安全(调用方随便改),代价是每次约 135us。
- `load_config_readonly()`:返回**缓存引用**,零拷贝、极快,但有个铁律 —— **调用方绝对不能修改它**(改了会污染全局)。

注释给了个真实用例:每次 API 轮次都要调的 `get_provider_request_timeout()`(取 provider 请求超时)走 readonly 路径,因为它只读不写、又在热路径上,省下的拷贝开销很可观。这是"用 API 命名(`_readonly` 后缀)把一个易错的契约显式化"的好例子。

### 4.5 FTS5 双表 CJK 搜索优化

**背景**:Hermes 把消息历史存进 SQLite,并用 **FTS5**(SQLite 的全文搜索扩展)做关键词检索。问题在于:FTS5 默认的 `unicode61` 分词器是按空格/标点切词的,这对英文等"有词边界"的语言很好,但对中日韩(CJK,Chinese/Japanese/Korean)这种"字与字之间没空格"的语言几乎没用 —— 它会把一长串 CJK 当成一个词,搜不出子串。作者用**两张表**分而治之。

```python
# hermes_state.py
# messages_fts (unicode61): 拉丁文本优化
# messages_fts_trigram (trigram): CJK子串搜索优化
# _contains_cjk() 检测 → 3+CJK字符路由到trigram → 1-2字符回退LIKE
```

- `messages_fts`(用 `unicode61` 分词器):为拉丁文本(英文等)优化。
- `messages_fts_trigram`(用 `trigram` 分词器):`trigram` 是"三字符滑窗"分词 —— 把文本切成所有连续 3 字符片段建索引,这正好能支持 CJK 的**子串搜索**(任意 3 字以上的连续片段都能命中)。
- 路由逻辑:`_contains_cjk()` 检测查询里有没有 CJK 字符;若有 **3 个及以上** CJK 字符,走 `trigram` 表;若只有 **1–2 个** CJK 字符(短到 trigram 也建不出三字窗),**回退到 `LIKE`** 模糊匹配兜底。

这套"按语言特征路由到不同索引、再对极短查询回退 LIKE"的设计,是同时把英文和 CJK 搜索都做对的务实方案。

## 六、总结

把前五章的细节抽象起来看,Hermes Agent 是一个**架构设计极为精良的大型 Python 项目**(2039 个 py 文件,约 98 万行代码)。它最突出的工程特征可以归纳为八条 —— 每一条都能在前面的具体小节里找到落点:

1. **分层严格**:入口 → CLI → Agent 核心 → 工具系统 → 网关 → 数据持久化,每一层职责清晰、互不越界。这是整个代码库能维持可读性的根基。
2. **抽象到位**:关键扩展点都抽象成 ABC(抽象基类),包括 `ProviderTransport`(§1.5)、`ContextEngine`、`MemoryProvider`、`BasePlatformAdapter`(§1.6、§五)、`SkillSource`。新增一种传输/内存/平台/技能源,都是"实现一个子类",而非改动主干。
3. **安全纵深**:防线是分层叠加的 —— 硬红线(Hardline,§3.1)→ 危险模式 → 智能审批 → 交互确认,外加技能多层安全扫描(§3.2)、线程上下文 fail-closed 传播(§3.3)、子进程密钥剥离(§3.4)。任一层失守,还有下一层。
4. **性能意识**:处处可见摊销与惰性 —— 三层系统提示词缓存(§1.4)、双层技能索引缓存(§1.3)、WAL 模式 SQLite、持久事件循环(§4.3)、懒加载 SDK(§4.1)、延迟可行性检查(§4.2)。
5. **自进化闭环**:Background Review(后台审查,§2.4)→ Skill 创建/改进 → Curator(技能策展)管理 → 系统提示词索引 → 让未来的对话受益。这是 agent "自我成长"的工程实现。
6. **测试工程化**:5 层密封 conftest(pytest 的配置/夹具文件)、按文件并行测试、实时系统保护、LPT(Longest Processing Time,最长处理时间优先)切片分发 —— 让大规模测试既快又稳(详见报告 10)。
7. **向后兼容**:重构与优化都没有破坏既有契约 —— `_ra()` 模式(§1.1)、`ProviderTransport` 的双路径迁移、`load_config_readonly` 热路径优化(§4.4)都是"改进而不破坏"的范例。
8. **平台可扩展**:对外开放了多种扩展面 —— MCP 协议(Model Context Protocol,工具集成标准)、插件系统、技能 Hub(10 源)、平台 registry(§1.6)、Provider 传输注册表。

> **延伸阅读**:本报告是架构调研系列(01–11)的总览收尾。若要深入"Hermes 向推理后端下发了哪些 agent hint、如何据此做 KV cache 主动管理与调度优化",见报告 12(各传输 hint 总表)、报告 13–15(源码 + 真实请求实测验证)、报告 16(Dynamo 风格 KV/调度设计)、报告 17(自研全栈的逐信号提取规格 + 坐标勘误)。
