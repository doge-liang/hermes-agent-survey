# Hermes Agent 技能系统架构深度分析

> **一句话导言**:Hermes 的"技能(skill)"是一种把可复用工作流程、领域知识、操作手册沉淀成磁盘文件的机制——每个技能就是一个目录,里面有一份 `SKILL.md`(YAML 头部声明元信息 + Markdown 正文写操作指引)外加若干辅助文件(参考文档、模板、脚本、静态资源)。技能既可以由用户手动 `/skill-name` 斜杠命令显式加载,也可以由 LLM 在对话中自主取用。本报告自底向上讲清这套系统:技能从哪来(创建)、怎么被找到(发现/索引)、怎么被装进对话(加载/使用)、磁盘上长什么样、安全上怎么防恶意技能、以及一个叫 Curator(策展器)的后台组件如何自动给技能做"保鲜与归档"。
>
> **读者画像**:本报告假设你懂推理服务/LLM 工程,但完全不了解 Hermes 内部。所有 Hermes 专有概念(技能、Curator、Hub、Guard、伞形技能等)都会先解释再使用。
>
> **本报告与 KV/调度主线的关系**:技能系统的产物——稳定的 system prompt 前缀、技能正文注入、工具集——最终都会进入发往推理后端的请求里,因此它是"共享前缀有多长、缓存能复用多少"的上游来源之一。涉及前缀/缓存的量级问题详见报告 15(真实 CLI 实测的 system/tools 量级)与报告 17 §1(信号提取规格)。

## 一、技能生命周期:从创建到归档的完整链路

先给一张总览图。一个技能从"诞生"到"退役"会经过下面几个阶段,后续各节逐一展开:

```
创建 (agent自主 / Hub安装 / 捆绑同步)
  → 发现 (双层缓存:LRU内存 + 磁盘快照)
  → 索引 (构建系统提示词中的技能表)
  → 加载 (/skill-name 命令 或 LLM自主调用)
  → 使用 (bump_use 记录遥测)
  → 改进 (Background Review → skill_manage patch)
  → 归档 (Curator: active → stale → archived)
```

这条链路的关键设计理念是:**技能是"活的"**——它不是写一次就固化的静态文档,而是会被使用频率(遥测)反向驱动改进,会被一个后台策展器按"多久没用过"自动降级和归档。下面逐段解释。

### 1.1 创建来源:技能可以从三个渠道进入系统

技能进入 `~/.hermes/skills/` 这个目录有三条独立的路径,分别面向"agent 自己写"、"从社区下载"、"仓库自带同步"三种场景。

**A. Agent 自主创建** —— 代码在 `tools/skill_manager_tool.py`。这是 LLM 在对话中自己决定"我应该把这套做法存成一个技能"时走的工具,提供一组类似文件系统编辑器的子操作:

- `create`:新建一个技能目录,并写入 `SKILL.md`(其结构是"YAML 头部 + Markdown 正文"——YAML 头声明技能的 name、description、适用平台等元信息,正文写实际的操作指引)。
- `edit`:对 `SKILL.md` 做**完整重写**(整篇替换,适合大改)。
- `patch`:用一个**模糊匹配引擎**做定向的"查找—替换"(适合小改;模糊匹配的意义是:即便正文里有细微的空白/格式漂移,也能定位到要改的那一段,不必逐字精确)。
- `delete`:移除技能目录;并支持一个 `absorbed_into` 声明——当一个技能被合并进另一个技能时,删除方可以用 `absorbed_into` 显式告诉系统"我的内容已经被吸收进了那个技能",这条声明后面会被 Curator 当作权威信号使用(见 §4.4)。
- `write_file` / `remove_file`:管理技能目录下的辅助文件(参考文档、模板、脚本等),不动 `SKILL.md` 本体。

**B. Hub 安装** —— 代码在 `tools/skills_hub.py`,这是"从外部源下载并安装技能"的渠道,内置 **10 个源适配器**(每个适配器知道如何与某个特定的技能托管站点对话)。安装走的是一条带安全闸门的流水线:

```
search(搜索) → fetch bundle(拉取技能包) → quarantine(放进隔离区) → 安全扫描 → install(通过扫描后才真正安装)
```

这里的关键是 **quarantine(隔离区)**:下载的技能不会直接落到生效目录,而是先进一个隔离暂存区,经过 §3 讲的安全扫描后才被"释放"到 `~/.hermes/skills/`。源适配器包括 HermesIndex、skills.sh、GitHub、clawhub.ai、LobeHub 等。系统还维护一份**信任仓库**名单:`openai`、`anthropics`、`huggingface`、`NVIDIA`——来自这些仓库的技能在安全策略里享有更宽松的判定(详见 §3.1 的策略矩阵)。

**C. 捆绑同步** —— 代码在 `tools/skills_sync.py`。Hermes 仓库自带一批内置技能,这条路径通过一份名为 `.bundled_manifest` 的清单,把这些仓库自带技能同步写入用户的 `~/.hermes/skills/` 目录。`.bundled_manifest` 记录了"哪些技能是内置同步来的",这样升级时才能正确地区分"内置技能"与"用户/agent 自建技能"。

### 1.2 发现与索引:双层缓存避免每轮重扫磁盘

每一轮对话都要在 system prompt 里告诉 LLM"你现在有哪些技能可用"。如果每次都去扫一遍磁盘上所有技能目录、读所有 `SKILL.md`,代价太高。Hermes 用**双层缓存**来摊薄这个成本:

1. **进程内 LRU 缓存**:在内存里缓存"已经算好的技能表"。缓存键由一组决定技能表内容的因素拼成——`(skills_dir, external_dirs, available_tools, ...)`(技能目录、外部目录、当前可用工具集等)。容量 **8 条**(LRU = 最近最少使用淘汰,即最多记住 8 种不同输入组合的结果)。只要这些输入没变,就直接命中内存,零磁盘 IO。
2. **磁盘快照**:`~/.hermes/.skills_prompt_snapshot.json`,把算好的技能提示词持久化到磁盘。进程重启后内存缓存没了,但磁盘快照还在。快照的有效性靠一份**文件清单按 mtime/size 校验**来判断——也就是记录每个技能文件的修改时间(mtime)和大小(size),下次启动时比对:如果文件清单没变,快照就还能用;一旦有技能文件被改动,快照失效、触发重建。

这两层叠加的效果是:常态下技能表几乎零成本复用;只有技能真正发生变更时才付出重新扫描的代价。

### 1.3 加载:斜杠命令如何把一个技能"装进"对话

当用户输入 `/skill-name` 这类斜杠命令时,处理逻辑在 `agent/skill_commands.py`,流程如下:

1. **解析命令**:把 `/skill-name --instruction` 拆成 `cmd_key`(技能标识)和可选的附加指令。
2. **加载技能内容**:通过内部函数 `_load_skill_payload` 读取该技能的完整内容。
3. **构建结构化消息**:把技能注入对话时,组装成一条信息丰富的消息,包含——激活说明(告诉 LLM"现在激活了哪个技能")、完整技能正文(`SKILL.md` 的 Markdown 全文)、skill 目录路径(让 LLM 知道辅助文件在哪)、配置值、以及辅助文件列表。这样 LLM 不仅拿到"怎么做"的指引,还知道"配套资源在哪、叫什么"。
4. **记录遥测**:调用 `skill_usage.bump_use()`,把"这个技能被用了一次"记进遥测——这是后面 Curator 判断技能冷热的数据来源。

另外,`agent/skill_preprocessing.py` 提供两类预处理能力:**模板变量替换**(把技能正文里的占位符替换成实际值)和 **inline-shell 展开**(在加载时执行内联 shell 片段并把结果嵌入正文)。这让技能正文可以是"带变量、带动态内容"的模板,而不是死文本。

## 二、磁盘物理结构:技能在文件系统上长什么样

理解了生命周期,再看技能在磁盘上的实际布局。所有技能都落在 `~/.hermes/skills/` 下,既有技能目录本身,也有一批以点号开头的元数据文件/目录(管理用,非技能本体):

```
~/.hermes/skills/
├── my-skill/
│   ├── SKILL.md          # YAML头 (name, description, platforms...) + markdown 正文
│   ├── references/       # 辅助文档
│   ├── templates/        # 模板文件
│   ├── scripts/          # 可执行脚本
│   └── assets/           # 静态资源
├── .usage.json           # Curator遥测(每个技能被查看/使用的次数与时间)
├── .bundled_manifest     # 内置技能同步清单(§1.1-C)
├── .hub/
│   ├── lock.json         # Hub安装来源记录(装的技能从哪个源、哪个版本来)
│   ├── quarantine/       # 安装前安全隔离区(§1.1-B 的 quarantine 落地处)
│   ├── audit.log         # 审计日志
│   └── taps.json         # 自定义GitHub taps(用户自配的 GitHub 技能源)
├── .curator_suppressed   # 已删除内置技能抑制列表(防止被同步重新拉回)
└── .archive/             # 归档技能(可恢复,非永久删除)
```

几个值得解释的点:

- **一个技能 = 一个目录**:`my-skill/` 里 `SKILL.md` 是主文件,`references/`、`templates/`、`scripts/`、`assets/` 是按用途分类的辅助资源子目录。
- **`.usage.json`** 是 Curator 的"体温计"——它记录每个技能的使用遥测,Curator 据此判断技能是热是冷(§4.2)。
- **`.archive/`** 体现了一条重要的安全设计:**Curator 绝不真删技能,只归档**。归档技能搬到 `.archive/` 下,随时可恢复(§4.5 的 `rollback()`)。
- **`.curator_suppressed`** 解决一个边界问题:如果用户删掉了某个内置技能,捆绑同步(§1.1-C)下次又会把它拉回来。这份抑制列表记录"哪些内置技能被刻意删除了",让同步逻辑跳过它们,尊重用户意图。

## 三、安全体系:防止恶意技能进入并执行

技能里可以带可执行脚本、可以注入 LLM 指令,因此"来历不明的技能"是个真实的攻击面。Hermes 用三道防线应对:正则/结构扫描(Skills Guard)、Python 反射模式的 AST 审计、以及对 agent 自建技能的可选扫描。

### 3.1 Skills Guard:约 130 种正则模式 + 结构检查

Skills Guard(代码在 `tools/skills_guard.py`)是安装前的主扫描器,内含**约 130 种正则威胁模式**,配合多层结构性检查:

1. **结构检查**:把明显异常的技能包挡在门外——文件数 **> 50**、总大小 **> 1MiB**、含二进制扩展名、含符号链接、文件带可执行权限位。这些不是绝对禁止,而是"对一个技能来说不正常"的信号(一个正常技能不该有 50+ 文件或 1MB+ 体积)。
2. **正则威胁模式(6 类)**:按攻击意图把模式分成六类——数据窃取(exfiltration)、提示注入(prompt injection,即在技能正文里藏指令试图操纵 LLM)、破坏性操作(如 `rm -rf` 删根/删 home)、持久化(如偷偷装 cron 任务)、混淆(obfuscation,用编码/变形隐藏真实意图)、供应链(supply chain,如篡改依赖)。
3. **隐形 Unicode 检测**:扫描 **16 种隐形字符**(零宽空格、双向覆盖、格式隔离字符等)。这类字符肉眼不可见,但能被用来在技能文本里夹藏不可见的注入指令,因此必须专门检测。
4. **`.skillignore` 支持**:类似 `.gitignore` 的风格,允许技能作者声明扫描时应排除哪些文件。

**安装策略矩阵**:扫描产出一个"安全判定"(safe / caution / dangerous),再结合技能的"来源信任级别"(builtin / trusted / community)共同决定是放行(allow)还是拦截(block)。矩阵如下——

| 来源 \ 判定 | safe | caution | dangerous |
|---|---|---|---|
| builtin | allow | allow | allow |
| trusted | allow | allow | block |
| community | allow | block | block |

读法与背后的逻辑:

- **builtin(仓库内置)**:全部 allow——内置技能被视为完全可信,即便扫到 dangerous 模式也放行(因为它本就是 Hermes 自带、经过审查的)。
- **trusted(信任仓库,如 openai/anthropics 等 §1.1-B 名单)**:safe 与 caution 都放行,只有 dangerous 才 block——给信任源较高的容忍度,但仍守住"明确危险"这条底线。
- **community(其余社区来源)**:只有 safe 才放行,caution 与 dangerous 一律 block——对不可信来源采取最严格策略,任何可疑迹象即拦截。

可以看出这是一个"信任越高、容忍越宽"的梯度策略:同一个 caution 判定,在 trusted 源会放行、在 community 源会拦截。

### 3.2 AST 深度审计:抓 Python 里的反射式隐藏调用

正则扫的是文本特征,但攻击者可以用 Python 的**反射机制**绕开静态特征——比如不直接写 `import os`,而是用动态方式在运行时才拼出模块名。AST 审计(代码在 `tools/skills_ast_audit.py`)专门对付这类手法:它把技能里的 Python 文件解析成抽象语法树(AST),扫描这些反射模式——

- `importlib.import_module()`:运行时加载任意模块。
- 动态 `__import__`(模块名非字面量):同样能在运行时加载任意模块。
- 计算属性 `getattr`(属性名非字面量):动态取属性,可绕过静态可见性。
- `__dict__[<computed>]`:用计算出来的键访问对象字典,动态属性访问的另一种形式。

这些模式本身不一定恶意,但它们是"把真实行为藏到运行时"的典型手段,因此被标记出来供人工/策略复核。

### 3.3 Agent 创建技能的安全扫描:默认关闭,可按配置开启

前两道防线主要针对"从外部安装"的技能。但 agent 自己创建的技能(§1.1-A)默认走的是另一套较宽松的策略:

- **前台创建**:安全扫描**默认关闭**——因为前台是用户在场、agent 在用户指示下创建技能,默认信任。
- 当配置项 `skills.guard_agent_created = true` 时,对 agent 自建技能也启用扫描:此时 `dangerous` / `caution` 判定会**触发回滚**(撤销这次创建)。
- **原子写入保证可回滚**:写入前先通过临时文件原子写入,这样一旦扫描判定要回滚,能干净地撤销,不留半成品。

## 四、Curator 策展器:让技能自动"保鲜与归档"

技能会越攒越多,其中很多是 agent 一时创建、之后再没用过的。Curator(策展器,代码在 `agent/curator.py`)是一个后台组件,负责给技能库做"自动整理":把长期不用的技能降级、归档,把零散的小技能合并成更合理的结构。它的设计处处体现"保守、可逆、不伤主会话"的原则。

### 4.1 核心职责与安全边界

Curator 在动手前先划清自己的边界,避免误伤:

- **只触及 agent 创建的技能**(通过 `is_agent_created()` 判断)——用户手写的、Hub 装的、内置同步的技能它一律不碰,只整理 agent 自己沉淀出来的那些。
- **绝不自动删除——只归档**:任何"退役"动作都是搬进 `.archive/`(§2),保证可恢复。
- **固定(pinned)技能绕过所有自动状态转换**:用户可以把重要技能标为 pinned,Curator 对它完全免疫,不降级、不归档、不合并。
- **使用辅助客户端,不触碰主会话的 prompt 缓存**:Curator 在需要调用 LLM 时(见 §4.3)用的是独立的辅助 LLM client,而非主对话用的那个。这样它的活动不会污染主会话精心维护的 prompt 缓存(KV cache 复用),不影响主线对话的缓存命中。

### 4.2 自动状态转换:纯函数判定,不调用 LLM

技能的"冷热状态"转换是一段**纯函数逻辑**(只看遥测里的时间戳,不调用 LLM,因此确定、廉价、可预测)。状态机如下:

```
active ──[last_activity <= stale_after_days]──→ stale      (太久没用 → 转为陈旧)
stale  ──[last_activity >  stale_after_days]──→ active     (又被用了 → 重新激活)
stale  ──[last_activity <= archive_after_days]─→ archived   (陈旧后再久不用 → 归档)
```

含义解释:技能默认处于 `active`(活跃);如果它的最后活动时间超过 `stale_after_days` 没有更新,就降为 `stale`(陈旧);处于 `stale` 的技能一旦又被使用,会**重新激活**回 `active`;若 `stale` 之后又过了 `archive_after_days` 仍无人问津,就被归档(`archived`)。

默认阈值:`stale_after_days = 30`、`archive_after_days = 90`——即默认 30 天不用转陈旧、90 天不用转归档。这两个值可通过配置调整。

### 4.3 LLM 审查 Pass:把零散技能合并成"伞形"结构

状态机只处理"冷热",但还有一类整理需要判断力:agent 可能把一件事拆成了五个零碎小技能,而人类维护者会把它们组织成一个更合理的结构。这一步需要 LLM 参与,Curator 称之为"伞形构建合并(umbrella build merge)"。

它的核心判断标准是一句很直白的话:**"一个人类维护者,会把这些写成 N 个独立技能,还是写成 1 个含有 N 个小节的技能?"**——以此决定是否把多个相关小技能收拢成一个"伞形技能"。

合并有三种具体做法:

1. **合并到现有伞形**:把小技能 `patch` 进一个已存在的伞形 `SKILL.md`,再把这个被吸收的同级小技能归档。
2. **创建新伞形 SKILL.md**:当没有现成伞形时,新建一个伞形技能把若干小技能纳入其下。
3. **降级为引用/模板/脚本**:某些"技能"其实更像是配套资源而非独立技能,就把它移到某个技能的子目录(`references/`、`templates/`、`scripts/`),并归档原始的独立技能。

### 4.4 分类调和:合并三种信号,既听模型的也做事实核对

LLM 审查会给出"该怎么合并"的意图,但 LLM 可能产生幻觉或遗漏。Curator 不盲信单一来源,而是**调和(reconcile)三种信号**,各有分工:

1. **模型声明的 `absorbed_into`**(权威):当模型删除一个技能时,会用 `absorbed_into` 字段告知"它被并进了哪个技能"(§1.1-A)。这是关于"归属"的权威信号。
2. **模型的结构化 YAML 块**(意图与理由的权威):模型用一个结构化 YAML 块表达"我打算怎么合并、为什么"。这是关于"意图"的权威信号。
3. **工具调用启发式**(真实审计):不依赖模型自述,而是从模型**实际发生的工具调用**里反推它到底做了什么。这是一份"客观审计",用来捕捉前两种自述里的**幻觉和遗漏**——即模型说要做但没做、或做了却没说的情况。

三者结合的效果是:既尊重模型表达的意图(信号 1、2),又用客观的执行轨迹(信号 3)兜底校验,防止模型"说一套做一套"导致技能库被搞乱。

### 4.5 每次运行都留报告、快照与回滚

Curator 的每一次运行都有完整的可审计与可回退保障:

- **每运行报告**:写入 `logs/curator/{YYYYMMDD-HHMMSS}/run.json` 加 `REPORT.md`——既有机器可读的 `run.json`,也有人读的 `REPORT.md`,记录这次都做了什么。
- **变更前快照**:在动任何东西之前,先对整个 `~/.hermes/skills/` 打一个 tar.gz 快照。
- **回滚** `rollback()`:回滚流程本身也很谨慎——先给**当前状态**也打一个安全快照(防止回滚动作本身丢数据)→ 再提取要恢复到的那个选定快照 → 最后恢复 cron 作业的技能链接(避免回滚后定时任务指向失效技能)。

## 五、Skill Bundles(技能捆绑包):一个名字激活一组技能

Skill Bundle 是一种"复合技能别名"——用一个斜杠名称,一次性激活多个技能。配置文件放在 `~/.hermes/skill-bundles/*.yaml`,形如:

```yaml
name: backend-dev
skills:
  - github-code-review
  - test-driven-development
  - github-pr-workflow
instruction: |
  Optional extra guidance
```

这个例子定义了一个叫 `backend-dev` 的 bundle,输入 `/backend-dev` 就会同时拉起三个技能(代码评审、测试驱动开发、PR 工作流),并可附带一段可选的额外指引(`instruction`)。相关行为细节:

- **bundle 与 skill 同名时,bundle 优先**:如果一个 bundle 和一个单独的 skill 共享同一个斜杠名称,激活时 bundle 优先。
- **缺失成员容错**:若 bundle 里引用的某个技能不存在,就跳过它并给出提示,但 **bundle 仍然加载**(不会因为一个成员缺失就整体失败)。
- **逐个计遥测**:bundle 激活时,其中每个技能都各自通过 `bump_use` 记录一次使用遥测——所以 bundle 不会让 Curator 的冷热判定"看不见"被它拉起的技能。

## 六、背景审查触发技能改进:技能如何自我进化

最后一块拼图是"技能怎么变得更好"。Hermes 有一条**后台自我改进**机制:agent 会在后台 fork 出一个审查进程,回顾近期使用情况并改进技能。这条链路由五个部件支撑:

1. **来源标记**:`skill_provenance.py` 用一个 `ContextVar`(上下文变量,可在调用链中传递而无需逐层传参)区分**前台(foreground,用户指示的写入)** 与 **后台审查(background_review,自我改进 fork 的写入)**。默认值是 `foreground`;后台审查 fork 会把它设为 `background_review`。这个区分很关键——它让系统知道一次技能写入到底是"用户当面要求的"还是"agent 后台自己改的",从而应用不同的策略(例如 §3.3 的扫描门控)。
2. **遥测收集**:每次技能被查看 / 使用 / 修补后,都会更新 `.usage.json`,并通过**文件锁序列化**(file lock 保证并发写入不冲突,因为前台和后台可能同时在写)。
3. **系统提示指引**:一段名为 `SKILLS_GUIDANCE` 的指引被注入 system prompt 头部,告诉 LLM 如何看待与运用技能系统。
4. **缓存失效**:当 `skill_manage`(技能管理操作)成功后,调用 `clear_skills_system_prompt_cache(clear_snapshot=True)` 主动失效缓存——既清内存 LRU 也清磁盘快照(§1.2 的双层缓存),确保下一轮 system prompt 反映最新的技能集。
5. **创建提醒**:配置项 `skills.creation_nudge_interval`(**默认 10 轮**)控制"提醒 agent 考虑创建技能"的频率——每隔约 10 轮对话,系统会轻推一下 agent,提示它"这段经验是否值得沉淀成技能"。

这五点合起来构成一个闭环:遥测记录使用情况 → 后台审查据此改进技能 → 缓存失效让改进即时生效 → 创建提醒推动新技能产生 → 来源标记保证前台/后台各走各的策略。技能系统因此不是静态文档库,而是一个随使用持续演化的资产库。
