# Hermes Agent 网关与多平台架构深度分析

> **一句话**:Gateway(网关)是 Hermes 把同一个 AI agent 接到 Telegram、WhatsApp、Slack 等 31 个聊天平台的那一层。它解决一个核心矛盾——每个平台的消息收发协议、限流规则、富文本格式、流式编辑能力都不一样,但上面跑的 agent 逻辑应该只写一份。Gateway 用"适配器模式 + 事件驱动流式管道"把这层差异隔离掉:平台特有的脏活由各自的 Adapter(适配器)封装,中间用一套与平台无关的"结构化流事件"作为通用语言。
>
> **本报告的读者画像**:你懂推理服务 / KV cache / LLM 工程,但不了解 Hermes 内部。下文凡是 Hermes 专有的概念(SessionKey、StreamConsumer、DeliveryRouter 等)都会先用一句话解释再展开。
>
> **与本报告关系最大的其他报告**:网关是"agent 随请求下发给推理后端的元数据(agent hint)"在生产里真正经过的那条路径——所以做 KV/调度信号采集时,网关路径就是线上路径。涉及 agent hint、host 门控、压缩阈值等推理侧细节的部分,详见报告 12(所有传输层 agent hints 总结)、报告 15(真实 CLI 探测测试台)、报告 16(Dynamo 风格 KV/调度信号采集)、报告 17(信号提取规格)。

## 一、整体架构:一条消息从进到出走过的全程

Gateway 本质是一个**多平台消息代理(message proxy)**。它的总线是一条单向流水线,每条用户消息都按下面这条路走完一个来回:

```
用户消息 → 平台适配器(Adapter) → GatewayRunner → AI Agent → StreamConsumer → 平台适配器 → 用户
```

逐段解释这条管线在做什么:

- **平台适配器(Adapter)**:把某个具体平台(比如 Telegram)收到的原始事件,翻译成 Hermes 内部统一的入站消息对象;反过来也负责把 agent 的输出翻译回该平台的发送 API。它是"平台方言 ↔ Hermes 标准语"的双向翻译器。
- **GatewayRunner**:总编排器(orchestrator)。它接到标准化后的消息,找到/新建对应的会话与 agent 实例,把消息喂进去,并管理整个网关进程的生命周期(启动、重连、关闭)。详见第六节。
- **AI Agent**:真正干活的推理循环——调模型、调工具、产生回答。它不知道自己是在为哪个平台服务,这正是适配器模式带来的解耦。
- **StreamConsumer**:流消费器。agent 是同步、逐 token 往外吐的;但聊天平台要的是"先发一条占位消息,再不断把它编辑成完整回答"这种渐进式体验。StreamConsumer 就是这两种节奏之间的缓冲与适配层。详见第四节。

"事件驱动流式管道"这个说法的含义是:agent 的输出不是一锤子的整块字符串,而是被切成一连串**类型化事件**(文本增量、工具调用开始、工具调用结束……),沿管线流动,各平台适配器再决定怎么把这些事件渲染成自己平台上看得见的东西。这套事件词汇表是整个解耦的关键,见 §4.1。

## 二、BasePlatformAdapter:所有平台适配器的抽象基类

**文件**:`gateway/platforms/base.py`,共 **4781 行**。`BasePlatformAdapter` 类定义在 **第 1764 行**(`base.py:1764`,`class BasePlatformAdapter(ABC)`)。`ABC` 是 Python 的抽象基类(Abstract Base Class),意思是它本身不能被实例化,只规定"一个平台适配器必须/可以提供哪些能力",具体实现留给 31 个子类(Telegram、WhatsApp 等)各自填。

这种设计的好处:新增一个平台时,只要继承这个基类、把"必须实现"的几个方法填上,其余的高级特性(流式编辑、媒体发送、输入状态指示)能用基类的默认降级行为先跑起来,再按需逐个增强。

### 2.1 子类必须实现的方法(没有它平台就不通)

- `connect()` / `disconnect()` —— 平台连接的生命周期。前者建立到平台的长连接 / 鉴权,后者干净地断开。
- `send(chat_id, content, ...)` —— 往某个聊天发一条消息。这是最基本、不可省略的能力。

### 2.2 子类可选重写的方法(不重写就走基类默认降级)

下面这些是"锦上添花"的能力。基类对每个都提供了一个保守的默认实现(通常是降级成最朴素的发消息),所以子类不实现也不会崩,只是体验差一点:

- `edit_message()` / `delete_message()` —— 流式编辑(把已发出的消息改写成更完整的内容)和删除消息。流式编辑是渐进式回答体验的基础;平台不支持时,StreamConsumer 会退回成"攒齐了再一次性发"。
- `send_image()` / `send_audio()` / `send_video()` / `send_document()` —— 原生媒体发送。不支持时通常降级成发链接或文本。
- `send_draft()` —— 原生流式草稿。这是较新的能力,例如 Telegram Bot API 9.5+ 才支持的"草稿流"——边生成边把草稿区刷新,体验比反复编辑一条已发消息更顺滑。
- `send_typing()` / `stop_typing()` —— 输入状态指示器,也就是聊天界面里那个"对方正在输入…"。
- `send_clarify()` —— 澄清提示,用来向用户抛多选或开放式问题(让用户在几个选项里选,或自由作答)。
- `render_message_event()` —— 把第四节那套"结构化流事件"渲染成本平台看得见的形式。不同平台对富文本/按钮/卡片的支持差异很大,所以渲染逻辑下放给各平台自己。
- `format_tool_event()` —— 工具调用进度的渲染,例如把"正在调用搜索工具…"显示给用户看。

### 2.3 并发控制:同一会话内多条消息怎么排队、怎么打断

聊天场景下,用户可能在 agent 还没答完时又发一条消息;同一群里多个用户也可能同时说话。基类用下面几个数据结构来管理"每个会话(session)的并发":

- `_active_sessions`(`base.py:1796`)—— 每个 session 对应一个"中断 Event"。当需要打断某个正在进行的 agent 回答时,就触发它对应的 Event。
- `_pending_messages`(`base.py:1797`)—— 当某个 session 正忙(agent 还在答上一条)时,新来的消息先在这里排队。
- `_session_tasks`(`base.py:1798`)—— 精确记录每个 session 对应的 asyncio.Task,用于在需要时精准取消那一个任务,而不是误伤别的会话。
- `_busy_text_mode`(`base.py:1799`)—— "忙碌时怎么处理新消息"的策略,取值 `queue`(排队,等当前答完再处理)或 `interrupt`(打断当前回答,优先处理新消息)。
- `_text_debounce`(`base.py:1809`)—— 消息消抖合并。用户连发好几条短消息时,把它们在一个时间窗口内合并成一条再交给 agent,避免触发多次独立回答。"消抖(debounce)"是这里的关键词:短时间内的连续输入合并成一次处理。

### 2.4 三个关键数据结构(平台无关的标准载体)

这三个结构是适配器与网关之间传递信息的"标准信封",定义同样在 `base.py`:

- `MessageEvent`(`base.py:1391`)—— 标准化的**入站**消息。无论来自哪个平台,进来后都被归一成这个对象,带上 `text`(文本)、`message_type`(消息类型)、`source`(来源信息,含平台/聊天/用户标识)、`media_urls`(媒体附件链接)等字段。下游逻辑只认这个对象,不关心原始平台格式。
- `SendResult`(`base.py:1520`)—— 发送的**结果**。带 `success`(是否成功)、`message_id`(平台返回的消息 ID,后续编辑要用)、`error`(错误信息)、`retryable`(该错误是否值得重试)等。`retryable` 让上层能区分"网络抖动可重试"和"内容违规不可重试"。
- `EphemeralReply`(`base.py:1536`,继承自 `str`)—— 带 TTL(存活时间)的自动删除回复。它本身是个字符串子类,所以能当普通文本用,但额外携带"发出去后多久自动删掉"的语义,常用于临时提示。

## 三、会话管理(session.py):谁和谁共享一段对话历史

**文件**:`gateway/session.py`。会话管理回答一个核心问题:**两条消息算不算"同一段对话"?** 这直接决定了对话历史(以及背后的 KV 前缀)怎么切分和复用——对 KV 调度尤其关键,因为"同一会话"通常意味着"共享同一段稳定前缀"。

### 3.1 SessionKey:会话的唯一身份

SessionKey 是会话的主键,由 `build_session_key()`(`session.py:600`)按下面的格式拼出:

```
格式: agent:main:<platform>:<chat_type>:<chat_id>[:thread_id][:<user_id>]
```

各段含义:`agent:main` 是固定前缀;`platform` 是平台名(telegram/slack…);`chat_type` 是聊天类型(dm 私聊 / group 群 / channel 频道);后面跟具体的聊天 ID、可选的线程 ID 和用户 ID。隔离规则:

- **DM(私聊)会话**:按 `chat_id` 和 `thread_id` 隔离(`session.py:636-640`)。同一个人不同线程互不串台。
- **群组 / 频道**:这里有两个配置开关决定隔离粒度:
  - `group_sessions_per_user`(默认 `True`,`session.py:582`)—— 群里**每个用户拥有独立会话**。也就是说,默认情况下群里 A 和 B 跟 bot 的对话历史是分开的,互不污染。
  - `thread_sessions_per_user`(默认 `False`,`session.py:583`)—— 同一线程内的多个用户**共享**一段会话。即在线程里大家是"围着同一个话题一起聊",历史是共用的。
- **WhatsApp**:统一 JID / LID 别名。WhatsApp 同一个人可能有多种标识(JID 是传统的电话号绑定标识,LID 是较新的隐私标识),这里把它们归一成同一会话,避免同一个人被当成两个人。

为什么这套规则重要:它决定了"哪些请求会落到同一段历史、共享同一段前缀"。默认"群里每人独立会话"意味着前缀复用以"人"为粒度;若改成线程共享,则以"话题"为粒度。做前缀亲和性 / KV 复用实验时,必须先搞清当前部署用的是哪种粒度。

### 3.2 SessionStore:会话状态存哪、怎么保证并发安全

`SessionStore`(`session.py:668`)是会话状态的存储层:

- **主存储 SQLite + JSONL 降级**:默认用 SQLite(通过 `SessionDB`)存会话元数据和消息记录(`session.py:672`);如果 SQLite 不可用,降级到旧的 JSONL 文件(`session.py:673`,启动时会打印一条 warning)。SQLite 被当作"事实来源(source of truth)",因为它能可靠保留完整的消息记录。
- **线程安全**:读操作靠 Python 的 GIL(全局解释器锁)天然保护,写操作另加一把 `_lock`(`session.py:682`,`threading.Lock`)。这是因为网关是多平台并发的,多个适配器可能同时读写会话。
- **重置策略**:`get_reset_policy` 支持四种模式(`session.py:768-817`):`none`(从不自动重置)/ `idle`(空闲超时后重置)/ `daily`(每天定时重置)/ `both`(空闲和每日都生效)。"重置"指清掉旧历史、开一段新会话,避免历史无限膨胀。被自动重置时会记录原因(`auto_reset_reason`,取值 `"idle"` 或 `"daily"`,`session.py:459`)。
- **`resume_pending` 标记**(`session.py:490`):网关进程崩溃 / 重启后,这个标记让会话能保留原 `session_id` 并在下次访问时自动恢复,而不是丢掉上下文重开。这对"长跑的 agent 不因网关重启而失忆"很重要。
- **PII 红名(脱敏)**:`session.py` 顶部有一组 PII 脱敏辅助函数。`_hash_id()`(`session.py:34`)用 SHA-256 取前 12 位十六进制,把标识符做成确定性哈希;`_hash_sender_id()` / `_hash_chat_id()` 在其上包装(例如 `telegram:12345` → `telegram:<hash>`)。WhatsApp / Signal / Telegram 默认对用户 ID 做哈希,这样落盘的会话数据里不直接出现明文身份。"确定性哈希"意味着同一个人每次哈希出同一个值,既能脱敏又不破坏会话归并。

## 四、流式输出管道:把"逐 token"变成"看着像在打字"

### 4.1 结构化流事件(stream_events.py):平台无关的输出词汇表

**文件**:`gateway/stream_events.py`。这里定义了一套**类型化事件词汇表**,目的是把"agent 想表达什么语义"和"某个平台具体怎么渲染"彻底解耦——agent 只管产出语义事件,平台适配器各自决定怎么画。下面 7 类事件是这套词汇表的全部:

| 事件 | 含义 | 定义位置 |
|------|------|----------|
| `MessageChunk(text)` | 增量文本 delta(回答正文的一小段,逐段累积) | `stream_events.py:44` |
| `MessageStop(final)` | 一个文本段落结束的信号 | `stream_events.py:56` |
| `Commentary(text)` | 工具调用之间的过渡性旁白(例如"我先查一下…") | `stream_events.py:72` |
| `ToolCallChunk(...)` | 一次工具调用开始 | `stream_events.py:85` |
| `ToolCallFinished(...)` | 一次工具调用完成 | `stream_events.py:104` |
| `LongToolHint(...)` | 长时间工具的提示(告诉用户"这步比较慢,请稍候") | `stream_events.py:122` |
| `GatewayNotice(...)` | 网关自身的原生控制消息(非 agent 内容,如系统通知) | `stream_events.py:135` |

读这张表的方式:`MessageChunk` 是正文增量,`MessageStop` 标记一段说完了;`Commentary` 与三个 `ToolCall*` 一起描绘"agent 正在用工具"的过程;`LongToolHint` 和 `GatewayNotice` 是体验性 / 控制性的旁路信息。平台适配器拿到这些事件后,通过 §2.2 的 `render_message_event()` 渲染成自己平台的具体形态(纯文本追加、富卡片、进度条等)。

### 4.2 流消费器(stream_consumer.py):跨线程缓冲 + 渐进编辑

**文件**:`gateway/stream_consumer.py`,共 **1346 行**,核心类 `GatewayStreamConsumer`(`stream_consumer.py:79`)。它要解决一个具体的工程难题:agent 跑在**同步**线程里逐 token 产出,而平台发送是**异步**的,且不能每来一个 token 就调一次平台 API(会被限流打死)。它的工作流是:

```
Agent线程(sync) → Queue → GatewayStreamConsumer(async) → 平台适配器
                                ├── 积累文本到 _accumulated
                                ├── 按间隔/阈值批量刷新
                                ├── 通过 edit_message 渐进编辑
                                ├── 原生草稿流(draft streaming)
                                ├── 溢出分块(超出平台单条长度限制)
                                └── 完成时最终编辑
```

逐项解释这几步:

- **跨线程交接**:同步的 agent 线程把产出丢进一个队列(Queue),异步的消费器从队列里取——这是把同步世界和异步世界缝合的标准做法。
- **积累到 `_accumulated`**(`stream_consumer.py:138`):把零碎的文本增量先攒在内存里,而不是逐字往平台发。
- **按间隔 / 阈值批量刷新**:每隔一段时间、或攒够一定量,才把当前积累的内容刷到平台。这是平衡"实时感"和"API 调用频率"的关键节流。
- **通过 `edit_message` 渐进编辑**:把同一条占位消息反复编辑成越来越完整的回答,用户看到的就是"文字一段段长出来"。
- **原生草稿流(draft streaming)**:平台支持时(如 Telegram DM),改用 `send_draft` 在草稿区刷新动画帧,而不是反复编辑已发消息(`stream_consumer.py:174-179`,状态变量 `_use_draft_streaming`)。注意草稿没有 `message_id`,所以最终答案仍要正式发一条,才会留在聊天记录里。
- **溢出分块**:回答超过平台单条消息长度上限时,自动切成多条发。
- **完成时最终编辑**:流结束时做一次定稿编辑,确保最后呈现的是完整答案。

**四种传输模式**(`stream_consumer.py:66-74`):`edit`(渐进编辑)/ `draft`(显式要原生草稿)/ `auto`(优先用草稿,不支持再退回编辑)/ `off`(关闭流式,攒齐一次发)。

**错误恢复——为什么"连续 3 次"是个硬阈值**:平台限流报错(flood control)时,消费器会累计连续失败次数(`_flood_strikes`,`stream_consumer.py:152`)。一旦连续失败达到 `_MAX_FLOOD_STRIKES = 3`(`stream_consumer.py:96`),它会**永久禁用本次流的渐进编辑**,改走 fallback send(攒齐一次性发)。这么做的根因:如果平台已经在限流你,继续高频编辑只会雪上加霜,不如退一步保证消息至少能发出去。

| `signal.py` | Signal | signal-cli REST JSON-RPC 2.0 + SSE、批量附件、限流调度器 |

读这张表要抓住几条规律:

- **同一平台多后端**:`whatsapp.py` 支持三种后端(官方 Business API、whatsapp-web.js、Baileys),因为 WhatsApp 没有统一的对接方式,不同后端在合规性、稳定性、功能上各有取舍。
- **富文本 / 卡片能力差异巨大**:Telegram 用 MarkdownV2 且支持 inline 按钮、DM Topic(私聊里的话题分区)和原生草稿,因此它的适配器最庞大(6,081 行);飞书 / 钉钉 / 元宝主打"卡片消息"(结构化富卡片);而 Email / SMS 几乎只有纯文本。这也是为什么 §4.1 的渲染要下放给各平台自己——能力天差地别。
- **限流与加密各自为政**:Signal 自带"限流调度器"(signal-cli 的发送速率有限),企业微信要做"加密回调"(回调内容是加密的,需专门解密,见 `wecom_crypto.py` / `wecom_callback.py`)。这些都是平台强加的真实约束,必须在适配器里逐一处理。

> 注:`gateway/platforms/` 目录下除上表外还有辅助模块(如 `signal_rate_limit.py`、`telegram_network.py`、`yuanbao_*.py` 系列、`qqbot/` 子目录、`homeassistant.py`、`webhook.py` 等),它们或是上述适配器的拆分实现,或是更小众的接入点。完整可注册平台总数为 31 个(见 §2 基类描述)。

## 六、GatewayRunner(run.py):整个网关进程的生命周期编排

**文件**:`gateway/run.py`。`GatewayRunner` 是核心编排器,管理从启动到优雅关闭的完整生命周期。下面这棵调用树展示它都做了什么:

```
start_gateway()
  ├── 重复实例检测(PID 文件)
  ├── start() → 异步启动
  │   ├── 安全告警检查
  │   ├── 发现 Python 插件
  │   ├── connect_platforms() → 并发连接所有适配器
  │   ├── 启动会话过期监控
  │   ├── 启动平台重连任务
  │   ├── 初始化 cron 调度器
  │   └── 后台进程恢复
  ├── [_running loop] → 等待 shutdown_event
  └── stop(restart) → 优雅关闭
      ├── 通知所有活跃会话
      ├── _drain_active_agents(timeout)
      ├── 断开所有适配器
      └── 释放平台锁
```

关键环节逐条解释:

- **重复实例检测(PID 文件)**:防止同一个网关被启动两次。启动时拿到当前进程的 PID(`run.py:3908`,`os.getpid()`),并通过 PID 文件检查是否已有实例在跑——两个网关同时连同一个平台会互相抢消息。
- **`connect_platforms()`**:并发地把所有配置好的适配器连上各自平台(而不是串行一个个连,省启动时间)。
- **会话过期监控 + 平台重连任务**:前者按 §3.2 的重置策略清理过期会话;后者在某个平台掉线时自动重连。
- **cron 调度器**:支持定时触发的 agent 任务(网关内置的计划任务能力)。
- **后台进程恢复**:配合 §3.2 的 `resume_pending`,在重启后把崩溃前的活跃会话接续起来。
- **`stop(restart)` 优雅关闭**:先通知所有活跃会话即将关闭,再调 `_drain_active_agents(timeout)`(`run.py:3532`)给正在跑的 agent 一段时限把手头的回答收尾(drain = 排空,等在途任务自然结束而不是硬杀),超时后才断开适配器、释放锁。`restart` 参数区分"关停"和"重启"。整个等待由 `_shutdown_event`(`run.py:1812`)驱动。

### Agent 缓存调优:为什么是 128 个 + 1 小时 TTL

网关为活跃会话缓存"已构建好的 agent / 会话来源"对象,避免每条消息都从头重建。这个缓存(`_agent_cache`,`run.py:1870`,一个 `OrderedDict`)用两条规则控制内存:

- **硬上限 128 个**:`_AGENT_CACHE_MAX_SIZE = 128`(`run.py:64`)。超过就按 LRU(最近最少使用)淘汰最久没碰过的那个。用 `OrderedDict` 正是为了能 O(1) 地把最久未用的弹出。
- **空闲 1 小时 TTL**:`_AGENT_CACHE_IDLE_TTL_SECS = 3600.0`(`run.py:65`,即 1 小时)。一个 agent 闲置超过 1 小时就被会话过期监控(`_session_expiry_watcher`)按 session_key 逐出。

这两条规则合在一起:既限制同时在内存里的活跃 agent 数量(防内存爆),又及时回收长时间没人用的会话(防泄漏)。LRU 顺序和 TTL 逐出分别由 `_enforce_agent_cache_cap()` 和 `_session_expiry_watcher()` 强制执行(`run.py:62-67`)。

## 七、投递路由(delivery.py):一条消息该送到哪去

**文件**:`gateway/delivery.py`。`DeliveryRouter` 解决"agent 产出的消息应该投递到哪个目的地"的问题——尤其是 agent 主动外发(不一定是回复来源聊天)的场景。目的地由 `DeliveryTarget.parse()`(`delivery.py:112`)解析一个字符串目标得到:

- `"origin"` → 回到发出命令的那个聊天(`delivery.py:125`,即"原路返回")。
- `"local"` → 不发任何平台,仅保存到本地文件(`delivery.py:101`)。
- `"telegram"` → 发到 Telegram 的 home channel(主频道,`chat_id` 为 None 时用主频道,`delivery.py:106`)。
- `"telegram:123456"` → 发到 Telegram 上指定的某个聊天(冒号后是具体 chat_id)。
- `"telegram:123456:topic_name"` → 发到某个 DM topic 内(若该 topic 不存在会自动创建)。

这套"平台:聊天:话题"的分层目标语法,让 agent 能把不同内容精准投递到不同位置(比如把日志存本地、把结论发主频道、把某事的进展发到专属话题)。

**反循环保护——为什么需要它**:`_is_silence_narration()`(`delivery.py:38`)用来拦截"纯静默标记"。当 agent 决定"这轮不需要回话"时,它可能输出一个表示沉默的占位符(例如 `(silent)`、`🔇`,匹配正则见 `delivery.py:32`)。如果不拦截,这种占位符被当成正常消息发出去,在 bot 与 bot 互相对话的场景下会引发**无限回响**(A 发"沉默"→ B 当成消息回"沉默"→ A 再回……)。这个函数把这类纯静默标记识别出来并丢弃,从而切断死循环。值得注意的是它做了精细处理:像 "The deployment ran silently"、"Silence is golden" 这种正常含有 "silent/silence" 的正经文本**不会**被误杀(`delivery.py:42`),只匹配"整条消息就是个沉默标记"的情况。

## 八、平台注册表(platform_registry.py):插件式平台怎么自注册

**文件**:`gateway/platform_registry.py`。`platform_registry`(`platform_registry.py:260`)是一个**模块级单例**(全进程唯一的一份注册表),用来支持"插件式平台适配器自注册"——也就是第三方可以在不改 Hermes 核心代码的前提下,把一个新平台挂进网关。`PlatformEntry`(`platform_registry.py:39`)是注册时填的条目结构,`register()`(`platform_registry.py:172`)是注册入口。一个注册示例:

```python
platform_registry.register(PlatformEntry(
    name="irc", label="IRC",
    adapter_factory=lambda cfg: IRCAdapter(cfg),
    check_fn=check_requirements,
    validate_config=lambda cfg: bool(cfg.extra.get("server")),
    required_env=["IRC_SERVER"],
))
```

逐字段解释:`name` 是平台内部标识;`label` 是展示名;`adapter_factory` 是一个工厂函数,给它配置就能造出该平台的适配器实例;`check_fn` 检查运行依赖是否齐全;`validate_config` 校验配置是否有效(此例要求 `cfg.extra` 里必须有 `server`);`required_env` 列出必需的环境变量(此例 `IRC_SERVER`)。注册完成后,通过 `create_adapter("irc", config)`(`platform_registry.py:28`)即可按名创建。

**查找顺序**:网关解析某个平台时,**优先查注册表**(看有没有插件注册过这个名字);**未命中时再回退到内置的 if/elif 分支**(Hermes 自带的那批平台)。这种"先插件、后内置"的顺序,既让内置平台开箱即用,又给第三方扩展留了不改核心的入口。
