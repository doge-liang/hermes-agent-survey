# Hermes Agent 网关与多平台架构深度分析

## 一、整体架构

Gateway是一个**多平台消息代理**，采用**适配器模式 + 事件驱动流式管道**：

```
用户消息 → 平台适配器(Adapter) → GatewayRunner → AI Agent → StreamConsumer → 平台适配器 → 用户
```

## 二、BasePlatformAdapter ABC (base.py, 4781行)

所有31个平台适配器的抽象基类，第1764行定义。

**必须实现：**
- `connect()` / `disconnect()` — 平台连接生命周期
- `send(chat_id, content, ...)` — 发送消息

**可选重写（默认降级行为）：**
- `edit_message()` / `delete_message()` — 流式编辑/消息删除
- `send_image/audio/video/document()` — 原生媒体发送
- `send_draft()` — 原生流式草稿（Telegram Bot API 9.5+）
- `send_typing()` / `stop_typing()` — 输入状态指示器
- `send_clarify()` — 多选/开放式澄清提示
- `render_message_event()` — 结构化流事件渲染
- `format_tool_event()` — 工具调用进度渲染

**并发控制：**
- `_active_sessions`：每session的中断Event
- `_pending_messages`：session忙碌时排队消息
- `_session_tasks`：精确的task取消追踪
- `_busy_text_mode`：忙碌处理模式（queue/interrupt）
- `_text_debounce`：消息消抖合并

**关键数据结构：**
- `MessageEvent`：标准化入站消息（text, message_type, source, media_urls...）
- `SendResult`：发送结果（success, message_id, error, retryable...）
- `EphemeralReply`：带TTL的自动删除回复

## 三、会话管理 (session.py)

### SessionKey构建规则

```
格式: agent:main:<platform>:<chat_type>:<chat_id>[:thread_id][:<user_id>]

DM会话: 按chat_id和thread_id隔离
群组/频道:
  - group_sessions_per_user=True (默认): 每个用户独立会话
  - thread_sessions_per_user=False: 线程内用户共享会话
WhatsApp: 统一JID/LID别名
```

### SessionStore

- SQLite (SessionDB) 主存储 + JSONL降级
- 线程安全（GIL保护读 + `_lock`保护写）
- 重置策略：`none`/`idle`/`daily`/`both`
- `resume_pending`标记：gateway崩溃后自动恢复会话
- PII红名支持：WhatsApp/Signal/Telegram默认哈希用户ID

## 四、流式输出管道

### 4.1 结构化流事件 (stream_events.py)

类型化事件词汇表，解耦语义输出与平台渲染：

| 事件 | 含义 |
|------|------|
| `MessageChunk(text)` | 增量文本delta |
| `MessageStop(final)` | 文本段落结束 |
| `Commentary(text)` | 工具调用间的过渡消息 |
| `ToolCallChunk(...)` | 工具调用开始 |
| `ToolCallFinished(...)` | 工具调用完成 |
| `LongToolHint(...)` | 长时间工具提示 |
| `GatewayNotice(...)` | gateway原生控制消息 |

### 4.2 流消费器 (stream_consumer.py, 1346行)

```
Agent线程(sync) → Queue → GatewayStreamConsumer(async) → 平台适配器
                                ├── 积累文本到_accumulated
                                ├── 按间隔/阈值批量刷新
                                ├── 通过edit_message渐进编辑
                                ├── 原生草稿流(draft streaming)
                                ├── 溢出分块(>平台限制)
                                └── 完成时最终编辑
```

**传输模式：** `edit`(渐进编辑) / `draft`(原生草稿) / `auto`(优先草稿) / `off`

**错误恢复：** 连续3次flood control → 禁用渐进编辑 → fallback send

## 六、GatewayRunner (run.py)

核心编排器，完整生命周期管理：

```
start_gateway()
  ├── 重复实例检测(PID文件)
  ├── start() → 异步启动
  │   ├── 安全告警检查
  │   ├── 发现Python插件
  │   ├── connect_platforms() → 并发连接所有适配器
  │   ├── 启动会话过期监控
  │   ├── 启动平台重连任务
  │   ├── 初始化cron调度器
  │   └── 后台进程恢复
  ├── [_running loop] → 等待shutdown_event
  └── stop(restart) → 优雅关闭
      ├── 通知所有活跃会话
      ├── _drain_active_agents(timeout)
      ├── 断开所有适配器
      └── 释放平台锁
```

**Agent缓存调优：** LRU缓存(Max 128个, 1h空闲TTL)、按session_key的过期监控

## 七、投递路由 (delivery.py)

`DeliveryRouter` 将消息路由到正确目的地：

**目标格式：**
- `"origin"` → 回到发送命令的聊天
- `"local"` → 仅保存到本地文件
- `"telegram"` → Telegram home channel
- `"telegram:123456"` → 特定聊天
- `"telegram:123456:topic_name"` → DM topic内（自动创建）

**反循环保护：** `_is_silence_narration()` 拦截纯静默标记（`(silent)`, `🔇`），防止bot-to-bot无限回响。

## 八、平台注册表 (platform_registry.py)

模块级单例，支持插件式平台适配器自注册：

```python
platform_registry.register(PlatformEntry(
    name="irc", label="IRC",
    adapter_factory=lambda cfg: IRCAdapter(cfg),
    check_fn=check_requirements,
    validate_config=lambda cfg: bool(cfg.extra.get("server")),
    required_env=["IRC_SERVER"],
))
```
