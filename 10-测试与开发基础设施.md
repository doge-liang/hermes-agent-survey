# Hermes Agent 测试与开发基础设施深度分析

> **一句话**:这份报告讲的是"Hermes 这套代码是怎么被测试、怎么被开发出来的"——它的测试目录怎么分层、为什么要自研一个并行测试运行器、它怎么在测试时把整台机器"密封"起来不让测试误伤真实系统,以及一批支撑日常开发的底层工具(进程自举、异步、重试、流式诊断、国际化、日志)。它不涉及 agent hint / KV 调度那条主线(那是报告 12–17 的事),而是回答"想给 Hermes 加功能或改 bug 时,脚手架长什么样、有哪些坑已经被前人填过"。
>
> **读者画像**:你懂推理服务 / LLM 工程,但没读过 Hermes 源码。下文出现的每个 Hermes 专有概念都会先解释再用。
>
> **几个先打底的名词**:
> - **Hermes** 是 Nous Research 出的一个 LLM agent 框架(命令行工具 `hermes`),能跑对话、调工具、接多种推理后端。
> - **gateway**(网关)是 Hermes 里负责对接外部消息平台(Telegram、Discord 等)的常驻服务进程,在 Linux 上以 systemd 单元 `hermes-gateway` 形式运行。后文"实时系统保护"会反复提到它——因为测试最怕误杀这个正在跑的生产进程。
> - **HERMES_HOME** 是 Hermes 的工作根目录(默认在用户主目录下),里面放配置 `config.yaml`、会话数据库、日志等。测试隔离的核心就是给每个测试换一个临时的 HERMES_HOME。
> - **conftest.py** 是 pytest 的约定文件:放在测试目录里,pytest 启动时会自动加载,常用来定义所有测试共享的 fixture(夹具)和全局钩子。

## 一、测试体系

### 1.1 目录结构:1270+ 个测试文件按职责分到 33 个子目录

整个仓库大约有 **1270+ 个测试文件**,分布在 33 个子目录里。之所以按子目录切分,是因为 Hermes 本身就是分层的(CLI 解析层、网关层、工具层、agent 核心、对话循环……),测试目录基本是对这些层的一一镜像——看测试目录的体量,也能反推出各层的复杂度。下表列出文件数最多的若干目录,以及每个目录管的是什么:

| 目录 | 文件数 | 说明 |
|------|--------|------|
| `tests/hermes_cli/` | ~292 | CLI 命令解析、配置加载 |
| `tests/gateway/` | ~281 | 网关路由、消息平台、API 端点 |
| `tests/tools/` | ~240 | 工具执行、调度、参数验证 |
| `tests/agent/` (含 lsp、transports) | ~143 | 核心 agent 逻辑、LSP、传输层 |
| `tests/run_agent/` | ~101 | 端到端对话循环、重试、压缩、中断 |
| `tests/cli/` | ~74 | CLI 交互、文件拖放、配置命令 |
| `tests/plugins/` | ~27 | 插件系统、浏览器、图像生成、记忆、语音 |
| `tests/cron/` | ~15 | 定时任务调度 |
| `tests/stress/` | ~8 | 并发、竞态、基准、模糊测试 |
| `tests/skills/` | ~11 | 技能系统 |
| `tests/docker/` | ~10 | Docker 构建、浏览器发现 |
| `tests/e2e/` | ~3 | 端到端(需真实网关) |

怎么读这张表:文件数最多的三块是 `hermes_cli`(~292)、`gateway`(~281)、`tools`(~240),说明"命令行解析 + 配置"、"网关与多平台对接"、"工具系统"是 Hermes 测试压力最集中的三个面——这也符合直觉,因为它们直面用户输入和外部世界,边界条件最多。`run_agent/`(~101)虽然文件数不算最多,但它测的是 Hermes 的心脏:**对话循环(conversation loop)**——也就是"发请求→收响应→调工具→把工具结果塞回去再发→……"这个反复迭代直到任务完成的主循环,外加重试、上下文压缩、中断处理等。报告 13–17 反复剖析的那条主线,源码就主要落在这一块对应的实现里。`tests/e2e/`(~3)只有寥寥几个文件,因为它要求"真实网关"才能跑,门槛高、跑得慢,所以数量克制。

### 1.2 测试分类:四种测试,跑法和依赖各不相同

Hermes 把测试按"依不依赖外部世界、测的是逻辑还是并发"分成四类。区分它们的实际意义是:有的能在任何机器上秒级跑完,有的必须连真实服务、只能在 CI(持续集成)里单独开一个作业跑。

- **Unit tests(单元测试)**:纯逻辑测试,不碰任何外部服务。绝大多数测试属于这类,快且稳定。
- **Integration tests(集成测试)**:用 `@pytest.mark.integration` 标记。"标记(marker)"是 pytest 给测试贴的标签,可以据此筛选哪些测试跑、哪些不跑。这类测试需要外部服务(真实网关、真实后端之类)才能跑,所以在 CI 里被拎出来当作**独立作业**单独运行,不和单元测试混在一起——避免外部依赖不稳定时拖垮整条流水线。
- **Stress tests(压力测试)**:专门压并发、竞态(race condition)和模糊测试(fuzzing,用随机/畸形输入轰炸代码找崩溃)。它们故意制造极端条件,跑得慢,因此独立成 `tests/stress/`。
- **Platform tests(平台测试)**:针对各个消息平台适配器(Telegram、Discord 等不同平台的对接代码)的测试。每个平台的消息格式、回调机制都不一样,需要各自的适配测试。

## 二、并行测试运行器:为什么自研 `scripts/run_tests_parallel.py`

### 2.1 设计哲学:按"文件"并行,而不是按"测试"并行

`scripts/run_tests_parallel.py` 是 Hermes 自研的并行测试运行器。它存在的理由,得先理解 Python 测试界的默认选项 **pytest-xdist**——这是 pytest 官方的并行插件,做法是预先起一批常驻 worker 进程,把成千上万个测试函数分派给它们跑。Hermes 没用它,而是自己写了一个**按文件并行(per-file)而非按测试并行(per-test)**的运行器。

### 2.2 为什么不用 xdist:两个具体痛点

**痛点一:模块级状态泄露。** xdist 的 worker 是持久的——一个 worker 在它生命周期里会接连跑很多个测试文件。Python 的"模块级状态"(import 时执行的全局变量、单例、缓存、注册表等)在同一个进程里是不会自动清空的。于是 A 文件跑完留下的全局状态,会"漏"到同一个 worker 接着跑的 B 文件里,造成测试之间互相污染、结果不可复现。Hermes 的对策是:**每个文件用一个全新的子进程跑,跑完就退**,进程一退,所有模块级状态随之消失,从根上杜绝跨文件泄露。

**痛点二:进程启动开销。** Hermes 的测试有一笔固定的"每进程启动开销",约 **250ms**(主要是 import 一堆重模块、初始化环境)。这笔开销乘以"要起多少个进程"就是总浪费:

- 如果**按测试粒度**并发(像 xdist 极端情况那样每个测试一个进程),约 **17000 个测试 × 250ms ≈ 70 分钟**纯启动开销——光是反复起进程就耗掉一个多小时,实际测试逻辑还没算。
- 如果**按文件粒度**并发,只需 **~850 个文件 × 250ms ≈ 3.5 分钟**启动开销。

两者差了 20 倍。这就是"按文件并行"的核心动机:把 250ms 这笔固定成本摊到"文件数"(几百)上,而不是"测试数"(上万)上。

> 注:1.1 节说约 1270+ 测试文件,这里说 ~850 个文件——前者是仓库里测试文件的总数,后者是这个运行器实际分发并发执行的文件数量级(用于估算启动开销),二者口径不同,不矛盾。

### 2.3 五个关键特性

1. **进程组隔离**:每个子进程都用 `start_new_session=True` 启动,意思是它会成为一个**独立的进程组/会话**的组长。这样做的好处是:当某个测试文件超时需要被强杀时,可以直接对整个进程组发 **SIGKILL**(强制杀死信号),把这个测试派生出来的所有子孙进程一锅端,不会留下游离的僵尸子进程。
2. **文件级超时**:每个测试文件默认 **600 秒**上限,由 `--file-timeout` 调整。超过就按上面的方式杀掉整个进程树。
3. **切片分发(用于分布式跑)**:`--slice I/N` 表示"我是 N 个跑测试的机器/分片里的第 I 个,只跑分给我的那一份"。怎么分才均匀?它读一个缓存文件 `test_durations.json`(记录每个文件历史耗时),用 **LPT 算法**(Longest Processing Time,最长作业优先——先把最耗时的文件分出去,逐步填平各分片的总时长)把文件均匀摊到 N 个分片上,避免某台机器分到一堆慢文件而其他机器早早跑完干等。
4. **退出码处理**:pytest 的**退出码 5** 意思是"没有收集到任何测试"。在普通 pytest 里这算异常,但在按文件并发的场景下,某个文件恰好一个测试都没有是很正常的(比如该文件的测试全被 marker 过滤掉了),所以运行器把退出码 5 **视为通过**,不报失败。
5. **实时进度**:跑的过程中实时刷新通过 / 失败 / 跳过的计数,并即时打印失败详情,不用等全部跑完才看到结果。

## 三、`conftest.py`:把测试机器"密封"起来的 5 层防护

测试 Hermes 有个特殊风险:它是个会真实操作系统、起子进程、连网络、读写凭据的 agent 框架。如果测试不加约束地跑,可能误删真实配置、误杀正在运行的生产网关、把开发者的真实 API key 泄进日志、或因为读取真实系统时钟 / 时区导致结果飘忽。为此,Hermes 在 `conftest.py` 里堆了 **5 层"密封环境(sealed environment)"**——每一层堵一类风险。下面逐层解释它在防什么、怎么防。

### 层级 1:凭据泄露防护

测试进程会继承开发者 shell 里的所有环境变量,其中往往包含真实的 API key、token、secret。这一层在测试开始前**自动清除所有"凭据形状"的环境变量**:既按后缀模式匹配(凡是以 `_API_KEY`、`_TOKEN`、`_SECRET` 等结尾的),也维护一份 **150+ 个显式变量名**的清单逐一清掉。目的是确保测试里任何代码都拿不到真实凭据——既防止真请求逃逸到外部、也防止凭据被打进日志。

### 层级 2:HERMES_HOME 隔离

如前所述,HERMES_HOME 是 Hermes 的工作根目录。这一层给**每个测试**单独发一个临时目录 `tmp_path/hermes_test` 当作它的 HERMES_HOME(`tmp_path` 是 pytest 提供的每测试唯一临时目录),并预先把必要的子目录建好。这样每个测试都在自己干净的沙盒里读写配置 / 数据,彼此不串、也绝不碰开发者真实的 HERMES_HOME。同时,它还隔离了 **200+ 个 `HERMES_*` 行为变量**(这些环境变量会改变 Hermes 的运行行为,不隔离的话测试结果会受开发者本地设置影响),以及所有平台相关的 home channel / allowlist 变量(消息平台的默认频道、白名单等)。

### 层级 3:确定性环境

测试要可复现,就不能依赖"今天几点、在哪个时区、本地用什么编码"。这一层强制三个变量固定下来:`TZ=UTC`(时区锁定为 UTC)、`LANG=C.UTF-8`(语言 / 编码锁定)、`PYTHONHASHSEED=0`(锁定 Python 哈希随机化的种子——否则 set / dict 的迭代顺序每次都变,会让某些测试时过时不过)。此外还**禁用 AWS IMDS 元数据查询**:IMDS(Instance Metadata Service)是 AWS 云主机上一个固定地址的内部服务,SDK 启动时常会去探一下"我是不是跑在 EC2 上"。在非 AWS 环境(比如开发者笔记本)这个探测会**白等约 2 秒超时**才失败——禁掉它能省下这笔无谓延迟。

### 层级 4:实时系统保护(`_live_system_guard`)

这是最关键的一层,专门防"测试误伤真实运行中的系统"。它通过 monkeypatch(运行时替换函数)在测试期间架起一道防线,拦截一切可能伤及真实进程 / 服务的操作:

- 拦截 `os.kill` / `os.killpg`:不允许测试向**自己进程树之外**的进程发信号(防止误杀真实进程)。
- 拦截所有可能操作 `hermes-gateway` systemd 单元的 `systemctl` 命令:防止测试把正在跑的生产网关给 stop / restart 掉。
- 拦截 `pkill` / `killall` / `taskkill` 这类"按名字批量杀进程"的命令:这些命令一旦匹配到名字相近的真实进程就是灾难。
- 覆盖 **9 种子进程启动 API**,把它们全套上护栏:`subprocess.run` / `Popen` / `call`、`os.system` / `popen`、`pty.spawn`、`asyncio.create_subprocess_exec` / `create_subprocess_shell`。之所以要覆盖这么多入口,是因为 Python 里"起一个外部进程"有很多条路,堵一个不够,得把所有口子都管住。
- 提供逃生舱:确实需要真起子进程 / 真发信号的测试,可以用 `@pytest.mark.live_system_guard_bypass` 标记**选择退出**这层保护(opt-out),不至于把合法的集成测试也卡死。

### 层级 5:异步兼容

Python 3.12+ 收紧了 asyncio 的行为:如果一个同步测试里间接用到了需要 event loop(事件循环)的代码却没有现成的 loop,会抛 `DeprecationWarning`(弃用警告)。这一层为同步测试预备一个默认 event loop,消除这类噪声警告,让测试输出干净、也为将来 Python 版本继续收紧做好准备。

## 四、Fake 测试替身:`tests/fakes/` 里的 FakeHAServer

测试要验证 Hermes 和外部服务的交互,但又不能真连外部服务(慢、不稳定、有副作用)。常规做法是用 mock(打桩),但 mock 往往只能伪造单次调用的返回值,测不了"完整握手 + 状态变化"这种有协议、有状态的交互。Hermes 的做法更彻底:写一个**功能完整的假服务器(fake)**,行为像真的一样,但完全受测试掌控。代表作是 **FakeHAServer**——Home Assistant(开源智能家居平台,简称 HA)服务器的完整假件。它的能力分四面:

- **真实 HTTP + WebSocket 服务器**:基于 `aiohttp.web`(Python 的异步 Web 框架)真起一个本地服务器,Hermes 用真实网络栈连它,而不是被 mock 拦在半路。
- **实现 HA 的 WebSocket 认证握手流程**:HA 连接要走一套特定的认证握手,这个 fake 把握手协议完整实现了,所以能测到 Hermes 认证逻辑的真实分支,而不是跳过握手。
- **提供 REST API 端点**:如 `/api/states`(查实体状态)、`/api/services/...`(调用服务),让 Hermes 的 HA 客户端代码有真实端点可打。
- **可观测性 + 可控性**:
  - **可观测**:暴露 `received_service_calls`(收到了哪些服务调用)、`received_notifications`(收到了哪些通知),测试据此断言"Hermes 确实发了正确的请求"。
  - **可控**:提供 `reject_auth`(强制拒绝认证)、`force_500`(强制返回 500 错误)等开关,让测试能主动制造失败场景,验证 Hermes 的错误处理和重试逻辑。

## 五、pytest 配置:默认安全的两条铁律

测试框架的全局配置写在 `pyproject.toml` 的 `[tool.pytest.ini_options]` 段里。它定义了所有测试默认怎么跑:

```toml
[tool.pytest.ini_options]
testpaths = ["tests"]
markers = [
    "integration: marks tests requiring external services",
    "real_concurrent_gate: ...",
]
addopts = "-m 'not integration' --timeout=30 --timeout-method=signal"
```

逐项解释:`testpaths = ["tests"]` 指定测试只在 `tests/` 目录里找;`markers` 段声明了自定义标记(integration 标记需要外部服务的测试、real_concurrent_gate 等),声明后 pytest 才认这些标签、不报"未知 marker"警告。

最关键的是 `addopts`(默认追加的命令行参数),它定下两条默认安全的铁律:

1. **`-m 'not integration'`——默认跳过集成测试**:开发者本地随手跑 pytest 时,只跑不依赖外部服务的单元测试,不会因为没连外部服务而一片失败;集成测试只在 CI 里显式打开。
2. **`--timeout=30 --timeout-method=signal`——30 秒硬超时**:任何单个测试跑超过 30 秒就被强制终止,防止某个卡死的测试无限挂起拖垮整轮。`signal` 方法是用操作系统信号来实现超时中断(相对线程方式更能打断卡在阻塞调用里的测试)。

## 六、核心开发基础设施:6 个支撑日常开发的底层工具

除了测试本身,Hermes 还有一批底层工具模块,专门解决"开发一个长跑的、连网络的、多线程 / 异步的 agent 进程"会遇到的共性麻烦。下面逐个讲它们各自踩过什么坑、怎么填的。

### 6.1 进程启动自举(`process_bootstrap.py`)

这个模块管"Hermes 进程刚启动那一刻"要做的准备工作,核心是三件:

- **懒加载 OpenAI SDK**:`import openai` 这个 SDK 本身就要 **~240ms**,但很多命令(比如只查个配置)根本用不到它,白白付这笔启动延迟不划算。对策是用一个代理对象 `_OpenAIProxy` 把真正的 import **推迟到第一次实际用到时**才发生。难点在于:推迟 import 不能破坏既有代码——既有代码会用 `isinstance(x, openai.Xxx)` 做类型判断、测试会用 `patch()` 打桩 openai 的符号。`_OpenAIProxy` 特意保持了对 `isinstance()` 和 `patch()` 的兼容,让"懒加载"对调用方透明。
- **崩溃安全的标准 IO**:进程退出 / 管道断开时,往 stdout / stderr 写东西可能抛两类异常——`OSError`(典型是 broken pipe,管道另一端已关闭)和 `ValueError`(线程退出时底层文件句柄已被关闭)。这些异常本身无害,但会污染退出日志甚至掩盖真正的退出原因。`_SafeWriter` 把这两类异常**静默吞掉**,保证进程能干净收尾。
- **HTTP 代理解析**:读取代理相关的环境变量(`HTTP_PROXY` 等)来决定请求走不走代理,同时**尊重 `NO_PROXY` 排除规则**(某些主机명单要直连、不走代理)。

### 6.2 异步工具(`async_utils.py`)

核心函数 `safe_schedule_threadsafe()` 是对标准库 `asyncio.run_coroutine_threadsafe()` 的安全包装。后者的作用是"从一个普通线程里,把一个协程(coroutine)安全地丢给另一个线程的 event loop 去跑"。问题在于:如果调度的那一刻 event loop 已经关闭了,标准函数会留下一个**从未被 await 的协程**,Python 会抛 "coroutine was never awaited" 警告,还会造成**帧泄漏**(协程对象持有的栈帧 / 引用无法释放)。`safe_schedule_threadsafe()` 在 loop 已关闭时主动 `close()` 掉这个协程,既消警告又不泄漏——这是长跑进程在关闭阶段非常容易踩的一类坑。

### 6.3 重试工具(`retry_utils.py`)

核心函数 `jittered_backoff()` 实现"带去相关抖动的指数退避(decorrelated jittered exponential backoff)"。这是请求失败后重试的标准策略,逐项拆解:

- **指数退避**:失败后等待时间随重试次数指数增长,给后端喘息空间。
- **抖动(jitter)**:在等待时间上叠加随机扰动,避免大量客户端在同一时刻一起重试造成"惊群"(thundering herd)同步打击后端。
- 公式:`min(base * 2^(attempt-1), max_delay) + random_jitter`——即"基准乘以 2 的幂,但封顶在 max_delay,再加一段随机抖动"。
- 默认参数:`base=5s`(首次退避基准 5 秒)、`max_delay=120s`(退避上限 2 分钟)、`jitter_ratio=0.5`(抖动幅度为基准的一半)。
- **去相关随机种子**:用 `time.time_ns() XOR 计数器` 生成随机种子,保证**不同线程 / 不同会话**拿到互不相关的随机序列——否则多个并发重试者用同一个种子,抖动就"相关"了,惊群问题又回来了。

### 6.4 流式诊断(`stream_diag.py`)

LLM 的流式(streaming)响应一旦出问题(中途断流、TTFB 异常、provider 报错),事后排查极难——因为是边收边丢的流。`stream_diag.py` 给每个流式请求挂了一条完整的诊断管线,把关键信息全程记录下来:

- **头部捕获**:抓取响应头里对排障有用的字段,如 `cf-ray`(Cloudflare 请求追踪 ID)、`x-openrouter-provider`(OpenRouter 实际路由到的底层 provider)、`x-request-id` 等。
- **异常链展平**:Python 异常常常一层套一层(`raise ... from ...`)。这里把异常链**展平**,最大展开深度 4 层,每层文本截断到 140 字符——既保留根因链路,又不让日志被超长堆栈撑爆。
- **结构化日志**:把诊断信息记成结构化字段,包括 `subagent_id`(子 agent 标识)、`depth`(委派 / 嵌套深度)、`provider`、`base_url`、`error_type`、`TTFB`(Time To First Byte,首字节延迟)、`elapsed`(总耗时)。结构化便于事后聚合分析。
- **用户可见线条**:在终端上给用户一条**紧凑的实时状态更新**(区别于上面给机器看的详细结构化日志),让用户知道流还在动。

### 6.5 国际化(`i18n.py`)

i18n(internationalization 的缩写,"i" 和 "n" 之间 18 个字母)即让界面文案支持多语言。Hermes 支持 **16 种语言**:en、zh、zh-hant、ja、de、es、fr、tr、uk、af、ko、it、ga、pt、ru、hu(英、简中、繁中、日、德、西、法、土、乌克兰、南非荷兰、韩、意、爱尔兰、葡、俄、匈)。机制如下:

- **翻译目录**:每种语言一个 YAML 文件 `locales/<lang>.yaml`,里面是"键→译文"的映射。
- **语言解析优先级**:显式参数 > 环境变量 `HERMES_LANGUAGE` > `config.yaml` 配置 > 默认 `"en"`。即调用时显式指定的优先级最高,逐级回退到默认英文。
- **灵活的语言标识**:支持自然语言别名(比如写 "Chinese")和 **BCP-47** 标签(标准语言标签格式,如 `zh-Hant`),不要求调用方用某种死板写法。
- **多级缺失回退**:如果当前语言缺某个键,回退到英文;如果英文也缺,**直接返回键名本身**而不是崩溃。这个"绝不因为缺翻译而崩"的设计很重要——文案缺失是常态,不该让程序挂掉。

### 6.6 日志系统(`hermes_logging.py`)

一个多组件、长跑、并发的 agent 进程,日志极易乱成一锅粥。`hermes_logging.py` 用几项机制把它理顺:

- **按文件分流**:不同严重级别 / 不同组件写到不同文件——`agent.log`(INFO 及以上)、`errors.log`(WARNING 及以上,专放问题)、`gateway.log`(网关)、`gui.log`(图形界面)。
- **组件分离**:`_ComponentFilter` 根据 **logger(记录器)名称前缀**把日志路由到对应文件——Python 的 logger 名通常是带点的层级名(如 `hermes.gateway.xxx`),按前缀就能判断它属于哪个组件。
- **会话上下文注入**:用**线程本地(thread-local)**的 `set_session_context(session_id)` 把当前 `session_id` 绑到线程上,再自动注入到每条 `LogRecord`。这样并发处理多个会话时,每条日志都带着自己的 `session_id`,排障时能把同一会话的日志串起来。"线程本地"意味着不同线程互不干扰,各记各的会话。
- **密文屏蔽**:所有日志处理程序统一套 `RedactingFormatter`,在写盘前把疑似密钥 / token 的内容打码,防止凭据随日志泄露——这和 conftest 层级 1 的凭据防护是一前一后两道闸。
- **日志轮转 + 外部轮转兼容**:`_ManagedRotatingFileHandler` 负责日志文件轮转(写满就切新文件),默认 **5MB / 保留 3 个备份**。它还带**外部轮转检测**——很多生产环境用系统级的 `logrotate` 工具来轮转日志,如果 Hermes 自己和 logrotate 各转各的会打架;这个检测让两者安全共存(logrotate 把文件挪走后,Hermes 能察觉并重新打开新文件)。
- **第三方库噪声抑制**:把 OpenAI SDK、httpx、httpcore、asyncio 等依赖库的日志级别压到 **WARNING**,过滤掉它们大量的 INFO/DEBUG 噪声,让 Hermes 自己的日志不被淹没。

## 七、小结

报告 10 描绘的是 Hermes 的"工程地基"而非"功能特性":

- **测试体系**(§1)按层镜像代码结构,1270+ 文件分 33 目录,四类测试各有跑法,默认安全(默认跳集成、30 秒硬超时)。
- **自研并行运行器**(§2)用"按文件并行"把 250ms 的每进程启动开销从 70 分钟级压到 3.5 分钟级,并用进程组隔离 + 文件超时 + LPT 切片把并行跑做稳。
- **5 层密封环境**(§3)层层设防,把测试关进沙盒,既防凭据泄露、又防误杀真实网关、还保证可复现。
- **Fake 替身**(§4)用功能完整的假服务器替代脆弱 mock,测得到完整协议握手和状态变化。
- **6 个底层工具**(§6)把长跑 agent 进程的共性麻烦——懒加载、异步收尾、重试抖动、流式排障、多语言、多组件日志——逐个填平。

这份报告与 agent hint / KV 调度主线(报告 12–17)相对独立:它回答的不是"Hermes 给后端发了什么",而是"想动 Hermes 的代码时,脚手架长什么样、哪些坑已经有人替你填过"。若需把信号采集 / 测试台真正落到这套测试基础设施之上,可结合报告 15(真实 CLI 探测测试台)与报告 17(信号提取规格)来看。
