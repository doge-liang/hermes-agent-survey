# Hermes Agent 内存与上下文管理架构深度分析

## 一、内存Provider体系

### 1.1 MemoryProvider ABC

插件化内存后端的标准接口：

| 方法 | 职责 |
|------|------|
| `initialize(session_id)` | 会话启动时建立连接 |
| `system_prompt_block()` | 静态文本块注入system prompt |
| `prefetch(query, session_id)` | 每轮前返回缓存记忆上下文 |
| `queue_prefetch(query, session_id)` | 每轮后排队后台预取 |
| `sync_turn(user, assistant, session_id, messages)` | 每轮后持久化 |
| `get_tool_schemas()` / `handle_tool_call()` | 工具暴露与调用 |
| `shutdown()` | 清理关闭 |

**可选钩子：** `on_turn_start`、`on_session_end`（事实提取/摘要）、`on_session_switch`（会话ID旋转）、`on_pre_compress`（压缩前从即将丢弃的消息中提取洞察）、`on_memory_write`（内置memory工具写入镜像）

### 1.2 MemoryManager编排器

**核心约束：** 只允许一个外部provider同时运行，防止工具schema膨胀和冲突。

**上下文围栏机制：**
- `sanitize_context(text)`：静态正则清除 `<memory-context>` 标签和系统注释
- `StreamingContextScrubber`：流式状态机处理跨chunk边界的 `memory-context` span，span内部丢弃所有内容直到看到 `</memory-context>`
- `build_memory_context_block(raw_context)`：将预取记忆包装在标记块中，带系统注释"Treat as authoritative reference data"

## 二、上下文压缩算法

### 2.1 ContextEngine ABC

上下文管理抽象层：决定何时压缩、如何压缩、工具暴露。只有**一个**引擎活跃（默认 `"compressor"`）。

**生命周期：** `on_session_start` → 每轮 `update_from_response` → `should_compress` → `compress` → `on_session_end`

### 2.2 ContextCompressor五阶段压缩

**关键参数：**
- `threshold_percent = 0.50`：达到模型上下文50%时触发
- `protect_first_n = 3`：保护头N条非系统消息
- `protect_last_n = 20`：始终保护最后N条
- `max_summary_tokens = min(context_length * 0.05, 12_000)`

**阶段1：工具输出裁剪（无LLM调用）**
- 去重旧工具结果（MD5哈希）、总结保护区外的工具输出、截断大型tool_call参数JSON
- 保护区确定：优先使用token预算（反向累加）、`protect_tail_count`为硬底线

**阶段2：确定头/尾边界**
- `_protect_head_size()` → 系统prompt + `protect_first_n` 消息
- `_find_tail_cut_by_tokens()` → 反向累加token到预算用尽
- **关键保护：** `_ensure_last_user_message_in_tail()` 确保最后一条用户消息在尾部（修复#10896）

**阶段3：LLM结构化摘要生成**
- 使用辅助模型（更便宜/更快），支持**迭代更新模式**（PREVIOUS SUMMARY + NEW TURNS）
- 结构化模板：Active Task、Goal、Constraints、Completed Actions、In Progress、Blocked、Key Decisions、Remaining Work、Critical Context
- 错误处理：无provider → 600秒冷却；404/503 → 回退主模型；完全失败 → 确定性回退摘要

**阶段4：组装压缩消息列表**
- 保留头部消息原样，插入摘要作为独立消息（选择不冲突的角色）
- 消除连续相同角色冲突（必要时合并到第一条尾部消息）

**阶段5：后处理清理**
- `_sanitize_tool_pairs()`：删除孤立tool结果，为缺失结果插入桩
- `_strip_historical_media()`：旧消息图像替换为文本占位符

**反抖动保护：** 最近2次压缩每次节省不到10%则跳过。

### 2.3 压缩集成 (conversation_compression.py)

`compress_context()` 编排完整流程：
1. **延迟可行性检查**：首次压缩时才探测辅助模型上下文窗口（节省~400ms冷启动）
2. **压缩锁**：基于session_id的原子锁存储在 `state.db` 中，防止并发压缩同一会话
3. **通知外部memory provider**：在消息被丢弃之前
4. **调用 `context_compressor.compress()`**
5. **会话轮换**：结束旧会话，创建新 `session_id`，通过 `parent_session_id` 链接
6. **通知所有memory provider**：`on_session_switch` 重置缓存的每会话状态
7. **清除文件去重缓存**：压缩后原始读取内容已被摘要

## 三、后台审查：自改进循环

### 3.1 触发机制

每轮后 `spawn_background_review_thread()` 启动守护线程，发送记忆/技能审查prompt。

### 3.2 Fork架构关键设计

- **继承父级凭证和运行时**：相同provider/model/base_url/api_key/api_mode
- **`skip_memory=True`**：防止审查fork污染外部内存插件
- **继承缓存的系统prompt逐字**：审查fork的HTTP请求命中相同前缀缓存，**节省约26%端到端成本**
- **工具白名单**：仅有 `memory` 和 `skill_manage`
- **非交互式审批**：自动拒绝所有危险命令

## 四、自底向上关键组件

### 4.1 错误分类器 (error_classifier.py)

重试/故障切换系统的**决策中枢**，20+种 `FailoverReason`，8级优先级流水线：

1. 供应商特定模式（thinking签名校验、长上下文tier gate）
2. HTTP状态码 + 消息体语义分析（401/403→auth旋转、402→区分计费vs用量限制、413→压缩、429→限流、400→多级分析、5xx→重试/快速失败）
3. 错误码分类
4. 消息模式匹配
5. SSL/TLS瞬时告警→归类为timeout
6. 服务端断连+大会话→上下文溢出
7. 传输错误类型名匹配
8. 兜底：unknown（可重试）

**关键洞察：** 402智能歧义消除 + 5xx请求校验信号快速失败

### 4.2 速率限制追踪器 (rate_limit_tracker.py)

解析12个标准 `x-ratelimit-*` 头，四个桶（RPM/RPH/TPM/TPH），两种格式化（完整ASCII进度条/紧凑单行）

### 4.3 Nous速率守卫 (nous_rate_guard.py)

跨会话断路器：当Nous Portal返回429时写入共享文件 `~/.hermes/rate_limits/nous.json`，所有会话发起请求前检查。区分真实速率限制 vs 上游供应商瞬时容量不足。

### 4.4 模型元数据处理 (model_metadata.py)

上下文长度解析的**10级优先级链路**：
显式配置 → 持久化缓存 → Bedrock静态映射 → 自定义端点探测 → 本地服务器查询 → Anthropic API → 供应商感知查找 → OpenRouter API → 硬编码默认值 → 最终兜底256K

### 4.5 消息净化 (message_sanitization.py)

多层清洗：代理字符清理 → 工具调用参数修复（JSON修复5个pass） → 非ASCII剥离 → 图片内容剥离

### 4.6 Prompt缓存策略 (prompt_caching.py)

`system_and_3` 策略：系统prompt + 最后3条非系统消息上放置最多4个cache_control断点，所有断点共享5分钟TTL。

### 4.7 敏感数据脱敏 (redact.py)

30+前缀模式 + 环境变量赋值 + JSON字段 + Authorization头 + Telegram Bot Token + 私钥块 + 数据库连接字符串 + JWT令牌 + E.164电话号码 + URL查询参数

### 4.8 文件安全 (file_safety.py)

精确拒绝列表 + 前缀拒绝列表 + Hermes控制面保护 + 安全根目录 + 跨Profile写入守卫 + 沙箱镜像写入守卫
