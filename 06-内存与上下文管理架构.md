>

>
> **几个贯穿全文的术语,先在这里一次性交代清楚**:
> - **provider**:这里特指"内存后端实现",不是"LLM 供应商"。一个 MemoryProvider 就是一种"把记忆存到哪、怎么取回来"的插件。
> - **session_id(会话 ID)**:Hermes 给每段对话分配的唯一标识,格式形如 `20260604_114054_43c77b`(时间戳 + 6 位 hex)。压缩、会话切换、内存持久化都围绕它展开。

## 一、内存 Provider 体系

内存系统要解决的问题是:**agent 怎么跨会话记住东西**。一次对话结束、上下文被丢弃后,下一次对话怎么还能"想起"用户上次说过的偏好、做过的决定?Hermes 的答案是一套**插件化的内存后端**——内存的存取逻辑被抽象成一个标准接口 `MemoryProvider`,具体存到哪(本地文件、向量库、外部服务)由实现类决定;再由一个编排器 `MemoryManager` 统一调度。

### 1.1 MemoryProvider ABC——插件化内存后端的标准接口

`MemoryProvider` 是一个抽象基类(ABC,Abstract Base Class,即"只定义方法签名、不给实现,留给子类填"的接口约定),位于 `memory_provider.py`。任何想接入 Hermes 的内存后端,都必须实现下面这组方法。换句话说,这张表就是"一个内存插件必须会做的事"的清单:

| 方法 | 职责 | 在对话循环里的时机 |
|------|------|------|
| `initialize(session_id)` | 会话启动时建立连接(打开数据库、加载索引等) | 会话开始 |
| `system_prompt_block()` | 返回一段静态文本,注入到 system prompt 里(比如"你有长期记忆,可用 memory 工具读写") | 会话开始,拼 system prompt 时 |
| `prefetch(query, session_id)` | 每轮**对话之前**,根据用户当前这句话返回相关的"已缓存记忆上下文"(同步、阻塞) | 每轮前 |
| `queue_prefetch(query, session_id)` | 每轮**之后**,把"为下一轮预取记忆"的任务排进后台队列(异步、不阻塞当前轮) | 每轮后 |
| `sync_turn(user, assistant, session_id, messages)` | 每轮之后,把这一轮的用户问、助手答持久化下来 | 每轮后 |
| `get_tool_schemas()` / `handle_tool_call()` | 把内存相关的工具(如"写一条记忆")暴露给模型 / 处理模型对这些工具的调用 | 工具组装 / 工具调用时 |
| `shutdown()` | 清理关闭(关连接、刷盘) | 会话结束 |

这里要点出一个容易混淆的设计:**`prefetch` 与 `queue_prefetch` 是一对"读—预热"组合**。`prefetch` 在每轮**之前**同步返回"现在已经准备好的"记忆(不能阻塞太久,否则拖慢响应);`queue_prefetch` 在每轮**之后**把"下一轮可能用到的"记忆预取任务丢进后台。这样下一轮 `prefetch` 被调用时,数据往往已经预热好了,避免每轮都现查现等。这是典型的"用后台预取换前台延迟"的工程权衡。

**可选钩子(optional hooks)**:除上面的必选方法外,`MemoryProvider` 还定义了一组**可选钩子**——子类可以选择性实现,用来在对话生命周期的特定节点插入逻辑。"钩子"的意思是"框架在某个事件发生时会主动回调你",而不需要你去轮询。它们分别是:

- `on_turn_start(turn, message, **kwargs)`(`memory_provider.py:25`):每轮开始时的"心跳",附带运行时上下文。
- `on_session_end(messages)`(`:26`):会话真正结束时触发,典型用途是**事实提取 / 摘要**——趁着完整历史还在,从中抽出值得长期记住的事实存进去。
- `on_session_switch(new_session_id, **kwargs)`(`:27`):**会话 ID 在进程中途被轮换**时触发(下文 §2.3 会讲压缩为什么要轮换 session_id)。内存插件借此重置"按会话缓存的状态"。
- `on_pre_compress(messages)`(`:28`):**压缩发生之前**触发,返回一段字符串。它的价值很关键——压缩会丢弃中段消息,这些消息里可能藏着值得记住的洞察;这个钩子让内存插件赶在消息被丢弃**之前**从中抽取信息。
- `on_memory_write(action, target, content, metadata=None)`(`:29`):当内置的 memory 工具写入一条记忆时,把这次写入**镜像**给外部内存插件,让外部插件也能同步感知。

这套"必选方法 + 可选钩子"的设计,本质上是给内存后端留足了"在对话各个关键点插手"的接缝,同时又用必选方法保证了最低限度的功能契约。

### 1.2 MemoryManager 编排器——只准一个外部 provider,外加一道"上下文围栏"

`MemoryManager`(`memory_manager.py`)是内存系统的总调度。它要做两件事:**约束并发**,以及**给注入的记忆套一道安全围栏**。

**核心约束:同一时刻只允许一个外部 provider 运行。** 源码注释写得很直白(`memory_manager.py:7-8, 245`):内置 provider 加上**至多一个**外部 provider;若想注册第二个外部 provider,会被直接拒绝并打 warning(`:273-274` "Only one external memory provider is already registered")。为什么要这么严?两个原因:其一,**防止工具 schema 膨胀**——每个内存 provider 都可能往模型暴露自己的工具,多个 provider 叠加会让工具列表越来越长,既浪费 token 又让模型挑花眼;其二,**防止后端冲突**——两个外部内存后端同时写,记忆语义会打架。此外还有一道工具名冲突检测(`:291` "Memory tool name conflict"),避免两个 provider 抢同一个工具名。

**上下文围栏机制(context fencing)——为什么需要它?** 这是内存系统里最容易被忽视、却很关键的一层安全设计。问题是这样的:内存系统会把"召回的记忆"作为上下文注入到对话里,但**召回的记忆本身是不可信的**——它可能是历史对话里夹带的、甚至是被污染的内容。如果不加处理直接喂给模型,就可能发生**提示注入(prompt injection)**:历史里的某段文本伪装成系统指令,劫持当前这轮的行为。Hermes 用三个工具把这道围栏围起来:

- **`sanitize_context(text)`(`memory_manager.py:54`)**:一个**静态正则清洗器**。它把召回文本里的 `<memory-context>` 标签和特定的系统注释模式(`:49` 那条匹配 "recalled memory context, NOT new user input… Treat as authoritative reference data" 的正则)清掉。目的是:防止有人在历史里伪造这些"权威标记",再借召回机制把伪造内容当成可信上下文塞进来。
- **`StreamingContextScrubber`(`memory_manager.py:62`)**:一个**流式状态机**。为什么不能只用上面那个一次性正则?因为模型是**流式(streaming)输出**的,文本一块一块(chunk)地来,一个 `<memory-context>…</memory-context>` 区间可能**横跨好几个 chunk**,一次性正则看不到完整跨界的内容(`:65` 注释明说 "cannot survive chunk boundaries")。这个状态机的做法是:一旦进入 `memory-context` span(区间),就**丢弃 span 内部的所有内容**,直到看见 `</memory-context>` 闭合标签才恢复。这样无论标签被切在哪个 chunk 边界,内部内容都不会泄漏出去。
- **`build_memory_context_block(raw_context)`(`memory_manager.py:227`)**:把"预取到的记忆"包装进一个带标记的块里。它先调 `sanitize_context` 清洗(`:231`),再加上一段系统注释(`:237`),明确告诉模型这是 "recalled memory context, NOT new user input. Treat as authoritative reference data"(召回的记忆上下文,不是新的用户输入,请当作权威参考数据)。注意这里的措辞设计:它一边声明"这是权威参考",一边强调"不是新的用户输入"——前者让模型敢用,后者防止模型把它当成用户的新指令去执行。

把这三件事串起来理解:**写入侧**(`build_memory_context_block`)给记忆贴上唯一可信的标记并清洗,**输出侧**(`sanitize_context` + `StreamingContextScrubber`)把模型生成里任何冒充这套标记的内容铲掉。一进一出形成闭环,这就是"上下文围栏"。

## 二、上下文压缩算法

上下文管理要解决的问题是:**对话历史越来越长,迟早超过模型上下文窗口,怎么办?** Hermes 的策略不是简单截断(那会丢失关键信息),而是**在快撑爆时,用 LLM 把中段历史压缩成结构化摘要**,头尾关键消息原样保留。这套机制由一个抽象层 `ContextEngine` 统领,默认实现是 `ContextCompressor`。

### 2.1 ContextEngine ABC——上下文管理的抽象层

`ContextEngine`(`context_engine.py`)是上下文管理的抽象基类,它定义"何时压缩、如何压缩、暴露哪些工具"。和内存系统类似,这里也有一条**强约束:只有一个引擎活跃**(`context_engine.py:10` "Only one engine is active"),默认就是内置的 `"compressor"`(即下文的 `ContextCompressor`)。为什么只准一个?因为"什么时候压、压成什么样"是个全局决策,多个引擎同时插手会互相打架。

引擎的生命周期(`context_engine.py:20-24`)是一条清晰的回调链:

1. `on_session_start()`——对话开始时调用。
2. `update_from_response(usage)`——每次 API 响应回来后调用,喂进本轮的 token 用量(usage),让引擎知道当前历史有多长。
3. `should_compress()`——每轮后检查"现在该不该压"。
4. `compress()`——当 `should_compress()` 返回 True 时执行真正的压缩。
5. `on_session_end()`——在真实会话边界(CLI 退出、`/reset` 等)调用。

这条链的核心思路是"**先观测、再判断、后行动**":每轮都用 usage 更新自己对历史长度的认知,够阈值了才压,而不是盲目地每轮都压。

### 2.2 ContextCompressor 五阶段压缩

`ContextCompressor`(`context_compressor.py`)是默认引擎,它把"压缩"这件事拆成五个阶段。在讲阶段之前,先把它的几个**关键参数**交代清楚,因为后面每个阶段都围着它们转。这些是压缩器构造时的默认参数(`context_compressor.py:587-589`),注意它们和"压缩触发阈值"是两回事(阈值公式见 §2.3 与报告 13/17):

| 参数 | 默认值 | 含义 |
|------|--------|------|
| `threshold_percent` | `0.50`(`:587`) | 当历史 token 达到模型上下文窗口的 **50%** 时触发压缩 |
| `protect_first_n` | `3`(`:588`) | 始终保护**头部前 N 条非系统消息**(任务起点、初始约束通常在这里) |
| `protect_last_n` | `20`(`:589`) | 始终保护**最后 N 条消息**(当前正在进行的工作在这里) |
| `max_summary_tokens` | `min(context_length × 0.05, 12_000)`(`:580-581`) | 摘要本身的 token 上限:取"窗口的 5%"和"12000"中的较小值 |

这里的设计哲学是"**护两头、压中间**":对话里最有价值的信息往往集中在开头(任务定义)和结尾(当前进展),中段则是大量可被摘要替代的过程性来回。`protect_first_n` 和 `protect_last_n` 就是把这两头钉死不动,只对中段下手。`max_summary_tokens` 则保证"摘要本身别又写太长"——它最多占窗口的 5%,且有 12000 的硬上限(源码常量 `_SUMMARY_TOKENS_CEILING = 12_000`,`:89`)。

**阶段 1:工具输出裁剪(无 LLM 调用)。** 这是最便宜的一步,完全靠确定性规则、不花一次 LLM 调用就能省下大量 token。做三件事:(a)**去重旧工具结果**——同样内容的工具输出在历史里出现多次时,用 MD5 哈希识别重复并去掉;(b)**总结保护区外的工具输出**——保护区(头尾)之外的大段工具输出被精简;(c)**截断大型 tool_call 参数 JSON**——某些工具调用的入参 JSON 巨大,直接截短。保护区的确定方式是:**优先用 token 预算反向累加**(从最后一条往前加 token,直到预算用尽),而 `protect_tail_count` 是一条**硬底线**(无论预算怎么算,至少保住这么多条尾部消息)。

**阶段 2:确定头/尾边界。** 这一步决定"哪些消息算头、哪些算尾、中间哪段要被摘要替换"。`_protect_head_size()` 算出头部边界 = 系统 prompt + `protect_first_n` 条消息;`_find_tail_cut_by_tokens()` 从尾部反向累加 token 到预算用尽,定出尾部起点。这里有一个**修过 bug 的关键保护**:`_ensure_last_user_message_in_tail()`(`context_compressor.py:1698`)确保**最后一条用户消息一定落在尾部、不会被压掉**。这修的是 issue **#10896**(`:1706`、`:1803`)——原本的边界对齐逻辑 `_align_boundary_backward` 可能把最后一条用户消息推到被压缩的中段去,导致"当前正在做的任务"在压缩中丢失。这条保护是在 `cut_idx = self._ensure_last_user_message_in_tail(...)`(`:1804`)处兜底插入的。

**阶段 3:LLM 结构化摘要生成。** 这是唯一需要花 LLM 调用的阶段,用前面提到的**辅助模型**(更便宜/更快)来写中段摘要。两个值得说清的设计:

- **迭代更新模式(incremental)**:摘要不是每次从零重写,而是支持 "PREVIOUS SUMMARY + NEW TURNS" 的增量更新——把上一次的摘要和这次新增的对话一起喂给辅助模型,让它在旧摘要基础上更新。这避免了重复总结早已总结过的内容。
- **结构化模板**:摘要不是一团自由文本,而是按固定字段组织:Active Task(当前任务)、Goal(目标)、Constraints(约束)、Completed Actions(已完成动作)、In Progress(进行中)、Blocked(被阻塞)、Key Decisions(关键决策)、Remaining Work(剩余工作)、Critical Context(关键上下文)。结构化的好处是:模型能稳定地从摘要里定位到"我现在做到哪、还差什么",而不是在一段散文里大海捞针。

**错误处理(三级降级)**:辅助模型不一定可用,所以这一步有三层兜底——(1)**完全没配置辅助 provider** → 进入 600 秒冷却(源码常量 `_SUMMARY_FAILURE_COOLDOWN_SECONDS = 600`,`:106`),避免反复重试一个不存在的东西;(2)**辅助模型返回 404/503** → 回退用主模型来写摘要;(3)**彻底失败** → 用一个确定性回退摘要(`_build_static_fallback_summary`,`:1001`),即不靠 LLM、纯规则拼出一段"摘要不可用,请从受保护的近期消息继续"的交接说明(`:1146`)。

> **关于摘要请求的 `max_tokens`,一个重要的勘误(详见报告 17 §2.1)**:有早期描述称摘要请求固定用 `max_tokens=2000`,这是**不准确**的。实际是 `max_tokens = int(summary_budget × 1.3)`(`context_compressor.py:1390`),其中 `summary_budget` 自身有地板:`max(2000, min(content×ratio, max_summary_tokens))`(`:935` 处 `max(_MIN_SUMMARY_TOKENS, min(budget, self.max_summary_tokens))`)。这里的 `2000` 只是**下限常量** `_MIN_SUMMARY_TOKENS`(`:85`),不是固定取值。换句话说:摘要预算是动态算出来的,2000 是地板而非定值。

**阶段 4:组装压缩后的消息列表。** 把"保护的头部消息原样保留 + 摘要作为一条独立消息插进去 + 保护的尾部消息"拼成新的消息列表。这里有一个容易被忽略的工程细节:插入摘要时要**选一个不与相邻消息冲突的角色(role)**,并**消除连续相同角色的冲突**——因为很多模型的 wire 协议不允许两条相邻消息是同一个 role(比如连着两条 `user`)。冲突时,必要会把摘要合并到第一条尾部消息里。

**阶段 5:后处理清理。** 压缩重排了消息,可能留下"残局",这一步收尾:(a)`_sanitize_tool_pairs()`——删除**孤立的 tool 结果**(工具调用被压掉了、但它的结果还在,这种悬空结果会让模型/后端报错),并为缺失结果的工具调用**插入桩(stub)**;(b)`_strip_historical_media()`——把旧消息里的图像替换成文本占位符(历史图像既占 token 又基本不再被需要)。

**反抖动保护(anti-thrashing)**:压缩不是越频繁越好——如果每次压缩省不下多少,却反复触发,就会陷入"压了又压"的抖动。Hermes 的对策是:**最近 2 次压缩,如果每次节省都不到 10%,就跳过本次压缩**(`:732` 附近,内部计数 `_ineffective_compression_count >= 2`)。直白说就是:连着两次"白压",就先别压了,等历史真正再长一截再说。

### 2.3 压缩集成(conversation_compression.py)——把五阶段串成完整流程

`ContextCompressor.compress()` 只管"怎么把消息列表压小";真正在对话循环里**编排整个压缩流程**的是 `conversation_compression.py` 里的 `compress_context()`。它按下面七步走,每一步都解决一个真实工程问题:

1. **延迟可行性检查(lazy feasibility check)**:压缩需要知道辅助模型的上下文窗口有多大,但探测这个要花约 400ms 的冷启动。Hermes 的做法是**等到第一次真要压缩时才探测**,而不是会话一开始就探。这样大多数从不触发压缩的短会话,完全不用付这 400ms 的代价。
2. **压缩锁(compaction lock)**:并发场景下,同一个会话可能被两条路径同时触发压缩,会把历史搞乱。对策是一把**基于 session_id 的原子锁**,存在 `state.db` 里,保证同一会话同一时刻只有一个压缩在跑。
3. **通知外部 memory provider**:**在消息被丢弃之前**回调内存插件(对应 §1.1 的 `on_pre_compress` 钩子),让插件趁机从即将丢弃的消息里抽取洞察。顺序很关键——必须赶在丢弃前。
4. **调用 `context_compressor.compress()`**:执行 §2.2 的五阶段,得到压缩后的消息列表。
5. **会话轮换(session rotation)**:结束旧会话,创建一个**新的 `session_id`**,并通过 `parent_session_id` 把新旧会话**链接**起来(`conversation_compression.py:507-523`,新 id 在 `:509` 生成,`create_session(parent_session_id=old)` 在 `:517-523`)。为什么压缩后要换 session_id?因为压缩相当于"对话进入了一个新阶段",换一个 id 能让下游(KV cache、日志、内存)清晰区分"压缩前/后"两段,同时 `parent_session_id` 保留血缘,不丢失"它从哪来"的信息。
6. **通知所有 memory provider 执行 `on_session_switch`**:对应 §1.1 的钩子,让所有内存插件**重置缓存的每会话状态**——因为 session_id 变了,旧的按会话缓存的东西都得作废重建。
7. **清除文件去重缓存**:压缩后,原本"读过哪些文件内容"的去重缓存要清掉——因为那些原始读取内容已经被摘要替换了,缓存里的旧指纹不再对应当前历史。

把这七步连起来看,会发现压缩远不止"调一次 compress()"那么简单:它要管并发(锁)、管成本(延迟探测)、管内存协同(两次回调)、管下游一致性(session 轮换 + 清缓存)。这正是"集成层"存在的意义。

> **压缩触发阈值的补充(本报告聚焦架构,实测细节详见报告 13 §一、报告 17 §1.3 之 I7)**:真正决定"何时触发"的阈值公式是 `threshold_tokens = max(context_length × 0.5, 64000)`,其中 `64000` 是硬地板(常量 `MINIMUM_CONTEXT_LENGTH = 64000`,`model_metadata.py:133`),且 `context_length` 会被模型真实窗口元数据覆盖。**注意区分**:`max_tokens` 在 Hermes 里**没有统一默认值**(按 profile 解析,None 时整键缺省),`64000` 是上面这条压缩阈值的地板,**不是** `max_tokens` 的默认值——这是两个完全不同的量,容易混淆。

## 三、后台审查:自改进循环

除了"记住事实"和"压缩历史",Hermes 还有一套更主动的机制:**每轮之后偷偷起一个后台线程,让模型审查自己的记忆和技能、看有没有可以沉淀/改进的**。这就是"后台审查(background review)"的自改进循环。

### 3.1 触发机制

每一轮对话之后,`spawn_background_review_thread()`(`background_review.py`)会启动一个**守护线程(daemon thread)**,向模型发一个"记忆/技能审查"的 prompt。"守护线程"意味着它在后台跑、不阻塞主对话,主进程退出时它也跟着退。它的任务是让模型回看刚才这轮,判断"有没有值得写进长期记忆的事实、有没有可以提炼成技能的操作"。

### 3.2 Fork 架构的关键设计——为什么能省约 26% 成本

这个审查线程本质上是把当前 agent **fork(派生)** 出一个副本去干活。fork 的几个设计决策都很有讲究(`background_review.py`):

- **继承父级凭证和运行时**:审查 fork 用和主 agent **完全相同**的 provider / model / base_url / api_key / api_mode。这保证审查在和主对话同一个后端、同一套身份下进行。
- **`skip_memory=True`(`:415`)**:审查 fork 关闭外部内存插件。原因很实际——审查本身会产生一堆中间消息,如果让它去写外部内存,会**污染**真正的内存后端。所以审查 fork 自己不碰外部内存。
- **逐字继承缓存的系统 prompt——这是省钱的关键(`:434-440`)**:审查 fork **一字不差**地复用父级缓存的 system prompt。为什么重要?因为 Anthropic / OpenRouter 等后端有**前缀缓存(prefix cache)**:如果两个请求的前缀(system prompt + 前面的消息)字节完全一致,后端可以复用已缓存的 KV,不必重新 prefill。审查 fork 逐字复用父级前缀,就能**命中父对话刚刚预热好的前缀缓存**,源码注释明确写了实测影响:**约 26% 的端到端成本降低**(`:440` "~26% end-to-end cost reduction")。反过来,注释也警告(`:438`):如果前缀不是字节精确一致,缓存键就会 miss,这个收益就没了。
- **工具白名单(`:459`)**:审查 fork **只能用 `memory` 和 `skill_manage` 两个工具**。这把审查的能力面缩到最小——它只该"读写记忆、管理技能",不该去执行别的危险操作。运行时白名单(`set_thread_tool_whitelist`,`:466`)会拦截任何非白名单工具(`:469` "Background review denied non-whitelisted tool")。
- **非交互式审批**:审查在后台无人值守地跑,所以它对所有危险命令**自动拒绝**——不能因为没人确认就放行高风险操作。

把这几条合起来:后台审查是一个"**沿用父级一切、但能力被严格收窄、且专门蹭前缀缓存省钱**"的影子 agent。它让 Hermes 能持续自我沉淀,而代价被压到很低。

## 四、自底向上关键组件

上面三套系统(内存、压缩、后台审查)之外,围绕它们还有一圈**基础组件**,各管一摊横切关注点(cross-cutting concern):错误怎么分类、限流怎么追踪、模型窗口多大、消息怎么净化、缓存断点放哪、敏感数据怎么脱敏、文件操作怎么保安全。下面逐个拆。

### 4.1 错误分类器(error_classifier.py)——重试/故障切换的决策中枢

LLM 调用会因各种原因失败:认证过期、计费用尽、限流、上下文溢出、服务端 5xx……每种失败该怎么应对(重试?换 key?压缩历史?快速失败?)完全不同。`error_classifier.py` 就是把"原始错误"翻译成"该采取什么动作"的**决策中枢**。

它定义了一个 `FailoverReason` 枚举(`error_classifier.py:24`)作为故障原因的结构化分类,**共 21 个成员**(报告原文记为"20+ 种",一致),涵盖如 `thinking_signature`(Anthropic thinking 块签名校验失败,`:58`)、`long_context_tier`(Anthropic 超长上下文的额度 tier gate,`:59`)等。分类逻辑是一条**8 级优先级流水线**(源码里用编号注释清晰标出 `:533`–`:718`),从最具体的供应商特定模式,一路降到最兜底的 unknown:

1. **供应商特定模式**(`:533`,最高优先级):比如 thinking 签名校验失败、长上下文 tier gate 这类只有特定供应商才有的错误。
2. **HTTP 状态码 + 消息体语义分析**(`:649`):这是最核心的一档。不只看状态码,还结合消息体语义——401/403 → 触发 auth 旋转(换凭证);**402 → 区分"计费问题"还是"用量限制"**(这两者状态码相同但应对完全不同,需要看消息体歧义消除);413 → 走压缩(请求体太大);429 → 限流处理;400 → 多级分析;5xx → 在"重试"和"快速失败"之间抉择。
3. **错误码分类**(`:662`)。
4. **消息模式匹配**(`:669`,无状态码时靠文本特征判断)。
5. **SSL/TLS 瞬时告警 → 归类为 timeout**(`:680`):把握手层的瞬时抖动当超时处理(可重试),而不是当成压缩信号。
6. **服务端断连 + 大会话 → 上下文溢出**(`:691`):服务端断开连接、又恰好是个大会话,大概率是上下文撑爆了。
7. **传输错误类型名匹配**(`:713`)。
8. **兜底:unknown**(`:718`,可重试)。

> **两个关键洞察**:其一,**402 的智能歧义消除**——同样是 402,"账户欠费"该让人去充值(不可重试),"超了用量限制"则可能换个 key 或等一等就好,分类器靠消息体把这两者分开。其二,**5xx 的请求校验信号快速失败**——不是所有 5xx 都该重试;如果 5xx 携带"请求本身有问题"的信号,继续重试只是白费,这时应快速失败。

### 4.2 速率限制追踪器(rate_limit_tracker.py)——把限流头变成可读的进度条

供应商在响应头里返回 `x-ratelimit-*` 系列头,告诉你还剩多少配额。`rate_limit_tracker.py` 负责解析这些头并展示。它解析 **12 个标准 `x-ratelimit-*` 头**(即 4 个桶 × limit/remaining/reset 三种,共 12 个具体命名头),归到**四个桶**:RPM(每分钟请求数)、RPH(每小时请求数)、TPM(每分钟 token 数)、TPH(每小时 token 数)(桶定义见 `rate_limit_tracker.py:9-12`)。它提供两种格式化展示:**完整 ASCII 进度条**(`:160`,形如 `[████████░░░░░░░░░░░░] 40%`)和**紧凑单行**(`:238` 起,形如 `RPM: 剩余/上限`)。

### 4.3 Nous 速率守卫(nous_rate_guard.py)——跨会话断路器

普通限流追踪是"单会话内"的。但如果 Nous Portal(Nous 自家的推理门户)对**整个账户**限流了,那么所有会话都该停,而不是各自撞墙。`nous_rate_guard.py` 就是这道**跨会话断路器(circuit breaker)**:当 Nous Portal 返回 429 时,把这个状态写进一个**共享文件** `~/.hermes/rate_limits/nous.json`;所有会话在发起请求**之前**都先读这个文件,被限流就先别发。它还会**区分两种 429**:真实的速率限制(你确实超额了)vs 上游供应商瞬时容量不足(只是后端暂时挤,不是你的错)——两者应对策略不同。

### 4.4 模型元数据处理(model_metadata.py)——上下文长度的 10 级优先级链路

压缩阈值要靠"模型上下文窗口有多大"来算(见 §2.3),但这个值从哪来?不同来源可靠度不同。`model_metadata.py` 用一条**10 级优先级链路**来解析上下文长度,从最可信的显式配置,一路降级到最后的硬兜底:

显式配置 → 持久化缓存 → Bedrock 静态映射 → 自定义端点探测 → 本地服务器查询 → Anthropic API → 供应商感知查找 → OpenRouter API → 硬编码默认值 → **最终兜底 256K**。

这条链的设计意图是"**能精确就精确,不能就退而求其次,最后绝不让它为空**"。最末端的兜底值是 **256K**(`model_metadata.py:115-119`,常量 `256_000`),注释解释选 256K 是因为它能覆盖 GPT-5.x 及当前许多大上下文模型。顺带一提,前面 §2.3 提到的压缩阈值地板 `MINIMUM_CONTEXT_LENGTH = 64_000` 也定义在这个文件(`:133`)。

### 4.5 消息净化(message_sanitization.py)——多层清洗,保证 wire 合法

模型生成的内容、历史里累积的内容,可能含有会让后端拒收的"脏东西"(代理字符、坏 JSON、非 ASCII、超大图像)。`message_sanitization.py` 是一条**多层清洗流水线**,按顺序做:**代理字符清理 → 工具调用参数修复(对坏 JSON 做 5 个 pass 的修复)→ 非 ASCII 剥离 → 图片内容剥离**。其中"工具调用参数修复"尤其实用——模型有时会吐出语法略坏的参数 JSON,这里用多轮(5 pass)修复尽量救回来,而不是直接报错。

### 4.6 Prompt 缓存策略(prompt_caching.py)——system_and_3 断点布局

这是直接服务于 KV 缓存复用的组件。Anthropic 协议允许在消息上放 `cache_control` 标记(断点),后端据此缓存对应前缀的 KV。`prompt_caching.py` 实现的策略叫 **`system_and_3`**:在 **system prompt + 最后 3 条非系统消息**上,放置**最多 4 个 `cache_control` 断点**,所有断点共享同一个 TTL(默认 5 分钟)。

把源码逻辑讲清楚(`prompt_caching.py:49-80`):函数 `apply_anthropic_cache_control` 先看第一条是不是 system,是就给它打一个断点、`breakpoints_used += 1`(`:71-72`);然后 `remaining = 4 - breakpoints_used`(`:74`),再从**所有非系统消息里取最后 `remaining` 条** `non_sys[-remaining:]` 打断点(`:76-77`)。所以"system_and_3"= 1 个 system 断点 + 最后 3 条非系统消息断点 = 最多 4 个,且全部同 TTL。

> **一个容易记错的细节(详见报告 17 §2.2 第 7 点)**:断点的构成是 `system + non_sys[-3:]`(当 system 断点被占用时)。在 **anthropic wire** 下,`system` 是 body 里的**独立字段**(不在 `messages` 数组里);而在内容(content)级缓存时,断点是打在消息的 `content` 上(native,内容级)还是消息顶层(envelope),由 `_apply_cache_marker(native_anthropic=…)` 这个参数决定(`:15-38`),不是两条独立的代码分支。

### 4.7 敏感数据脱敏(redact.py)——把日志里的秘密抹掉

凡是要落日志、要展示的内容,都可能夹带密钥、token、电话号码等敏感信息。`redact.py` 用一大套模式把它们抹掉,覆盖面很广:**30+ 前缀模式 + 环境变量赋值 + JSON 字段 + Authorization 头 + Telegram Bot Token + 私钥块 + 数据库连接字符串 + JWT 令牌 + E.164 电话号码 + URL 查询参数**。这里"E.164"是国际电话号码的标准格式(带国家码的那种);"私钥块"指 PEM 格式的 `-----BEGIN … PRIVATE KEY-----` 区块。把这些都纳入脱敏,是为了防止任何一类秘密从日志渠道泄漏。

### 4.8 文件安全(file_safety.py)——多重护栏防止越权读写

agent 能读写文件,这本身是高风险能力。`file_safety.py` 用多重护栏把它框住:**精确拒绝列表 + 前缀拒绝列表 + Hermes 控制面保护 + 安全根目录 + 跨 Profile 写入守卫 + 沙箱镜像写入守卫**。逐个解释:**精确/前缀拒绝列表**是黑名单(某些确切路径、或某些前缀下的一切都禁止);**Hermes 控制面保护**防止 agent 改写 Hermes 自己的控制文件(否则它能改自己的配置/权限);**安全根目录**把可操作范围限定在某个根之下;**跨 Profile 写入守卫**防止一个 profile 的操作越界写到另一个 profile;**沙箱镜像写入守卫**保护沙箱镜像不被污染。这几道护栏叠加,目的就是"agent 再怎么自由,也别越出安全边界"。

## 五、与下游 KV 调度的关系(交叉引用)

本报告拆的是"内存与上下文管理"的源码架构;它产出的很多东西,正是下游 KV/调度想要的 **agent hint**。简单对应一下,方便你顺着读下去:

- §2 的**压缩**会改变历史前缀、并触发 **session 轮换 + parent_session_id**——这对应报告 17 的信号 I7(压缩阈值)、I11(`SUMMARY_PREFIX` 作为 KV 失效指令)、I12(session 轮换 + 父链),是 KV 主动管理(G1)的核心输入。
- §4.6 的 **`cache_control` 断点布局(system_and_3)**对应报告 17 的 W7,是 KV 缓存复用的直接抓手。
- §3 的**后台审查逐字复用前缀缓存省约 26%**,正是"前缀缓存命中带来的实测收益"的一个鲜活例证。

如果你要的是"每个信号在哪、怎么取、什么保真度、host 门控如何影响采集",请转**报告 17**(提取规格)与**报告 15 §4.3**(host 门控从零讲清);如果你要的是"这些行为在真实请求里长什么样的实测捕获",请转**报告 13–14**。
